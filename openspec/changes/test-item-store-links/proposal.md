## Why

Sub-proposal 4.4 of the `test-coverage` initiative (parent `tasks.md` §4.4, GitHub issue #43) — a capability-flow carve-out for the `item-store-links` capability. The carve-out is the single client component that implements the capability's JS-testable behavior: `app/(main)/items/ui/components/StoreLinks.tsx` (184 LOC). The parent line scopes this as "store-links UI and any associated reads." There are **no store-specific reads** to cover: stores ride along on `ItemDisplay.stores` (`lib/types.ts`), populated by the item reads owned by the items-browser-chrome (4.5) and list-item-management (4.9) flows — `StoreLinks` is a pure client component that receives `item` as a prop and reads no server data, no `'use cache'` surface, no DAL/action. This carve-out is therefore UI-only.

The `item-store-links` capability already has a **substantive active spec** at `openspec/specs/item-store-links/spec.md` (created/extended by archiving `replace-storelinks-expand-with-popover` and `consolidate-row-layouts`) with ~11 requirements. The spec's Purpose is still the post-archive placeholder ("TBD — Update Purpose after archive."). Most of the spec's requirements describe the **CSS row/grid anatomy** (grid templates, breakpoints, corner radii, choose-items reuse) owned by `Item.tsx` + `store-links.css` — those are out of carve-out (see Non-Goals) and are not JS-coverage-testable in jsdom. The requirements this carve-out can lock with unit tests are the `StoreLinks.tsx` behavioral contracts: the `+N` `<Menu>` popover (sorted ascending, all stores incl. primary, prices, `target`/`rel`, upward-with-flip placement) and the hover-open contract (220 ms grace, anchor-wrapper boundary). Reading the source surfaces three load-bearing invariants the spec does not currently lock — the **store-validity predicate**, the **empty-state fallback**, and **click-isolation (`stopPropagation`)** — which this sub-proposal ADDS.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-housekeeping`, `test-pure-libs`, `test-button-system`, `test-chip-system`, `test-form-field-system`, `test-menu-system`, `test-popover-trigger-system`, `test-segmented-control-system`, `test-misc-primitives`, `test-app-frame`) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Every requirement applies verbatim.
- `item-store-links` (active spec) — owns ~11 requirements. This carve-out LOCKS the JS-testable behavioral subset (the `+N` `<Menu>` popover requirement and the hover-open requirement), MODIFIES the placeholder Purpose, and ADDS three requirements for source-enforced invariants no requirement currently states (store-validity predicate, empty-state fallback, click-isolation). No requirement is REMOVED; the CSS-anatomy requirements are untouched.
- `menu-system` (active) — `StoreLinks.tsx` renders `<Menu>` + `<MenuLinkItem>` from the menu capability. The primitive itself is already tested under `test-menu-system` (archived); this carve-out does NOT mock `<Menu>` — it renders through the real primitive and asserts the integration (the popover's open/closed mount, the rendered `<MenuLinkItem>` anchors). No `menu-system` SHALL is asserted directly from here.
- `button-system` (active) — `StoreLinks.tsx` renders one `<LinkButton variant="primary">` (the primary buy-link) and one `<Button variant="ghost">` (the `+N` trigger). Both primitives are already tested under `test-button-system` (archived); rendered through, not mocked. No `button-system` SHALL is asserted directly.
- `list-item-management` (4.9) / `items-browser-chrome` (4.5) — OWN `Item.tsx`, the shared row/card that renders `<StoreLinks>` and the `.item-price-row` ancestor styling. The grid-anatomy, breakpoint-reflow, choose-items-reuse, and corner-radius requirements in the `item-store-links` spec are CSS contracts realized in `Item.tsx` + `store-links.css` + `item.css`; they belong to those flows' carve-outs and to code review, not to this UI-only unit carve-out.
- Cache-tag rule does NOT apply — `StoreLinks` reads no server data and has no `'use cache'`/`cacheTag` surface.

## What Changes

- **NEW** colocated test file `app/(main)/items/ui/components/__tests__/StoreLinks.test.tsx` (jsdom) — one test file for the one executable source file in the carve-out. It exercises:

  - **Store validity + sort:** the `validStores` filter (`s?.name && s?.link && !Number.isNaN(Number(s.price))`) drops stores missing a name, missing a link, or with a non-numeric price; `sortedStores` orders the survivors price-ascending; the cheapest survivor becomes `primary` (the `lowestPrice` selection) and the `.item-price` reflects it. Tests feed an `item.stores` array mixing valid + invalid entries and assert the rendered primary chip, the `+N` count, and the popover rows reflect only valid stores in ascending order.
  - **Empty-state fallback:** when no valid store survives (`!lowestPrice`), the component renders `children` wrapped in `<div className="item-action-row">` — or returns `null` when no children are passed. No `.item-price-row`, no `.storeLinks` chips. (Two distinct branches.)
  - **Price row:** `.item-price-row > .item-price` renders the lowest price formatted `$X.XX` (`Number(primary.price).toFixed(2)`).
  - **showStores gate:** with `showStores={false}`, the price row renders but the `.storeLinks` chip block does NOT (locks the `showStores` prop's default-`true` contract and the gated branch).
  - **Single vs multi store:** exactly one valid store → primary `<LinkButton>` only, NO `.storeLinks-more` `+N` trigger, NO `has-extras` class on `.storeLinks`; two-or-more valid stores → primary chip + `+N` trigger inside `.storeLinks-more-anchor`, `has-extras` class present, trigger `aria-label` pluralization (`Show 1 more store` vs `Show 3 more stores`).
  - **Primary chip contract:** `<LinkButton variant="primary" className="storeLinks-link">` with `href={primary.link}`, `target="_blank"`, `rel="noreferrer"`, the store name text, and an `MdOpenInNew` icon.
  - **`+N` popover (locks the existing "+N trigger SHALL open a `<Menu>`" requirement):** clicking the trigger toggles `open` → `aria-expanded` flips, the `<Menu>` mounts (it returns `null` when closed), the popover contains exactly M `<MenuLinkItem>` rows for M valid stores **including the primary**, each row shows `.storeLinks-menu-name` + `.storeLinks-menu-price` (`$X.XX`), rows are ordered price-ascending (primary first), each anchor carries `href`/`target="_blank"`/`rel="noreferrer"`; clicking a `<MenuLinkItem>` calls `setOpen(false)` (closes); clicking the trigger again closes; the `<Menu>`'s own outside-click / Escape dismiss closes (rendered through the real primitive).
  - **Hover-open (locks the existing "Hover-open" requirement):** `mouseenter` on `.storeLinks-more-anchor` opens immediately (`cancelCollapseAndOpen`); `mouseleave` schedules a close after `COLLAPSE_DELAY_MS` (220 ms) — verified with `vi.useFakeTimers()` + `vi.advanceTimersByTime`; a `mouseenter` before the timer fires cancels the scheduled close; the anchor wrapper is the single boundary element containing both trigger and `<Menu>` panel.
  - **Placement (locks the existing "opens upward / flips below" scenarios):** opening triggers `computePlacement`; the `.storeLinks-more-anchor` carries `placement-above` or `placement-below`. jsdom does not implement layout (`getBoundingClientRect` returns zeroed rects, `getComputedStyle().overflowY` is `'visible'`), so the test STUBS `Element.prototype.getBoundingClientRect` and `window.getComputedStyle` to drive the scroll-clipping-ancestor walk and the `roomAbove >= panelHeight || roomAbove >= roomBelow` ternary through both the `'above'` and `'below'` outcomes (the default no-room-info case resolves `'above'`; a configured stub with a clipping ancestor and insufficient room above resolves `'below'`).
  - **Click-isolation (Decision: ADDED SHALL):** clicking the primary `<LinkButton>` and clicking the `+N` `<Button>` each call `e.stopPropagation()` — a `click` dispatched on either does NOT bubble to a parent click handler. Verified by wrapping `<StoreLinks>` in a parent `<div onClick={spy}>` and asserting the spy is not called when the chip / trigger is clicked (while the toggle still flips `open`).

- **NO** separate test for `Item.tsx` — out of carve-out (owned by 4.5 / 4.9).
- **NO** test for `store-links.css` / `item.css` — CSS is not in the JS coverage report; the grid-anatomy / breakpoint / corner-radius / choose-items-reuse requirements are CSS contracts owned by 4.5 / 4.9 and verified by code review (and, for the reflow behaviors, by the e2e carve-outs 6.x).
- **NO** DAL/action coverage — there are no store-specific reads; `item.stores` is populated upstream by item reads owned by other flows.
- **NEW** ESLint override in `eslint.config.mjs`: append `app/(main)/items/ui/components/StoreLinks.tsx` to the existing per-file `sonarjs/cognitive-complexity = error` override array, under a `// test-item-store-links (sub-proposal 4.4)` header comment. Measured complexity at HEAD is expected well under the ceiling of 15 (`computePlacement`'s ancestor-walk loop + the placement ternary are the only non-trivial branches).
- **NEW** per-file threshold in `vitest.config.ts` for `app/(main)/items/ui/components/StoreLinks.tsx`, referencing the existing `COVERAGE_FLOOR` constant (one entry, no per-file numeric variation per `test-housekeeping`'s single-constant rule).
- **MODIFIED** `item-store-links` spec — the placeholder Purpose ("TBD — Update Purpose after archive.") is replaced with a real one-paragraph Purpose describing the capability. No existing requirement's scenarios are reworded.
- **ADDED** three `item-store-links` requirements for source-enforced invariants no requirement currently locks:
  - **ADDED** Requirement: a store SHALL be considered valid only when it has a non-empty `name`, a non-empty `link`, and a numeric `price`; invalid stores SHALL be excluded from both primary selection and the popover — locks the `validStores` predicate that decides what renders at all.
  - **ADDED** Requirement: when an item has no valid store, `<StoreLinks>` SHALL render its `children` inside a `.item-action-row` wrapper, or `null` when no children are supplied; it SHALL NOT render the price row or any chip — locks the empty-state fallback that keeps the claim/purchase affordance visible for store-less items.
  - **ADDED** Requirement: activating the primary buy-link or the `+N` trigger SHALL NOT propagate the click to the enclosing card/row — locks the `stopPropagation` calls that prevent a buy-link or popover toggle from also triggering row selection / navigation.
  These elevations are documented in `design.md`.
- **NEW** `testing-foundation` carve-out bookkeeping record (sub-proposal-archive ONLY, per `test-coverage` design D13 two-tier rollup): the `item-store-links` capability carve-out has landed at the universal `COVERAGE_FLOOR`, the `sonarjs/cognitive-complexity` promotion to `error` is in effect for `StoreLinks.tsx`, and subsequent sub-proposals rendering `<StoreLinks>` inherit the assumption that it is tested and complexity-locked. Recorded in `openspec/changes/test-item-store-links/specs/testing-foundation/spec.md` ONLY — does NOT roll into the parent `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md`.
- **NEW** four-audit findings recorded in `tasks.md`. Anticipated dispositions:
  - **Assertion-substance audit (on the new tests)** — every test names an observable property (DOM attribute, exact class string, rendered text, anchor `href`/`target`/`rel`, spy call argument, popover row count + order).
  - **Duplication audit** — anticipated shared fixture: an `ItemDisplay`-with-stores builder. Single test file, so it stays inline (a local `makeItem(stores)` helper). Extraction to `test/fixtures/` only if a second file needs the same builder.
  - **Complexity audit** — expected to pass; `StoreLinks.tsx` ≈ moderate (the `computePlacement` ancestor walk is the peak), below ceiling 15.
  - **Testability audit** — two attention points, both disposed without source refactor: (1) jsdom has no layout — STUB `getBoundingClientRect` + `getComputedStyle` to exercise `computePlacement`'s branches; (2) the 220 ms hover-grace timer — use `vi.useFakeTimers()`. No source change anticipated.

## Impact

- **New files:** one `.test.tsx` file under `app/(main)/items/ui/components/__tests__/` (the first `__tests__/` directory under that path). No shared fixture extracted on first authoring; the duplication audit may surface an `ItemDisplay` builder if a second consumer appears.
- **Modified config:** `eslint.config.mjs` gains one path in the per-file `sonarjs/cognitive-complexity = error` override array. `vitest.config.ts` gains one entry in the per-file `thresholds` map, pointing at `COVERAGE_FLOOR`.
- **Modified source:** none expected. The two testability attention points (jsdom layout, hover timer) are handled in-test, not by source refactor.
- **Modified specs:** `openspec/specs/item-store-links/spec.md` gets a rewritten Purpose and three ADDED requirements. No removals, no scenario rewording. The carve-out bookkeeping requirement lives ONLY in this sub-proposal's archive directory (`openspec/changes/archive/<date>-test-item-store-links/specs/testing-foundation/spec.md`); it does NOT roll into the parent `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md` (per `test-coverage` design D13 two-tier rollup).
- **CI:** the existing four-gate workflow runs unchanged; the `test` job runtime grows by one jsdom file. The placement test installs local `getBoundingClientRect` / `getComputedStyle` stubs and the hover test uses fake timers — both reset in `afterEach`, scoped to the file.
- **Dependencies:** none added.
- **Downstream unblock:** the parent `test-coverage` change's checkbox for sub-proposal 4.4 flips on archive. Flows that render `<StoreLinks>` through `Item.tsx` (4.5 `test-items-browser-chrome`, 4.9 `test-list-item-management`) inherit a known-good, complexity-locked store-links component with its validity / empty-state / click-isolation contracts elevated in the spec.
- **Risk:** low. One client component, fully testable as-shipped. Notable risks:
  - **jsdom has no layout engine.** `computePlacement` reads `getBoundingClientRect()` and `getComputedStyle().overflowY`, both of which return inert values in jsdom (zeroed rects, `'visible'` overflow). Mitigation: stub both via `vi.spyOn(Element.prototype, 'getBoundingClientRect')` and `vi.spyOn(window, 'getComputedStyle')` to drive the scroll-clipping-ancestor walk and the placement ternary through `'above'` and `'below'`. Reset in `afterEach`.
  - **The 220 ms hover-grace timer.** `scheduleCollapse` defers the close via `setTimeout`. Mitigation: `vi.useFakeTimers()` + `vi.advanceTimersByTime(220)`; assert the cancel path by re-entering before the timer fires.
  - **The `<Menu>` primitive is rendered un-mocked and attaches `document` listeners while open.** Mitigation: same pattern as `test-menu-system` — the real primitive is the integration target; assertions are on the popover's mount/unmount (`Menu` returns `null` when closed) and the rendered `<MenuLinkItem>` anchors, not on `menu-system` internals.
  - **Store-validity predicate is a behavioral claim worth defending.** A future contributor relaxing `!Number.isNaN(Number(s.price))` (e.g. to allow `null` prices) would change what renders. The ADDED SHALL + tests name the regression specifically.
- **No runtime behavior change** — no source edits anticipated.
