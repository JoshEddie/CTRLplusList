## Why

Sub-proposal 4.13 of the `test-coverage` initiative — the **HIGH-stakes** authorization carve-out. The carve-out is every Next.js server action under `app/actions/**` plus every API route handler under `app/api/**` that touches user-owned data: `app/actions/lists.ts`, `app/actions/items.ts`, `app/actions/follows.ts`, `app/actions/user.ts`, and `app/api/image-search/route.ts`. The parent `test-coverage` change's `tasks.md` §4.13 line is explicit: every server action and API route is asserted authorized for every caller class — **owner / authenticated non-owner / unauthenticated**. Authorization is the app's primary defense against cross-user data access, so this carve-out exists to build the regression net under that defense, not to discover that it is missing — the code was already hardened by the now-archived `harden-server-action-authorization` (2026-05-22) and `harden-remaining-server-actions` (2026-05-22) changes. (The parent issue's "in-flight" coordination note is stale; both hardening changes archived before this sub-proposal opened. This carve-out tests the hardened code at HEAD.)

The `server-endpoint-authorization` capability already has a substantive active spec at `openspec/specs/server-endpoint-authorization/spec.md` (created by archiving `harden-server-action-authorization`) with six requirements covering session-identity resolution, ownership verification, paid-quota API auth, per-user rate limiting, cache-invalidation-on-success-only, and the `after()` request-context boundary. This sub-proposal therefore does NOT create a new capability spec. It (a) locks the existing SHALLs against colocated `.test.ts` integration files at the universal `COVERAGE_FLOOR`, (b) **MODIFIES one stale requirement in `list-item-management`** where the spec mandates an enforcement mechanism the production driver cannot provide (the `SELECT … FOR UPDATE` transactional row-lock — see "What Changes" and Decision 5), (c) **ADDS one requirement to `server-endpoint-authorization`** extending the actor-resolution / no-actor-parameter rule to the follow-graph mutation actions, which the source enforces today but the spec's enumeration (lists / items / purchases) omits, and (d) records the carve-out bookkeeping under `testing-foundation` Tier 2 per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, the five primitive-family carve-outs, `test-housekeeping`, `test-misc-primitives`, and `test-app-frame`) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, `.test.ts` → **node** project (DB-integration tests run under node, not jsdom — these action/route tests are `.test.ts`), universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Two mocking allowances apply verbatim and are load-bearing here: **(1) NextAuth is a network boundary** — `@/lib/auth`'s `auth()` / `signIn` / `signOut` are mocked, never the internal modules they wrap; **(2) the image-search upstream provider is a metered boundary** — its `fetch` is intercepted, never called for real. **Internal modules (DAL reads, `lib/listAccess`, `lib/sqlstate`) are NOT mocked** — the tests run against a real pglite-backed database per the `test-foundation-spike` DB-under-test decision.
- `server-endpoint-authorization` (active spec) — six requirements (see Why). Every requirement is asserted against as the authoritative authorization contract. The tests are the regression net for: session-identity resolution (no client `user_id`), ownership verification before update/delete, paid-quota API auth (401), per-user rate limiting (429) + query-length cap (400), cache-invalidation-only-on-success (no `updateTag` on the unauthorized path), and the `after()` request-context boundary (no `auth()` / `headers()` inside `after()`). **One requirement is ADDED** (follow-graph actor resolution — see What Changes).
- `list-item-management` (active spec) — owns the purchase-claim contract: R "createPurchase SHALL authenticate the claimer and forbid client-supplied user_id" (with the non-viewable-item rejection scenario), R "Purchase capacity SHALL be enforced atomically against concurrent callers", and R "removePurchase for guest callers SHALL require the purchase row id". The createPurchase + removePurchase tests assert against these as the authoritative contract. **One requirement MUST be MODIFIED**: the capacity-enforcement requirement currently mandates a single-transaction `SELECT … FOR UPDATE` row lock, which the `drizzle-orm/neon-http` driver cannot provide (DATABASE.md: "Interactive transactions are not supported … If you find code claiming to use a transaction here, it's broken"). The source enforces capacity via an in-app check plus a partial unique index, and accepts the multi-claim capacity race as residual. The spec is corrected to match the testable reality — see Decision 5.
- `following` (active spec) — owns the behavioral contract for `followUser` / `unfollowUser` / `removeFollower` / `blockUser` / `unblockUser`: self-follow rejection, both-direction block gating, follow idempotency, removeFollower scoping (`(follower=that_user, followee=viewer)`), block-removes-follow-both-directions, and "Follow-graph mutations SHALL NOT use interactive transactions". The follows tests assert against these as the authoritative behavioral contract. This carve-out ADDS the cross-cutting *authorization* angle (actor exclusively from session; no actor parameter) to `server-endpoint-authorization`, not to `following` — the behavioral semantics stay where they live.
- `visit-history` (active spec) — owns "bookmarkList SHALL enforce the 'viewable list only' predicate" and the per-row-remove / bulk-clear contract for `clearVisitHistory` / `removeVisit`. The lists-action tests for `bookmarkList` / `unbookmarkList` / `clearVisitHistory` / `removeVisit` assert against these. No new `visit-history` SHALL is added — the predicate is already specced.
- `list-visibility` (active spec) — owns `setListVisibility`'s three-state model, `shared_at` transition rule, and legacy `shared` dual-write. The `setListVisibility` test asserts the authorization gate (owner-only) and the visibility-state transitions against these SHALLs. No new `list-visibility` SHALL.
- **Active `extract-visibility-constants` change (Stage 1 of a 3-stage rollout)** — `app/actions/lists.ts` consumes `VISIBILITY` constants and the `fromDb()` tolerant decoder from `@/lib/visibility`. Stage 2 will flip the constant DB values; Stage 3 runs the DB `UPDATE`. **Coordination requirement:** the `setListVisibility` / `bookmarkList` tests SHALL reference `VISIBILITY.*` and `fromDb()` (NOT the literal DB strings `'private'` / `'unlisted'` / `'public'`) so they survive the Stage 2 value flip without a rewrite. See Decision 6.
- Cross-cutting design-system rule does NOT apply: this carve-out touches no interactive-surface primitive (no buttons, chips, fields, menus). Cache-tag rule DOES apply and is directly under test: every mutation's `updateTag(...)` call is asserted on the success path, and the absence of `updateTag` is asserted on the unauthorized path (locking `server-endpoint-authorization`'s "Authorization rejections SHALL NOT invalidate caches" requirement).
