## Why

Sub-proposal 4.14 of the `test-coverage` initiative ([issue #52](https://github.com/JoshEddie/CTRLplusList/issues/52)). The carve-out is the `visit-history` capability ‚Äî the visit-recording upsert, the bookmark toggle, and the visit-history page mutations, plus their backing DAL reads. The parent `test-coverage` `tasks.md` ¬ß4.14 names this as "`visit-history` capability + visit dedupe race (partial unique index backstop)."

This is the FIRST capability-flow carve-out to test backing **DAL reads** and **server-action mutations** against the real test database. The six primitive-family carve-outs (3.1‚Äì3.8) and `test-app-frame` (4.1) were all jsdom component/hook tests; none exercised `lib/dal.ts` or `app/actions/*.ts`. So this sub-proposal is the first real consumer of the PGlite DB-under-test harness (`test/helpers/db.ts` `bootPglite()`) and the `next/cache` mock (`test/helpers/next-cache.ts` `mockNextCache()`) established by `test-foundation` ‚Äî both proven against the visit-history-adjacent surfaces in the `test-foundation-spike` PoC (`dal-cache.test.ts`, `race.test.ts`).

The active `visit-history` spec at `openspec/specs/visit-history/spec.md` already has six substantive requirements (created by archiving `add-following-and-history`). So this sub-proposal does NOT create a new capability spec ‚Äî it (a) locks the existing SHALLs against new tests at the universal `COVERAGE_FLOOR`, (b) **MODIFIES two scenarios in R3** where the spec describes pre-rework "remove disabled for bookmarked rows" UX but the source has evolved to a strictly-better "remove from history while preserving the bookmark" behavior (see `design.md` Decision 5), (c) **ADDS one SHALL** locking the `last_visited_at IS NULL` exclusion that connects the mutation actions to the history-page read (see Decision 6), and (d) records carve-out bookkeeping under `testing-foundation` Tier 2 per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active accumulator delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation-spike`, `test-foundation`, `test-housekeeping`, `test-pure-libs`, and the six primitive carve-outs + `test-app-frame`) ‚Äî runner (vitest 4.x), the two-project split (`.test.tsx` ‚Üí jsdom, `.test.ts` ‚Üí node for DAL/DB integration), `__tests__/` colocation, the universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, the no-backdoor disposition rule, the `<State>_<Behavior>` `it()` shape, the three-role `describe()` convention, the four-audit + invariant-elevation obligations, the assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out, the **"DAL functions are not mocked from action tests ‚Äî run against the real test database"** rule, and the **"internal modules SHALL NOT be mocked; mock only the network boundary"** rule. Every requirement applies verbatim. **One requirement composes badly with this carve-out** ‚Äî the universal per-file floor on multi-capability shared files (`lib/dal.ts`, `app/actions/lists.ts`) ‚Äî surfaced and disposed of in Decision 2.
- `visit-history` (active spec) ‚Äî six active requirements (visit recording, bookmark toggle survives in history, per-row remove + bulk clear, the `saved_lists` migration, the `bookmarkList` viewability gate). The tests assert against these SHALLs as the authoritative contract. **Two R3 scenarios MUST be MODIFIED** (the "remove disabled for bookmarked" pair ‚Äî see "What Changes"). **One NEW SHALL is ADDED** (the history-view `last_visited_at IS NULL` exclusion). The migration requirement (R4 ‚Äî `saved_lists` data copy) is a one-time historical migration outcome verified at archive of `add-following-and-history`; it is NOT re-exercised here (Decision 8).
- `list-collections` (active) ‚Äî OWNS the `/lists/bookmarks` and `/lists/history` **page chrome** (`ListCard.tsx`, the collection grids, the route shells). Per the boundary `test-app-frame` (4.1) already drew, these two routes belong to `list-collections` (4.6). This carve-out therefore does NOT cover `BookmarksPage.tsx` / `BookmarksList.tsx` / `HistoryPage.tsx` / `HistoryList.tsx` / the `page.tsx` shells / `ListCard.tsx` ‚Äî it covers the visit-history-SPECIFIC interactive widgets those pages render (`BookmarkButton.tsx`, `HistoryActions.tsx`, `HistoryCard.tsx`), which ¬ß4.6's file list does not name. See Decision 1.
- `button-system` (active) ‚Äî `BookmarkButton.tsx` and `HistoryActions.tsx` render `<Button variant="on-dark">` / `<Button variant="secondary|primary|danger|ghost">`. The primitive is already tested under `test-button-system` (archived); this carve-out asserts the widgets' behavior THROUGH the rendered button, not the button-system internals.
- `list-visibility` (active) + `harden-remaining-server-actions` (in-flight, aligns with 4.13) ‚Äî `bookmarkList`'s viewability gate (reject when caller is non-owner AND `visibility === VISIBILITY.OWNER`, with a deliberately non-specific `'List not viewable'` error) is an authorization invariant. The active `visit-history` spec ALREADY locks it ("bookmarkList SHALL enforce the viewable-list-only predicate"). This carve-out TESTS that existing SHALL; it does not add a new one (the authorization-matrix sweep is 4.13's job).

Cache-tag note (per the proposal rule): the visit-history reads consume the `list_visits` cache tag (`getBookmarkStatus` is `'use cache'` + `cacheTag('list_visits')`; `getBookmarkedListsByUser` and `getVisitHistoryByUser` are deliberately NOT cached because they join `users` for owner name ‚Äî documented in source). Every visit-history mutation path (`bookmarkList`, `unbookmarkList`, `clearVisitHistory`, `removeVisit`, and the `ListHeroSection` visit-recording `after()` block) calls `updateTag('list_visits')`. The tests assert the tag-invalidation call shape via `mockNextCache()` (per the spike's recommendation (b): mock `next/cache`, assert the mutation invokes `updateTag('list_visits')` ‚Äî real-runtime invalidation is e2e's job, 6.x).

## What Changes

- **NEW** colocated test files. Two `.test.ts` (node project, DAL + actions against PGlite) and three `.test.tsx` (jsdom, the interactive widgets):

  - `lib/__tests__/visitHistory.dal.test.ts` (node) ‚Äî exercises the three visit-history DAL reads against `bootPglite()` with `next/cache` mocked. Functions: `getBookmarkedListsByUser` (returns only `favorited_at IS NOT NULL` rows, ordered `favorited_at DESC`, with the joined owner `name`), `getBookmarkStatus` (`true` iff a `favorited_at IS NOT NULL` row exists for `(listId, userId)`), `getVisitHistoryByUser` (returns only `last_visited_at IS NOT NULL` rows, ordered `last_visited_at DESC`, honoring `limit`/`offset`). The production functions are imported and run directly (NOT re-implemented in the test) so coverage attributes to `lib/dal.ts`; `@/db` is mocked to the PGlite instance per Decision 3. Asserts the **history-view exclusion SHALL** (Decision 6): a row with `last_visited_at = NULL` but `favorited_at` set appears in `getBookmarkedListsByUser` but NOT in `getVisitHistoryByUser`.

  - `app/actions/__tests__/visitHistory.actions.test.ts` (node) ‚Äî exercises the four visit-history mutations + the visit-recording upsert against `bootPglite()`. `@/lib/auth`'s `auth()` is mocked at the NextAuth boundary (per testing-foundation's allowance) to control the caller; `@/db` is mocked to PGlite; `next/cache` is mocked. Covers:
    - `bookmarkList` ‚Äî happy path (upsert sets `favorited_at`, returns `{ success: true }`, calls `updateTag('list_visits')`); **viewability gate** (non-owner + `VISIBILITY.OWNER` list ‚Üí `{ success: false, error: 'List not viewable' }`, NO row inserted, NO `updateTag` call); owner can bookmark own private list; any authed user can bookmark `LINK` / `FOLLOWERS` list; unauthenticated ‚Üí `{ success: false, error: 'Unauthorized' }`; bookmark-before-any-visit creates the row.
    - `unbookmarkList` ‚Äî nulls `favorited_at`, preserves `last_visited_at` + `visit_count`; unauthenticated ‚Üí Unauthorized.
    - `clearVisitHistory({ includeBookmarked: false })` ‚Äî deletes `favorited_at IS NULL` rows, nulls `last_visited_at` on bookmarked rows (bookmark survives, row leaves history view); `{ includeBookmarked: true }` ‚Äî deletes ALL rows for the user; unauthenticated ‚Üí Unauthorized.
    - `removeVisit` ‚Äî non-bookmarked row ‚Üí deleted; **bookmarked row ‚Üí `last_visited_at` nulled, row + bookmark preserved** (Decision 5 ‚Äî the source behavior the spec MODIFICATION locks); no-matching-row ‚Üí `{ success: true, message: 'No history row' }`; unauthenticated ‚Üí Unauthorized.
    - **Visit-recording upsert + dedupe race** ‚Äî the upsert logic the `ListHeroSection` `after()` block runs (`insert ‚Ä¶ onConflictDoUpdate(target: [user_id, list_id])`) is exercised directly against PGlite: first visit ‚Üí row with `visit_count = 1`; repeat visit ‚Üí same row, `visit_count` incremented, `last_visited_at` advanced, `favorited_at` preserved; **two concurrent upserts to the same `(user_id, list_id)` via `Promise.allSettled` ‚Üí both fulfill, exactly one row exists, `visit_count` is consistent** (Decision 4 ‚Äî the dedupe backstop is the composite PRIMARY KEY, NOT a partial unique index; `onConflictDoUpdate` ABSORBS the conflict rather than raising 23505).

  - `app/(main)/lists/ui/components/__tests__/BookmarkButton.test.tsx` (jsdom) ‚Äî renders `<BookmarkButton listId initialBookmarked>` with `@/app/actions/lists` mocked (`bookmarkList`/`unbookmarkList` are the network-equivalent action boundary; mocking them is allowed because the actions themselves are covered by the node test, and the widget test asserts the optimistic-UI contract, not the action's DB behavior ‚Äî Decision 7). Asserts: initial pressed state reflects `initialBookmarked`; click optimistically flips `aria-pressed` + the label (`Bookmark`‚Üî`Bookmarked`) + the icon (`FaRegBookmark`‚Üî`FaBookmark`); on action success, the state persists and `router.refresh()` fires; on action failure, the optimistic state **rolls back** and `toast.error(message)` fires; a click while `isPending` is a no-op (the `aria-disabled` guard). `react-hot-toast` and `next/navigation`'s `useRouter` are mocked.

  - `app/(main)/lists/history/__tests__/HistoryActions.test.tsx` (jsdom) ‚Äî covers both exports. `RemoveVisitButton`: click invokes `removeVisit(listId)`; success ‚Üí `router.refresh()`; failure ‚Üí `toast.error`; `isPending` guard no-op; the button is **always enabled** (the `aria-label="Remove from history"` √ó affordance renders regardless of bookmark state ‚Äî Decision 5's MODIFIED-scenario contract). `ClearHistoryButton`: opens a `role="dialog"` confirm modal; "Clear non-bookmarked" calls `clearVisitHistory({ includeBookmarked: false })`; "Clear all" calls `clearVisitHistory({ includeBookmarked: true })`; "Cancel" closes without calling the action; success closes the modal + `toast.success` + `router.refresh()`; failure shows `toast.error`.
