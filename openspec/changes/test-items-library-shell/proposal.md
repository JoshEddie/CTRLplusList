## Why

Sub-proposal 4.18 of the `test-coverage` initiative — a capability-flow carve-out covering the **items-library page shell** under `app/(main)/items/`. The parent `test-coverage/tasks.md` §4.18 names the carve-out as `ItemsContainer.tsx`, `ItemsPage.tsx`, `app/(main)/items/page.tsx`, `app/(main)/items/loading.tsx`, with the explicit instruction to coordinate the `redirect()`-on-unauthenticated paths with §4.13 (server-endpoint authorization) and confirms it is the deferred boundary from `test-items-browser-chrome` §9.6. This sub-proposal is that close-out: it pins the four-file boundary, covers the shell at the universal `COVERAGE_FLOOR`, and creates a new `items-library-shell` capability spec for the latent invariants the shell enforces that no requirement currently locks.

These four files are the page-level server/route shells the `test-items-browser-chrome` carve-out (4.5) deliberately scoped out: they mix `auth()` + DAL reads (`getItemsByUser` / `getItemsByListId` / `getListsByUser`) + `next/headers` cookie parsing + `redirect()` + the new-item form mount — concerns the browser-chrome spec's own "chrome within `ItemsBrowser`" scope statement excludes. There is no existing capability spec for the page shell, so this sub-proposal CREATES one (`items-library-shell`); it does not modify `items-browser-chrome`.

Coordination with §4.13 (resolved here, not deferred): the `server-endpoint-authorization` carve-out's scope is every server action under `app/actions/**` plus every API route handler under `app/api/**` — it does NOT cover page-level React Server Component shells. The `redirect()`-on-unauthenticated guard in `page.tsx` and `ItemsContainer` is a page-RSC guard, not an action/route guard, so it is owned HERE. §4.13 locks that no server action or API route trusts a client-supplied actor; this sub-proposal locks that the items-library *page* refuses to render for an unauthenticated or unresolvable viewer. The two are complementary halves of the same defense, with no overlap.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active accumulator in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas through `test-app-frame` / `test-home-digest`) — vitest 4.x runner with the jsdom/node two-project split (`.test.tsx` → jsdom, `.test.ts` → node), four-gate pre-merge, `__tests__/` colocation, the universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, the no-backdoor disposition rule (`/* v8 ignore */` with a named reason, never lower the floor), the `<State>_<Behavior>` `it()` shape, the three-role `describe()` convention, the four-audit obligation (assertion-substance, duplication, complexity, invariant-elevation), the assertion-substance bar, the `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy, and the two load-bearing mocking allowances: **(1) NextAuth is a network boundary** — `@/lib/auth`'s `auth()` is mocked, never the modules it wraps; **(2) `next/navigation`'s `redirect()` is mocked to throw a sentinel** per the established `lib/__tests__/listAccess.test.ts` / `app/(main)/following/__tests__/FollowingPage.test.tsx` pattern. The DAL reads are mocked at the `@/lib/dal` boundary (per `FollowingPage.test.tsx`), because the shell's contract under test is *which* read it requests with *which* flags and *which* props it forwards — not the DAL's query internals (spoiler-filtering, list-scoping, visibility), which are owned by their own carve-outs. Every requirement applies verbatim.
- `items-browser-chrome` (active spec) — owns the chrome *within* `ItemsBrowser` (toolbar, view mode, pagination) and the five behavioral requirements `test-items-browser-chrome` (4.5) elevated, including **R-D: the `items_page_size` cookie write contract** (`path=/`, `max-age=31536000`, `SameSite=Lax`, option set `{12,24,48,96}`, `DEFAULT_PAGE_SIZE` normalization). This sub-proposal locks the **server READ half** of that same cross-request contract (the shell reads the same cookie name and applies the same option-set normalization to seed `initialPageSize`). `ItemsBrowser` / `ItemsPage`'s `ItemsToolbar` are rendered as already-tested children where the shell mounts them — they are module-mocked to stubs in the shell tests (out of THIS carve-out, tested under 4.5), so the shell's wiring is exercised without re-owning chrome internals.
- `items-price-filter`, `item-store-links`, `list-item-management` (active specs) — own the filter popovers, the item-card content, the item create/edit form (`ItemFormContainer`, mounted by `ItemsPage`'s "New Item" toggle), and the purchase/claim modal. `ItemFormContainer` is module-mocked where `ItemsPage` mounts it (out of carve-out, 4.9 / 4.16 / 4.17).
- `empty-state-system` (active spec) — owns the `Empty` primitive `ItemsPage` renders for the active-tab empty state. It is already tested under its primitive carve-out and is rendered (not mocked) so the shell's `type="item"` + `setShowNewItem` wiring is exercised; no `empty-state-system` SHALL is asserted directly here.
- `button-system`, `app-frame` (active specs) — own the `Button` ("New Item", `FaPlus`) and the `Header` the shell composes. Already tested; rendered for real; no primitive SHALL asserted here.
- `server-endpoint-authorization` (active spec, §4.13) — owns action/route authorization. Does NOT own page-RSC redirects (see coordination note above). No requirement of it is asserted or modified here.
- Cross-cutting design-system rule: the shell composes governed primitives (`Header`, `Button`, `Empty`, `LoadingIndicator`) without page-scoped overrides of their classes; no primitive variant/size is added, so there is no primitive-spec MODIFIED entry. Cache-tag rule: the shell's server reads (`getItemsByUser`, `getItemsByListId`, `getListsByUser`) are `'use cache'`-tagged in `lib/dal.ts` and consumed read-only here — the shell triggers no mutation, so it has no `revalidateTag` obligation; the tests assert the reads' *outputs* against a seeded DB, not cache behavior.

## Carve-out boundary

**IN the carve-out (4 executable files):**

- `app/(main)/items/page.tsx` — the `/items` route RSC: `auth()` → `getUserIdByEmail` → `redirect('/')` on no viewer; `purchases` search-param → `showSpoilers`; `items_page_size` cookie read + normalization → `initialPageSize`; parallel `getItemsByUser` active + archived loads; `getListsByUser`; the `firstLastInitial` display-name derivation; the `.container--items-library` wrapper around `ItemsPage`.
- `app/(main)/items/ui/components/ItemsContainer.tsx` — the list-vs-library mode RSC: `auth()` → `getUserIdByEmail`; the `!listId && !user` and library-branch `redirect('/')` guards; `readPageSizeCookie`; `getItemsByListId` (viewer / owner / spoiler flags) vs `getItemsByUser`; the same `firstLastInitial` derivation; the `Suspense`-wrapped `ItemsBrowser` (`mode="list"`) vs `Items` branch.
- `app/(main)/items/ui/components/ItemsPage.tsx` — the client tab shell: active/archived tab state from `?tab=`, the per-tab count labels, `setTab` (`router.replace`, `page` removed, `tab=archived` set/cleared), the per-tab empty states (active → `Empty type="item"`; archived → the custom "No archived items" block), the `ItemsBrowser` mount per tab, and the "New Item" `ItemFormContainer` toggle.
- `app/(main)/items/loading.tsx` — the route-level Suspense fallback (`LoadingIndicator size="page"`).

**DEFERRED (with rationale — not this carve-out):**

- `ItemsBrowser.tsx`, `Items.tsx`, `ItemsToolbar` and the rest of the chrome — owned by `items-browser-chrome` (4.5, archived). Module-mocked to stubs where the shell mounts them.
- `ItemFormContainer` and the `itemform/*` tree — owned by `list-item-management` (4.9). Module-mocked where `ItemsPage` mounts the "New Item" form.
- `Empty`, `Header`, `Button`, `LoadingIndicator` — governed primitives, already tested; rendered for real (not mocked).
- `PriceFilterPopover.tsx` (4.16), `StoreFilterPopover.tsx` (4.17) — sibling discovered sub-proposals from `test-items-browser-chrome` §9.6; not reachable from these four shell files.

## Capabilities

### New Capabilities

- `items-library-shell`: the page-level server/route shell of the items library under `app/(main)/items/` — the `/items` route RSC, the list-vs-library `ItemsContainer`, the active/archived tab shell `ItemsPage`, and the route loading boundary. Covers the viewer-authentication guard, the reveal-spoiler URL-param contract, the server-side `items_page_size` cookie-read normalization (the read half of the `items-browser-chrome` R-D write contract), the active/archived dual-load + tab partition, and the `ItemsContainer` list-vs-library routing.

### Modified Capabilities

None. The shell's invariants are not currently owned by any spec; they are introduced as the new `items-library-shell` capability. `items-browser-chrome` is unchanged (its R-D write contract is referenced, not modified). `server-endpoint-authorization` is unchanged (it owns actions/routes, not page-RSC guards).

## Impact

- **New files:** four `*.test.tsx` files under `app/(main)/items/__tests__/` and `app/(main)/items/ui/components/__tests__/`. No shared test helper extracted on first authoring; the duplication audit may surface a viewer-seeding / `auth()`-mock factory or an `ItemDisplay` fixture builder if reused across 3+ files — extracted to `test/helpers/` or `test/fixtures/` only if so.
- **Modified config:** `eslint.config.mjs` gains the four paths in the per-file `sonarjs/cognitive-complexity = error` override array; `vitest.config.ts` gains four entries in the per-file `thresholds` map, each pointing at `COVERAGE_FLOOR`.
- **Modified source:** none expected. The `firstLastInitial` derivation is duplicated verbatim between `page.tsx` and `ItemsContainer.tsx`, and the page-size cookie-read normalization is duplicated between them and conceptually mirrors `ItemsBrowser`'s client-side `normalizePageSize`; the **duplication audit** will dispose of these — likely by extracting `normalizePageSize(raw)` into `paginationConstants.ts` (already the home of the option set) and the name-derivation into a co-located `utils.ts`, with the new tests proving behavior preservation. Any such extraction is in-carve-out single-file refactor authority, recorded in `tasks.md` §audits.
- **New specs:** `openspec/specs/items-library-shell/spec.md` is created with five requirements (via this change's `specs/items-library-shell/spec.md` ADDED delta). No existing spec is modified or removed. The carve-out bookkeeping record lives ONLY in this sub-proposal's archive delta — it does NOT roll into the parent `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md`.
- **Governance:** `openspec/changes/test-coverage/tasks.md` §4.18 flips to done on archive; this sub-proposal closes the last of `test-items-browser-chrome` §9.6's three deferred boundaries (alongside 4.16 / 4.17).
- **CI:** the four-gate workflow runs unchanged; the `test` job grows by four jsdom files. `page.test.tsx` and `ItemsContainer.test.tsx` are the heaviest (real pglite DB boot + seed, `auth()` / `redirect()` / `cookies()` boundary mocks, async-RSC element-tree inspection).
- **Dependencies:** none added.
- **Risk:** low. Four files, all testable as-shipped against established precedents (`FollowingPage.test.tsx` for async-RSC + DAL-mock + redirect-sentinel; `app/(main)/__tests__/page.test.tsx` for the wrapper + Suspense-fallback element-tree pattern). Notable risks:
  - **Async server components cannot be RTL-`render`ed directly (they return a promise).** Mitigation: follow the `FollowingPage.test.tsx` pattern — mock `auth()` / the DAL reads / `cookies()`, `await` the component as a function, then `render` the resolved tree (or inspect the element tree), asserting forwarded props on the stubbed child and the DAL read's call arguments. Documented in `design.md`.
  - **`page.tsx` and `ItemsContainer` share the same auth/name/cookie logic.** The duplication audit decides extraction vs. keep-separate; the tests assert behavior (forwarded props, redirect, normalized size), not the shared code's location, so they survive an extraction.
  - **The five elevated requirements are real behavioral claims worth defending** — each survives a reasonable reimplementation and protects a real failure mode (unauthenticated render, spoiler leak, page-size preference reset, archived/active leakage, wrong list-vs-library read). The new tests fail loudly if any is regressed.
- **No runtime behavior change** other than any audit-surfaced in-place refactor (behavior-preserving, proven by the new tests).
