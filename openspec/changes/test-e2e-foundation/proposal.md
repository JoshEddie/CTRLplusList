## Why

Sub-proposal **6.0** of the `test-coverage` initiative ([issue #102](https://github.com/JoshEddie/CTRLplusList/issues/102)). It establishes the **e2e execution foundation** and **blocks 6.1 (`test-e2e-critical-flows`) and 6.2 (`test-e2e-pwa-offline`)** so those stay focused on authoring flow specs rather than re-deriving how Playwright runs.

`testing-foundation` (1.2) noted "CI does not currently run Playwright" and treated `test:e2e` as author-run-local — not as a rule, but because **there were no Playwright tests yet**. 6.1 is the first change to create them, so the e2e *execution* model — how Playwright runs, against which DB, in CI — needs a home of its own rather than being smuggled into every flow-authoring change. This is that home. The two pressures it resolves: **run cheaply** (don't burn the metered live Neon branch on every local re-run) and **run faithfully** (exercise the real `'use cache'` / `revalidateTag` layer, which `next dev` disables).

Inherited constraints surfaced by spec-grep (binding SHALLs that apply verbatim):

- **`testing-foundation`** (active accumulator at `openspec/changes/test-coverage/specs/testing-foundation/spec.md`): e2e specs live under `e2e/`; `scripts/seed-dev-users.ts` is the canonical **seed-as-fixture** (a seed edit is a breaking change to the suite); **no real external services** — "NextAuth is not invoked against real Google"; the assertion-substance bar; and the Playwright naming convention `<PageOrFlow>_<Action>_<ExpectedOutcome>`. The DAL/action **pglite** harness (Tier 1) is the unit/integration tier and is untouched by this change — e2e is a separate, out-of-process tier. **This change's `testing-foundation` delta is Tier 1** (per `test-coverage` design D13): the e2e execution model is cross-cutting foundation, exactly the "likely elevates into `testing-foundation`" the parent §6.0 anticipated.
- **No-transactions driver constraint** (`DATABASE.md`, `openspec/config.yaml`): production is `drizzle-orm/neon-http`, which cannot speak to vanilla Postgres and forbids `db.transaction(...)` / `FOR UPDATE`. The local driver-switch gains transactions the prod driver lacks — a fidelity gap mitigated by the existing no-transactions rule (no call site uses them) plus the pre-promote ephemeral-Neon gate below, which re-runs against the real driver.
- **Dev-bypass hard guardrail** (`CLAUDE.md`, [lib/auth.ts:59](lib/auth.ts) `bypassEnabled()`): the bypass refuses to activate when `NODE_ENV === 'production'`. Because faithful e2e requires `next start` (which forces `NODE_ENV=production`), that guardrail and the `AUTH_BYPASS` flag are **replaced** by a stronger single-source guardrail — the `USE_PG_DRIVER` localhost-only boot guard.
- **`list-visibility`** (public lists are URL-accessible to anyone) and **`server-endpoint-authorization`** (the sanctioned unauthenticated `createPurchase` guest-write path): the contracts the guest-claim flow the harness enables will exercise. Referenced here, **not modified** — 6.1 owns the flow assertions.

## What Changes

- **BREAKING (local-dev workflow): remove `AUTH_BYPASS`; unify "local mode" onto `USE_PG_DRIVER`.** When `USE_PG_DRIVER=1`, the app (a) swaps the DB driver to `postgres-js` against a localhost Docker Postgres, and (b) bypasses real Google OAuth (sessions are synthesized, never negotiated). When unset, the app uses `neon-http` → Neon and requires real Google sign-in. Local preview becomes **one flag**, and it is the same flag the e2e servers set.
- **NEW orthogonal session-identity selector** (`BYPASS_SESSION_USER`, name finalized in design). Bypassing auth and *having a session* are separate concerns: this selector names **which seeded user** the synthesized session represents — default `dev-test-viewer` (preserves today's preview behavior), the literal `guest` ⇒ **no session** (logged-out). The seam accepts **any** seeded user id, so the harness is not locked to a single logged-in identity; future cross-user flows (owner observes a friend's claim) become a one-line project addition rather than a redesign.
- **NEW driver-switch in [db/index.ts](db/index.ts)**: `postgres-js` when `USE_PG_DRIVER=1`, else `neon-http` (unchanged prod path), with a **localhost-only boot guard** that refuses to start if `USE_PG_DRIVER=1` and `DATABASE_URL` is not localhost. This guard is the production safety invariant: on Vercel the flag is unset (→ neon-http + real auth); a misconfigured `USE_PG_DRIVER=1` fails loudly at boot rather than leaking.
- **NEW `docker-compose.e2e.yml`** (Postgres sidecar, committed *non-secret* test credentials per the budget_eddiefamily precedent); schema applied via **`drizzle-kit push`** (fast, no migration replay); **seed-as-fixture** invoked through the same switch (`USE_PG_DRIVER=1 DATABASE_URL=<docker> npm run db:seed:dev` — the seed already imports `db` from `@/db`, so it routes to Docker automatically).
- **NEW reshaped [playwright.config.ts](playwright.config.ts)**: `next build && next start` (production) instead of `next dev`, so the cache/tag-revalidation layer is genuinely under test; **two projects** — `authenticated` (session = seeded viewer) and `guest` (no session) — as a **reusable harness**. Plus thin helper script(s) to start/teardown the Docker DB and run the suite.
- **NEW** dedicated npm scripts so local mode is a single command, never hand-set env: `dev:local` sets `USE_PG_DRIVER=1` + the localhost `DATABASE_URL` together (Docker DB + bypass, replacing the old `AUTH_BYPASS=true` preview), while plain `dev` stays on Neon + real auth; `test:e2e` is inherently local (the harness sets both). The localhost `DATABASE_URL` has **one source of truth** shared by the scripts, compose, and `e2e/helpers/constants.ts`. The localhost boot guard is thus a defense-in-depth backstop, not a normal-workflow step.
- **NEW CI tiers**:
  - a **per-PR, no-secret Postgres-sidecar e2e job** (fork-safe) that boots the sidecar, `drizzle-kit push` + seed, and runs both Playwright projects;
  - a **pre-promote gate** (trusted branches only; needs a Neon API secret) that branches the **production** Neon project (copy-on-write, prod untouched), runs `drizzle-kit migrate` against it, then re-seeds the test fixture and reads through the production `neon-http` driver — the only place migration-*replay*-against-the-real-prod-schema (a migration prod is missing / drift) and `neon-http` driver divergence are caught before production. **Scope boundary:** this validates *replay* via `drizzle-kit migrate`; it does **not** prove the manual SQL apply run against prod (a different mechanism with different transaction/statement handling) is safe — that step stays a manual responsibility, out of scope for 6.0.
- **Sync the descriptive note** in `openspec/config.yaml` (and the matching `testing-foundation` prose) that says "CI does not currently run Playwright" — CI now does.
- **Docs:** rewrite the `CLAUDE.md` "Dev auth bypass for preview" section and the `README.md` bypass note to the `USE_PG_DRIVER` model; update the `AUTH_BYPASS` mention in the seed-file header.
- **Writes no flow specs** (6.1's `e2e-critical-flows`) and **no PWA/offline specs** (6.2). This change is the execution foundation and the harness contract only.

## Capabilities

### New Capabilities

- (none) — the execution model is elevated into the existing `testing-foundation` capability rather than spawning a new one, per the parent §6.0 and the single-home convention.

### Modified Capabilities

- `testing-foundation` (**Tier 1** per `test-coverage` design D13): **ADDED** requirements for the e2e execution model — (1) the local Docker DB target via the `USE_PG_DRIVER` driver-switch with its localhost boot guard; (2) auth bypass unified onto `USE_PG_DRIVER` with an orthogonal, any-seeded-identity session selector; (3) `next start` production execution so the cache layer is under test; (4) the two-project bypass-on / bypass-off reusable harness; (5) docker + `drizzle-kit push` schema + seed-as-fixture wiring; (6) the two CI tiers (per-PR sidecar + pre-promote ephemeral-Neon `migrate`). As a Tier 1 delta it is authored to this change's `specs/testing-foundation/spec.md` and rolls into the parent accumulator (`openspec/changes/test-coverage/specs/testing-foundation/spec.md`).

## Impact

- **Production source:** [db/index.ts](db/index.ts) (conditional driver-switch + localhost guard) and [lib/auth.ts](lib/auth.ts) (bypass keyed on `USE_PG_DRIVER` + the session selector; `AUTH_BYPASS` and the `NODE_ENV` guard removed). **Vercel runtime behavior is unchanged** — `USE_PG_DRIVER` unset means neon-http + real Google auth exactly as today.
- **New dependency:** `postgres` (postgres-js), imported by `db/index.ts` and used only when the flag is set. **Docker** becomes a prerequisite for local e2e and for bypassed local preview.
- **Test / infra:** [playwright.config.ts](playwright.config.ts) reshaped; new `docker-compose.e2e.yml`, e2e helper script(s), and likely an `e2e/helpers/constants.ts` for the committed test config; [.github/workflows/ci.yml](.github/workflows/ci.yml) gains the per-PR e2e job and the pre-promote migrate gate (the latter requires a `NEON_API_KEY`-class secret on trusted branches).
- **Config / docs:** `openspec/config.yaml` note sync; `CLAUDE.md` + `README.md` bypass sections; the `AUTH_BYPASS` comment in [scripts/seed-dev-users.ts](scripts/seed-dev-users.ts); developer-managed `.env.local`.
- **Untouched:** `vitest.config.ts` and the pglite unit/integration harness (e2e is a separate tier contributing no per-file coverage); `db/schema.ts`; all DAL reads and cache tags (no server-side read added or modified, so no new revalidation paths).
- **Downstream:** unblocks 6.1 and 6.2, which assert their flows against this harness.
