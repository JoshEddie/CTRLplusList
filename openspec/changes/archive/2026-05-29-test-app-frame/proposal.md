## Why

Sub-proposal 4.1 of the `test-coverage` initiative — the first capability-flow carve-out after the primitive-family carve-outs (3.1–3.6) and the foundation work (1.1–1.2, 2.1, 0.1) archived. The carve-out is the seven UI modules and one hook owned by the `app-frame` capability: `app/ui/components/AppFrame.tsx`, `app/ui/components/AppNav.tsx`, `app/ui/components/AppMenu.tsx`, `app/ui/components/AppLogo.tsx`, `app/ui/components/Logo.tsx`, `app/ui/components/Header.tsx`, `app/ui/components/Nav.tsx`, and `app/ui/hooks/useKeyboardOffset.ts`. The parent `test-coverage` change's `tasks.md` §4.1 line is explicit on the file list. The `app-frame` capability already has a substantive active spec at `openspec/specs/app-frame/spec.md` (created by archiving `redesign-home-and-tokens`); this sub-proposal therefore does NOT create a new capability spec — it (a) locks the existing SHALLs against colocated test files at the universal `COVERAGE_FLOOR`, (b) MODIFIES one stale scenario where the spec describes pre-rework behavior but the source has evolved past it (the mobile nav scenario — see Decision 3 in `design.md`), (c) ADDS new SHALLs for invariants the source enforces today that no requirement currently locks (mobile-menu dismissal contract, `/lists/bookmarks` / `/lists/history` peer-route exclusion from the Lists-pill active match, `useKeyboardOffset`'s `--keyboard-offset` CSS-variable contract), and (d) adds the carve-out bookkeeping under `testing-foundation` Tier 2 per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, `test-button-system`, `test-housekeeping`, `test-chip-system`, `test-form-field-system`, `test-menu-system`, `test-popover-trigger-system`, `test-segmented-control-system`) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy, NextAuth-as-network-boundary mocking allowance. Every requirement applies verbatim.
- `app-frame` (active spec) — 5 active requirements covering: the `(main)` route group renders the persistent frame; the gradient nav renders the brand lockup, primary nav, and viewer avatar; the active nav pill reflects the current route; `global.css` declares the design-token set used by the frame; pages under `(main)/` consume tokens (not literals). The tests assert against these SHALLs as the authoritative contract. **One scenario MUST be MODIFIED** (the mobile-nav scenario in R2 — see "What Changes" below). Three NEW invariants surfaced by reading source at HEAD are ADDED alongside the existing requirements — see "Capabilities" below.
- `button-system` — `Nav.tsx` renders three `<LinkButton variant="on-dark">` instances; the existing `button-system` spec governs the visual + a11y contract of the rendered `<a>` elements. Token-surface link only; no `button-system` SHALL is asserted from this carve-out's tests directly. The link is honored via the `LinkButton` primitive being tested separately under `test-button-system` (already archived). No new test coverage of `button-system` invariants from here.
- `following`, `list-collections`, `items-browser-chrome`, `list-visibility`, `home-digest` — these capabilities OWN the routes that AppNav's `isActive` matcher classifies (`/`, `/lists`, `/lists/bookmarks`, `/lists/history`, `/items`, `/purchased`). Their behavior contracts are not part of this carve-out; this carve-out only locks the route-prefix matching rules `AppNav` applies. Note specifically: `/lists/bookmarks` and `/lists/history` are owned by `list-collections` (they are the bookmark / visit-history peer collections), and AppNav's source contains a `LISTS_PEERS_EXCLUDED_FROM_ACTIVE` set excluding them from the `/lists` prefix match. This refinement is not stated in the active `app-frame` spec.
- `pwa-shell` — unrelated; AppFrame does not render the service-worker registration component (that lives in the root layout, outside the `(main)/` group).
- Cross-cutting design-system rule applies: this proposal is a spec-modification, NOT a page-scoped one-off. The app-frame is a capability with a primitive-family-like governance footprint (every `(main)/` page renders inside it), so spec edits here EXTEND and MODIFY the existing `app-frame` spec. No new sibling capability is created. Cache-tag rule does NOT apply (no server reads other than `auth()` in `Nav.tsx` and `AppMenu.tsx`'s child `<User>`, which read the NextAuth session — not a `'use cache'`/`cacheTag` surface).
