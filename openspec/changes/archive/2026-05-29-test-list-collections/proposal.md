## Why

Sub-proposal 4.6 of the `test-coverage` initiative — a capability-flow carve-out following the foundation work (1.1–1.2, 2.1, 0.1), the six primitive-family carve-outs (3.1–3.8), and the first capability-flow carve-out (4.1 `test-app-frame`, archived). The carve-out is the four page-UI modules owned by the `list-collections` capability per the parent `test-coverage` change's `tasks.md` §4.6: `app/ui/components/ListCard.tsx`, `app/ui/components/ListCardRow.tsx`, `app/ui/components/MoreCard.tsx`, and `app/ui/components/ListCollectionsNav.tsx`. The `list-collections` capability already has a substantive active spec at `openspec/specs/list-collections/spec.md` (created by archiving `redesign-home-and-tokens`) with four requirements covering the peer-group sub-nav; this sub-proposal therefore does NOT create a new capability spec — it (a) locks the existing sub-nav SHALLs (R2 tab-strip-with-active-marking, R3 right-side-actions-slot) against colocated test files at the universal `COVERAGE_FLOOR`, (b) ADDS new SHALLs for the card-rendering invariants the source enforces today that no requirement currently locks (the UTC-stable date + subtitle-placeholder layout contract; the conditional bookmark indicator + owner-byline gating; the `ListCardRow` empty-state + dual-condition "+N more" affordance + `MoreCard` accessible label), and (c) adds the carve-out bookkeeping under `testing-foundation` Tier 2 per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, `test-button-system`, `test-housekeeping`, `test-chip-system`, `test-form-field-system`, `test-menu-system`, `test-popover-trigger-system`, `test-segmented-control-system`, `test-loading-indicator-system`, `test-misc-primitives`, `test-app-frame`) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Every requirement applies verbatim.
- `list-collections` (active spec) — 4 active requirements: R1 (the four routes `/lists`, `/lists/bookmarks`, `/lists/history`, `/following` form a peer group sharing a sub-nav), R2 (the sub-nav renders as a tab strip with the active peer marked and serving as the page heading), R3 (the sub-nav hosts page-level primary actions on its right side), R4 (the global app-nav SHALL NOT show an active pill on `/lists/bookmarks`, `/lists/history`, `/following`). This carve-out's `ListCollectionsNav.test.tsx` locks the COMPONENT's contribution to R2 and R3 (tab strip, exact-match active marking + `aria-current`, inactive tabs as real links, the children → `.list-collections-actions` slot with omission when empty). R1's page-set enumeration and the R2 "active tab IS the page heading / no separate `<Header>`" clause are page-composition contracts NOT reachable from this component in isolation (see `design.md` Decision 5). R4 is already regression-locked by `test-app-frame`'s ADDED "Lists nav pill SHALL NOT activate on `/lists/bookmarks` or `/lists/history`" requirement in the `app-frame` spec — this carve-out does NOT re-assert it.
- `home-digest`, `following` — these capabilities also render the card components (`ListCard` / `ListCardRow` / `MoreCard` appear in the home rails, `FollowingPage`, `BookmarksPage`, `HistoryPage`, `MyListsPage`, `ProfilePage`, and `UserCardGrid`). The cards are shared UI; the parent `tasks.md` §4.6 nonetheless assigns their test coverage to THIS carve-out, and issue #45 explicitly designates `list-collections` as the elevation target for their latent invariants ("MAY elevate latent invariants to the `list-collections` spec"). The card-rendering SHALLs ADDED here are owned by `list-collections` accordingly; the consuming capabilities inherit them. See `design.md` Decision 4.
- `button-system`, `chip-system`, etc. — none of the four carve-out files render a design-system primitive. `ListCard`, `MoreCard`, and the tabs in `ListCollectionsNav` are raw `next/link` anchors; the bookmark / owner icons are `react-icons` SVGs. No primitive-family SHALL is asserted from this carve-out.
- Cross-cutting design-system rule applies: this is a spec-modification (ADD to an existing capability spec), NOT a page-scoped one-off. Cache-tag rule does NOT apply (no server reads — all four files are pure render / client components with no `auth()` or `'use cache'` surface).

## What Changes

- **NEW** colocated test files under `__tests__/` directories mirroring the source layout. Four test files in total — one per source file, all under `app/ui/components/__tests__/`, all jsdom:

  - `ListCard.test.tsx` (jsdom) — renders `<ListCard list={fixture} />` and asserts: the rendered root is an `<a class="list-card" href="/lists/${id}">`; the list name renders in `<span class="list-card-name-text" title={list.name}>` with the `title` attribute equal to the name (native-tooltip-on-truncation contract); the occasion renders in `<span class="list-card-occasion">`; the date renders in `<span class="list-card-date">` formatted via `toLocaleDateString('en-US', { year:'numeric', month:'short', day:'2-digit', timeZone:'UTC' })` — the test constructs a `Date` whose UTC and local calendar days differ (e.g. `new Date('2025-01-01T00:30:00Z')` evaluated where local time is the prior day) and asserts the rendered string reflects the **UTC** day, locking the off-by-one-protection contract; when `list.subtitle` is a non-empty string, `<div class="list-card-subtitle">` renders with that text and NO `.list-card-subtitle-placeholder` appears; when `list.subtitle` is null/undefined/empty, `<div class="list-card-subtitle-placeholder" aria-hidden>` renders instead (layout-height-preservation contract) and NO `.list-card-subtitle` appears; when `bookmarked` is true, `<FaBookmark class="list-card-bookmark-indicator" aria-label="Bookmarked">` renders inside `.list-card-name`; when `bookmarked` is false (default), no element with `aria-label="Bookmarked"` is present; when `showOwner` is true AND `list.user?.name` is a non-empty string, `<div class="list-card-byline">` renders with a `<FaUser aria-hidden>` and the owner name text; when `showOwner` is true but `list.user` is null OR `list.user.name` is null, NO byline renders (the `showOwner && user.name` gating); when `showOwner` is false (default), NO byline renders even if `list.user.name` is present. `next/link` is module-mocked to the shared `MockNextLink` from `./test-helpers` (real anchor in jsdom); `react-icons` SVGs render real markup (no mock).

  - `ListCardRow.test.tsx` (jsdom) — renders `<ListCardRow lists={...} emptyMessage={...} />` and asserts: when `lists` is empty, the rendered root is `<div class="list-card-row-empty">` containing the `emptyMessage` node, and NO `[role="list"]` is present; when `lists` is non-empty, the root is `<div class="list-card-row" role="list">` with one `<div class="list-card-row-item" role="listitem">` per list, each wrapping a `<ListCard>` (the rendered `.list-card` anchor) in source order; `showOwner` threads through to each card (asserted via the presence/absence of the byline when `showOwner` + owner name supplied); `bookmarkedIds` threads through as `bookmarked={bookmarkedIds.has(list.id)}` per card (a list whose id is in the set renders the `aria-label="Bookmarked"` indicator; one not in the set does not; absent `bookmarkedIds` defaults every card to not-bookmarked via the `?? false`); the "+N more" `MoreCard` row item renders ONLY when BOTH `moreCount > 0` AND `seeAllHref` is a non-empty string (the dual-condition `showMore` gate) — asserted across the four combinations (count>0 + href present → MoreCard renders; count>0 + href absent → no MoreCard; count===0 + href present → no MoreCard; count===0 + href absent → no MoreCard); when `MoreCard` renders, it is the LAST `.list-card-row-item` after all the list cards.

  - `MoreCard.test.tsx` (jsdom) — renders `<MoreCard moreCount={3} href="/lists" />` and asserts: the rendered root is an `<a class="more-card" href="/lists">` with `aria-label="3 more — see all"` (the exact accessible-name template, including the em-dash); the visible text content is `+3 more →` rendered as `<span class="more-card-text">+{moreCount} more <span aria-hidden="true">→</span></span>` — the arrow glyph is wrapped in an `aria-hidden="true"` span so the accessible name comes from the `aria-label`, not the decorative arrow. `next/link` → `MockNextLink`.

  - `ListCollectionsNav.test.tsx` (jsdom) — mocks `next/navigation`'s `usePathname` (controlled per test, same pattern as `AppNav.test.tsx`) and renders `<ListCollectionsNav>`. Asserts: the rendered root is `<div class="list-collections-nav">` containing `<nav class="list-collections-tabs" aria-label="List collections">`; exactly four tab links render in source order with hrefs `/lists`, `/lists/bookmarks`, `/lists/history`, `/following` and labels `My Lists`, `Bookmarks`, `Recently visited`, `Following` (locks R2's tab strip); when `pathname === '/lists'`, ONLY the My Lists tab carries `list-collections-tab--active` + `aria-current="page"` and the other three carry the plain `list-collections-tab` class with no `aria-current`; the same exact-match assertion for `pathname === '/lists/bookmarks'` (Bookmarks active), `/lists/history` (Recently visited active), `/following` (Following active) — locks R2's active-marking, including the negative case that `/lists/bookmarks` does NOT activate the `/lists` (My Lists) tab (exact match, not prefix); when `pathname === '/user/abc123'` (a non-peer route where `ProfilePage` also renders the nav), NO tab is active (the exact-match negative case); when `children` is provided, `<div class="list-collections-actions">` renders containing the children (locks R3's actions slot); when `children` is omitted, NO `.list-collections-actions` element renders (the `{children && ...}` omission). `next/link` → `MockNextLink`.
