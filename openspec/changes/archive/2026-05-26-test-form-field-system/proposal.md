## Why

Sub-proposal 3.3 of the `test-coverage` initiative. With `test-button-system` (3.1), `test-housekeeping`, and `test-chip-system` (3.2) archived, the form-field family is the next primitive carve-out. The carve-out is `app/ui/components/field/` — ten executable files (`FormField.tsx`, `TextField.tsx`, `TextareaField.tsx`, `SelectField.tsx`, `DateField.tsx`, `DatalistField.tsx`, `PriceField.tsx`, `SearchField.tsx`, `CheckboxField.tsx`, `FieldError.tsx`) plus two non-executable supporting files (`field-icons.tsx` — a `FIELD_ICONS` constant table of React nodes; `types.ts` — type-only). `index.ts` re-exports only. The parent `test-coverage` change's tasks.md §3.3 line is explicit: "Elevate to `form-field-system` spec." That spec already exists at `openspec/specs/form-field-system/spec.md` (created by archiving `standardize-form-fields`); this sub-proposal therefore does NOT create a new capability spec — it locks the existing SHALLs against colocated test files, elevates the small set of non-trivial *call-time* invariants the new tests assert that are not yet in the spec (notably `PriceField`'s cents-as-integer math and the `<FormField>` known-display-name dev warning), and adds the carve-out bookkeeping under `testing-foundation` per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, `test-button-system`, `test-housekeeping`, `test-chip-system`) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation (per `test-housekeeping`), universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule (write the test OR `/* v8 ignore */` with named rationale — never lower the floor), `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Every requirement applies verbatim.
- `form-field-system` (active spec) — 19 active requirements covering token surface, 44px touch-target floor, chrome-on-wrapper rule, border/focus/error contrast floors, FormField label/id/aria orchestration, icon position grid, className-on-input prohibition, per-component contracts (TextField/TextareaField/SelectField/DateField/DatalistField/PriceField/SearchField/CheckboxField/FieldError), tooltip-as-error removal, single-mechanism required state, placeholder contrast, FIELD_ICONS registry, call-site migration completion, FormSelect deletion path, QuantityLimitField rename. The tests assert against these SHALLs as the authoritative contract. **No requirements are removed.** A small number of NEW invariants surfaced by reading the source at HEAD are elevated alongside the existing requirements — see "Capabilities" below.
- `button-system` — `form-field-system` mentions buttons only via the CheckboxField's box focus ring and the SearchField clear-button affordance. The clear button's `type="button"` and `aria-label="Clear search"` are owned by `form-field-system` (the SearchField requirement), not by `button-system`. No transitive SHALLs cross over.
- `popover-trigger-system`, `menu-system`, `segmented-control-system`, `chip-system` — unrelated; field primitives do not render through them.
- `items-browser-chrome`, `items-price-filter`, `list-item-management`, `list-visibility`, `following`, `home-digest` — these capabilities consume field primitives at call sites but their behavior contracts are not part of this carve-out. Capability-flow sub-proposals (4.x) own those.
- Cross-cutting design-system rule applies: this proposal is a spec-modification, NOT a page-scoped one-off. Fields are an interactive surface governed by a primitive-family spec; the spec edits here EXTEND the existing `form-field-system` spec with elevated call-time SHALLs surfaced during testing. No new size variant, no new sibling primitive. Cache-tag rule does NOT apply (no server reads).

## What Changes

- **NEW** colocated test files under `app/ui/components/field/__tests__/` (per `test-housekeeping`'s relocated convention). One `.test.tsx` per executable JSX component (under jsdom), no `.test.ts` (the field carve-out has no pure-helper files):

  - `FormField.test.tsx` — chrome wrapper, label/htmlFor wiring, useId-driven id/`aria-describedby`/`aria-invalid`/`aria-required` injection, icon-position grid switching (left default, right opt-in, no icon), required-indicator real-DOM rendering inside `<span aria-hidden="true">`, error path renders `<FieldError>` with linked id, description path renders `<p>` with linked id, both-description-and-error space-joined `aria-describedby`, `size="sm"` wrapper class, `className` forwarded to outer `.form_field_group` div, `Children.only` enforcement (one child), `KNOWN_CHILD_DISPLAY_NAMES` dev-warning path (`process.env.NODE_ENV !== 'production'` + unrecognized `displayName` → `console.error` is called; recognized displayName → no error; missing displayName → no error). Tests use the jsdom project's RTL setup.
  - `TextField.test.tsx` — renders a native `<input class="form_field_input">` inside `<FormField>`; `type` prop accepts `text|email|url|tel|password|search|number` (default `text`); `ref` forwarding lands on the underlying input; the input element receives the injected `id` and `aria-*` from `FormField`; `disabled` propagates to the input; pass-through input props (e.g. `placeholder`, `name`) reach the input.
  - `TextareaField.test.tsx` — renders `<textarea class="form_field_textarea">` inside `<FormField>`; `ref` forwarding; `disabled` propagation; pass-through props.
  - `SelectField.test.tsx` — renders `<select class="form_field_select">` inside `<FormField>`; `options` array renders an `<option>` per entry with matching `key`/`value`/`label`; `children` path (when `options` is undefined) renders the JSX children verbatim; `fieldSize="sm"` forwards to `<FormField size="sm">`; `ref` forwarding; `disabled` propagation.
  - `DateField.test.tsx` — renders `<input type="date" class="form_field_input">` inside `<FormField>`; `ref` forwarding; `min`/`max` pass-through.
  - `DatalistField.test.tsx` — renders `<input type="text" list={generatedId}>` + sibling `<datalist id={generatedId}>` with the caller-provided `options` ReactNode subtree; the `list` attribute and `<datalist id>` are equal AND non-empty AND derived from `useId`; two instances on the same render have distinct ids; `ref` forwarding to the input.
  - `PriceField.test.tsx` — renders the dollar icon (left, locked, `aria-hidden="true"`) and a transparent `<input type="text" inputMode="numeric">`; cents-as-integer math: typing `'1234'` displays `'12.34'` and calls `onChange(12.34)`; typing `'0'` displays `''` (empty), `onChange(0)`; `amount={null}` → display empty; `amount={12.34}` → display `'12.34'`; `allowNegative={false}` (default) suppresses negatives even if a `-` appears in input; `allowNegative={true}` + value containing `-` flips `isNegative` and yields `onChange(-x.xx)`; toggling negative off (trailing `-` while already negative) clears `isNegative`; the `Math.abs(amount)`/`isNegative` display path produces a leading `-` when current state is negative; `placeholder="0.00"` present; `disabled` propagates; `autoFocus`, `id`, `aria-label`, `className` pass through.
  - `SearchField.test.tsx` — renders `<div class="form_field search_field">` with the search icon and a `<input type="search">`; clear-button auto-rendered when `onClear` is provided AND `value` is non-empty; clear button has `type="button"`, `aria-label="Clear search"`, and clicking it invokes `onClear`; clear button NOT rendered when `value === ''`, NOT rendered when `value` is undefined, NOT rendered when `onClear` is undefined; `trailing` ReactNode REPLACES the auto clear button (`hasTrailingNode` short-circuits the clear-button branch even if `onClear` is also provided — though the discriminated-union type prevents that at the call site, the runtime contract still wins for `hasTrailingNode === true`); `no_trailing` class is added when neither trailing nor clear button renders; `ref` forwarding; `className` appended to the outer div.
  - `CheckboxField.test.tsx` — renders a `<label class="checkbox_field">` containing a native `<input type="checkbox" class="checkbox_field_box">` and a `<span>` with the label text; `className` is appended to the label classes (`"checkbox_field foo"`); `ref` forwarding; `disabled` propagation; pass-through props (`name`, `checked`, `onChange`).
  - `FieldError.test.tsx` — renders `<p id={id} class="field_error">{children}</p>` when `children` is truthy; renders `null` (no element produced) when `children` is `undefined`, `null`, `false`, `0` — every falsy case; no `role="alert"`, no `aria-live`.

- **NO** test for `index.ts` — re-export only, zero executable statements after TS erasure. Excluded under the existing `app/ui/components/*/index.ts` glob in `vitest.config.ts`'s `coverage.exclude` (no change needed).
- **NO** test for `types.ts` — type-only, zero executable statements after TS erasure. Already implicitly excluded (no runtime entry); confirm absence from the report.
- **NO** test for `field-icons.tsx` — module-scope literal table of pre-built ReactNodes (`{ name: <MdTextFields aria-hidden="true" />, … }`); the only "behavior" is "constant is exported with the expected keys". A purely-data assertion (`expect(Object.keys(FIELD_ICONS)).toEqual([...])`) is a tautology against the source. Disposition: exclude from coverage via a new entry in `vitest.config.ts`'s `coverage.exclude` (matching `app/ui/components/field/field-icons.tsx` explicitly, since the existing `index.ts` glob doesn't catch it). Rationale recorded in the entry. Alternative considered and rejected in design.
- **NEW** ESLint override in `eslint.config.mjs`: extend the existing per-file `sonarjs/cognitive-complexity = error` override array to include each of the ten executable field-component files (`FormField.tsx`, `TextField.tsx`, `TextareaField.tsx`, `SelectField.tsx`, `DateField.tsx`, `DatalistField.tsx`, `PriceField.tsx`, `SearchField.tsx`, `CheckboxField.tsx`, `FieldError.tsx`). Measured complexity at HEAD is in the 1–4 range for every file (the most complex is `PriceField`'s `handleChange` at ~4; `FormField`'s body at ~3–4); the override locks the ceiling at 15 for future edits.
- **NEW** per-file thresholds in `vitest.config.ts` for the ten executable files, each referencing the existing `COVERAGE_FLOOR` constant (one entry each — NO per-file numeric variation, per `test-housekeeping`'s single-constant rule).
- **NEW** exclude entry in `vitest.config.ts`'s `coverage.exclude`: `app/ui/components/field/field-icons.tsx` (constant table of ReactNodes; no behavior). The existing `app/ui/components/*/index.ts` pattern continues to cover `index.ts`.
- **MODIFIED** `form-field-system` spec — ADDS three new requirements that elevate call-time invariants the new tests lock against and that are currently implicit in the source:
  - `<FormField>` SHALL enforce `Children.only` and SHALL warn in development when the child's `displayName` is not in the known set (`TextField`, `TextareaField`, `SelectField`, `DateField`, `DatalistField`, `PriceField`). Locks the "use a field-type wrapper" contract that `FormField.tsx` already enforces but the spec does not state.
  - `<PriceField>` SHALL use cents-as-integer parsing math (`Number(value.replace(/\D/g, ''))` → integer cents → `cents / 100` → dollars). The display string SHALL be derived from `Math.abs(amount)` formatted to two fraction digits, prefixed with `'-'` when the internal `isNegative` state is true. The `allowNegative` prop defaults to `false`; when `false`, any `-` in input is suppressed; when `true`, a `-` flips `isNegative`, and a trailing `-` on a value that is already negative clears `isNegative`. Locks the toggle path the existing spec leaves underspecified.
  - `<SearchField>`'s trailing-slot SHALL be a discriminated union enforced at runtime: when `trailing` is provided as a non-null ReactNode, it renders in the trailing slot; when `trailing` is absent AND `onClear` is provided AND `value` is non-empty, an auto clear `<button>` is rendered; otherwise no trailing element is rendered and the outer div receives a `no_trailing` class. Locks the three-branch decision the existing spec describes only in two scenarios.
  These elevations are documented in `design.md` Decision 3.
- **NEW** `testing-foundation` carve-out bookkeeping record (sub-proposal-archive ONLY, per `test-coverage` design D13 two-tier rollup): the form-field-system primitive carve-out has landed at the universal `COVERAGE_FLOOR`, the `sonarjs/cognitive-complexity` promotion to `error` is in effect for the ten executable files, and subsequent sub-proposals consuming any field primitive inherit the assumption that those modules are tested and complexity-locked. Recorded in `openspec/changes/test-form-field-system/specs/testing-foundation/spec.md` ONLY (and at archive in `openspec/changes/archive/<date>-test-form-field-system/specs/testing-foundation/spec.md`) — does NOT roll into the parent's `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md` (which is created at test-coverage's archive time from the parent accumulator).
- **NEW** four-audit findings recorded in `tasks.md`. Carve-out is the largest primitive family yet under test (ten components). Anticipated dispositions: assertion-substance pass on the new tests; duplication likely produces a `__tests__/test-helpers.ts` with a small render harness covering the field-wrapper composition shape (≥3 use sites: TextField/TextareaField/DateField at minimum share the "render a `FieldWrapperProps`-shaped chrome over a single input" pattern); complexity audit expected to pass on all ten files; testability audit expected to surface one or two patterns (the `PriceField` `useState` internal state path; the `FormField` `Children.only` dev-only warning gated by `process.env.NODE_ENV`).
- **NO source refactors expected.** The ten components are testable as-shipped; every branch surfaces as observable DOM, observable callback shape, or observable `console.error` (for the dev-warning path).

## Capabilities

### New Capabilities

None. The `form-field-system` spec already exists.

### Modified Capabilities

- `form-field-system`: ADDS three requirements (FormField child-displayName warning, PriceField cents-math + negative-toggle path, SearchField trailing-slot three-branch decision) that elevate call-time invariants the new tests lock against. No existing requirements are removed or behavior-changed.
- `testing-foundation`: a carve-out bookkeeping record is written to this sub-proposal's archive-only delta (see "What Changes" above and `test-coverage` design D13). No change to the active `testing-foundation` spec; no change to the parent `test-coverage` accumulator.

## Impact

- **New files:** ten test files under `app/ui/components/field/__tests__/`, one per executable component. If duplication surfaces during the §7 audit, a `test-helpers.ts` joins them under the same `__tests__/` directory (excluded from coverage via the existing `**/__tests__/**` glob). No new shared helpers in `test/helpers/` (the foundation already provides RTL + jest-dom + jsdom shims).
- **Modified config:** `eslint.config.mjs` gains ten paths in the per-file override array. `vitest.config.ts` gains ten entries in the `thresholds` map, each pointing at `COVERAGE_FLOOR`; plus one entry in `coverage.exclude` for `field-icons.tsx`.
- **Modified source:** none expected. Any audit finding requiring source change is recorded with disposition in `tasks.md`; refactor-in-place commits become part of this change.
- **Modified specs:** `openspec/specs/form-field-system/spec.md` gains three ADDED requirements (FormField child contract, PriceField math contract, SearchField trailing-union contract). No removals. The carve-out bookkeeping requirement lives ONLY in this sub-proposal's archive directory (`openspec/changes/archive/<date>-test-form-field-system/specs/testing-foundation/spec.md`); it does NOT roll into the parent `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md` (per `test-coverage` design D13 two-tier rollup).
- **CI:** the existing four-gate workflow runs unchanged; the `test` job runtime grows by the ten new files (jsdom render tests; expected sub-second per file on average).
- **Dependencies:** none added.
- **Downstream unblock:** the parent `test-coverage` change's checkbox for sub-proposal 3.3 flips on archive. Capability-flow sub-proposals that render field primitives at their call sites (4.5 `test-items-browser-chrome` for items-toolbar search and filter inputs; 4.9 `test-list-item-management` for the item form and its `QuantityLimitField`; 4.11 `test-list-visibility` insofar as visibility settings render through `SelectField`; the price-filter behavior in `items-price-filter`) benefit from each field primitive being a known-good module with its contract elevated in the spec.
- **Risk:** moderate-ish for a primitive carve-out. Ten components vs. two for chip; one component (`PriceField`) carries non-trivial internal state and string-math. Realistic miss: failing to model the trailing-`-` `isNegative` clearing path in `PriceField.handleChange` (input `'12.34-'` while `isNegative=true` → branch (a)). The new spec requirement (Decision 3) is explicit on this; the test names this case as a discrete `<State>_<Behavior>` `it()`. Secondary risk: the `Children.only` + dev-warning path in `FormField` is gated by `process.env.NODE_ENV !== 'production'`; the test SHALL exercise it under the default jsdom env (vitest sets `NODE_ENV='test'`, which is `!== 'production'` and therefore activates the warning branch).
- **No runtime behavior change.**
