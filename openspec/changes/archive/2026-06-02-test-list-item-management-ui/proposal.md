## Why

Sub-proposal **4.9b** of the `test-coverage` initiative and the **UI half** of the `test-list-item-management` mid-flight split (GitHub issue #83). The action half — `app/actions/items.ts` / `app/actions/lists.ts` — was brought to the per-file `COVERAGE_FLOOR` by sibling sub-proposal 4.9a (`test-list-item-management`, issue #48, archived). That split was forced by the coverage mechanism: vitest per-file thresholds are whole-file, so the two server-action files had to reach the floor in one change. The UI is a disjoint set of files and was deferred here per the parent's "MAY split mid-flight → open a sibling issue" instruction. The parent `test-coverage/tasks.md` §4.9 checkbox flips only when BOTH 4.9a and this carve-out archive.

**Stakes: HIGH.** The two surfaces this carve-out covers — the claim/purchase modal and the `@dnd-kit/sortable` drag-reorder UX — sit directly on top of the now-floored action layer (claim under `quantity_limit`, `updatePriority` fractional reorder). A regression in the claim-modal flow branch can leak a spoiler or claim under the wrong identity; a regression in the reorder surface can call `updatePriority` with the wrong target and silently corrupt a shared gift list's order. These UI paths have **zero** test coverage today (23–24 files, ~2,500 LOC, including the 426-line `Item.tsx`).

This carve-out is the largest remaining `app/(main)/items/` UI bucket. It is the natural home for every item-management UI file that the sibling carve-outs explicitly deferred: `test-item-store-links` (4.4) names `Item.tsx` as "OWNED by 4.5 / 4.9" and `test-items-browser-chrome` (4.5) floored `Items.tsx` / `ItemsBrowser.tsx` while module-mocking `Item.tsx` and the `itemform/*` tree out of its scope. None of these files is enumerated in `vitest.config.ts` today (verified against the live thresholds map), so this carve-out claims and floors them.

Inherited constraints surfaced by spec-grep (every active `spec.md` was grepped for the component names, file paths, DOM strings, cache tags, and behaviors these files touch):

- **`testing-foundation`** (active accumulator at `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas through `test-items-library-shell`) — vitest 4.x runner with the jsdom/node two-project split (`.test.tsx` → jsdom, `.test.ts` → node), four-gate pre-merge, `__tests__/` colocation, the universal per-file `COVERAGE_FLOOR` (`lines:98 / statements:98 / branches:95 / functions:100`) referenced from the single constant in `vitest.config.ts`, the no-backdoor disposition rule (`/* v8 ignore */` with a named reason, never lower the floor), the assertion-substance bar, the `<State>_<Behavior>` `it()` shape, the three-role `describe()` convention, the four-audit + invariant-elevation obligations, and the `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Two load-bearing mocking allowances apply: **(1) NextAuth is the network boundary** — `@/lib/auth`'s `auth()` is mocked, never the modules it wraps; **(2) `next/navigation`'s `redirect()` is mocked to throw a sentinel** for page-RSC guards (the `FollowingPage.test.tsx` / `ItemsContainer.test.tsx` precedent). Every requirement applies verbatim.
- **`list-item-management`** (active, substantive — 13 requirements) — the **owning capability**. Its UI-side requirements are the authoritative contract these tests assert against: the choose-items page (owner-only affordance + redirect guards, library-with-membership-precheck, the add+remove save diff, re-add-loses-position, the URL-param filter/sort toolbar, client-side filter derivation, the `<Item preview />` + `<CheckboxField>` row primitive), the `returnTo` same-origin validation + navigation contract, the Archive-vs-Delete distinct-semantics requirement, and the Delete-confirmation dialog copy/button matrix. The action-side requirements (`createPurchase` / `removePurchase` identity rules, capacity, `updatePriority` algorithm) were locked by 4.9a against the action files; this carve-out asserts the **UI side** of the same contracts (which claim flow renders for which viewer; what the drag-end handler passes to `updatePriority`). Source-grep surfaces UI invariants no requirement currently states — candidates for elevation (see What Changes / `design.md`).
- **`server-endpoint-authorization`** (active, §4.13) — owns action/route authorization for `app/actions/**` + `app/api/**`. It does NOT own page-RSC shells. Per the `test-items-library-shell` (4.18) precedent, the **page-RSC redirect guards** in `ChooseItemsBody`, `ItemFormBody`, `SortItemsContainer`, and `ListItemsSection` (non-owner / unauthenticated → redirect) are owned **here**, not by §4.13. The image-search route's auth + 30/min rate-limit SHALLs stay with §4.13 / 5.1; this carve-out asserts only the UI-side consumption of that route. No SHALL is added to `server-endpoint-authorization`.
- **Primitive-family specs rendered *through*, not re-owned** — `form-field-system` owns `QuantityLimitField`'s `<TextField type="number">` + `<CheckboxField>` composition and the `ItemNameInput` / `StoreInput` field primitives; `popover-trigger-system` owns the `ListSelection` `.if-lp-trigger` → `<PopoverTrigger>` migration; `chip-system` owns the list-picker selected-list chips; `confirm-dialog-system` owns `ConfirmDialog` (rendered by `DeleteItemButton`); `button-system`, `menu-system`, `tooltip-system`, `loading-indicator-system`, `item-store-links` (`StoreLinks`, floored under 4.4), and `empty-state-system` own their respective primitives. All are already tested and floored; this carve-out renders through the **real** primitives (not mocked) to exercise the item-management integration, and asserts no primitive SHALL directly.

Cache-tag note: these UI files trigger the mutation side of the `'use cache'` reads in `lib/dal.ts` (via the server actions `setListItems`, `createItem`/`updateItem`, `archiveItem`/`deleteItem`, `createPurchase`/`removePurchase`, `updatePriority`). The actions own the `revalidateTag('items'|'lists')` obligation and were tested for it in 4.9a; the server actions are **module-mocked** in these component tests (the established `vi.mock('@/app/actions/*')` pattern), so the tests assert *which action the UI calls with which payload*, not the tag invalidation (which is 4.9a's contract). The server-component shells (`ChooseItemsBody`, `ItemFormBody`, `SortItemsContainer`, `ListItemsSection`) read through DAL/`auth()` and are tested via the async-RSC pattern (mock `auth()`, mock the DAL reads, sentinel `redirect()`).
