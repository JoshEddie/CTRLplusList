## Why

Sub-proposal 4.3 of the `test-coverage` initiative — a capability-flow carve-out (the third after 4.1 `test-app-frame` and the foundation/primitive work). The carve-out (per parent `test-coverage` `tasks.md` §4.3) is "DAL reads powering `app/(main)/HomePage.tsx`, page UI, recency sorting." The `home-digest` capability already has a substantive active spec at `openspec/specs/home-digest/spec.md` (created by archiving `add-following-and-history`, with 10 real requirements), so this sub-proposal does NOT create a new capability spec — it (a) locks the home-page composition + rail-capping + collapse + migration-toast behavior against colocated test files at the universal `COVERAGE_FLOOR`, (b) MODIFIES the spec's placeholder `Purpose`, (c) ADDS one SHALL for an invariant the source enforces today that no requirement locks (the migration toast's pre-hydration no-flash contract), and (d) adds the carve-out bookkeeping under `testing-foundation` Tier 2 per the parent's D13 two-tier rollup.

The defining feature of this carve-out is a **scope boundary that has to be drawn carefully**, because every DAL read the home rails consume is a *shared, multi-consumer* read owned by a more-specific sibling carve-out. Spec-grep + usage-grep findings:

- `app/(main)/HomePage.tsx` imports exactly ONE DAL read: `getUserIdByEmail`. The four rails each import their own read. So "DAL reads powering `HomePage.tsx`" resolves literally to `getUserIdByEmail` — the only DAL function the page module itself calls. This carve-out OWNS `getUserIdByEmail` (it is also consumed by ~20 other pages, but no sibling carve-out names it).
- `getListsByUser` (My Lists rail + `/lists` page + items pages + actions), `getBookmarkedListsByUser` (Bookmarks rail + `/lists/bookmarks`), `getVisitHistoryByUser` (Recently-visited rail + `/lists/history`), and `getFollowingFeedUsers` (Following rail + `/following`) are all consumed by sibling-owned routes. Their read contracts (notably the SQL `ORDER BY` recency sort) belong to **4.6 `test-list-collections`** (`getListsByUser`, `getBookmarkedListsByUser`), **4.14 `test-visit-history`** (`getVisitHistoryByUser`), and **4.2 `test-following`** (explicitly "DAL `getFollowing*`"). This carve-out does NOT assert those reads' sort contracts; it exercises the *rails' own* capping/mapping/empty-state logic against seeded data.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active accumulator delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation-spike`, `test-foundation`, `test-pure-libs`, `test-housekeeping`, the six primitive carve-outs, and `test-app-frame`) — runner (vitest 4.x, two projects: jsdom for `.test.tsx`, node for `.test.ts`/DB integration), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit + invariant-elevation obligations, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy, **NextAuth-as-network-boundary mocking allowance**, and the **"DAL functions SHALL NOT be mocked — integration tests SHALL exercise them against the real test database"** rule. Every requirement applies verbatim.
- `home-digest` (active spec) — 10 active requirements covering the four-rail digest, per-rail caps, sort orders, collapse + `localStorage` persistence, the migration toast, horizontal-scroll layout, card slots, hover behavior, and the trailing "+N more" tile. The tests assert against the structural + behavioral SHALLs this carve-out can isolate; the CSS-layout SHALLs (card widths 236/260/190px, horizontal scroll, hover appearance) are NOT unit-asserted (jsdom computes no layout — same disposition as `test-app-frame` Decision 9 for `app-frame` R4/R5).
- `button-system` — `CollapsibleRail.tsx` renders a `<LinkButton variant="link">` for "See all". Token-surface link only; the primitive is already tested under `test-button-system` (archived) and renders real in jsdom. No `button-system` SHALL is asserted from here.
- `list-collections` — OWNS the card primitives the rails render through (`ListCard.tsx`, `ListCardRow.tsx`, `MoreCard.tsx`) and the `/lists`, `/lists/bookmarks` routes. `visit-history` OWNS `HistoryCard`. `following` / `users` OWN `UserCard`. This carve-out does NOT render those child components in its rail tests (it inspects the React element the rail returns — the props handed to the child — rather than rendering the child), so no sibling primitive's contract is asserted here.
- DB-driver constraint (`drizzle-orm/neon-http`, no interactive transactions) — irrelevant to this carve-out (all reads, no mutations).

## What Changes

- **NEW** colocated test files under `__tests__/` directories mirroring the source layout. Nine test files in total:

  **Node project (`.test.ts` — DB integration / async-server-component prop-inspection; no jsdom render):**

  - `lib/__tests__/getUserIdByEmail.test.ts` — the FIRST DAL-against-PGlite integration test in the repo. Boots PGlite via `test/helpers/db.ts`, swaps the `@/db` connection to the PGlite instance (the DB-under-test mechanism the `test-foundation-spike` chose — this swaps the *driver/connection*, NOT the DAL function, so it honors the "DAL SHALL NOT be mocked" rule), seeds `users` rows, and asserts: a matching email returns the full `users` row; a non-matching email returns `null`; an empty result set returns `null` (the `result[0] || null` branch). The React `cache()` wrapper (not `'use cache'`) needs no `next/cache` mock.

  - `app/(main)/__tests__/HomePage.test.ts` — async server component. `@/lib/auth`'s `auth()` is mocked (NextAuth network-boundary allowance); `next/navigation`'s `redirect` is mocked; `getUserIdByEmail` runs against seeded PGlite (real DAL read). Asserts: when `auth()` resolves to a session with no `user.email`, `redirect('/sign-in')` is called and no digest tree is returned; when the email resolves to no `users` row (`getUserIdByEmail` → `null`), `redirect('/sign-in')` is called; when a viewer resolves, the returned element tree contains, in source order, `<BookmarkMigrationToast>`, then four `<CollapsibleRail>` elements with the exact `name` / `title` / `seeAllHref` props (`my-lists`→`/lists`, `following`→`/following`, `bookmarks`→`/lists/bookmarks`, `recently-visited`→`/lists/history`), each wrapping a `<Suspense>` whose child is the matching rail element with `userId={viewer.id}`, with three `home-rail-divider` separators interleaved. Assertions inspect the returned element's `type`/`props` (no render — the rails inside are element descriptors, never invoked).

  - `app/(main)/lists/ui/components/rails/__tests__/MyListsRail.test.ts` — async server component, integration. Seeds N owned lists in PGlite, invokes `await MyListsRail({ userId })`, inspects the returned `<ListCardRow>` element's props: when N > 5, `lists.length === 5` and `moreCount === N − 5` and `seeAllHref === '/lists'`; when N ≤ 5, `lists.length === N` and `moreCount === 0`; when N = 0, `lists.length === 0` and the `emptyMessage` prop is `"No lists yet. Create your first one."`. Does NOT assert sort order (sibling-owned — see scope note).

  - `app/(main)/lists/ui/components/rails/__tests__/BookmarksRail.test.ts` — as above for `getBookmarkedListsByUser`; additionally asserts the rail's bespoke field-mapping (`list_id`→`id`, `list.name`→`name`, `list.subtitle ?? null`→`subtitle`, `list.occasion`/`date`/`user`) on the `lists` prop, the `showOwner` prop, `moreCount`, and `seeAllHref === '/lists/bookmarks'`.

  - `app/(main)/lists/ui/components/rails/__tests__/FollowingRail.test.ts` — as above for `getFollowingFeedUsers`; the rail returns a `<div className="list-card-row">` tree (not a single card component). Asserts: empty → `<div className="list-card-row-empty">Not following anyone yet.</div>`; populated → the mapped `<UserCard>` children count caps at 5, each receives `user`/`newCount`/`latestSharedAt`/`compact` props, and a trailing `<MoreCard moreCount href="/following">` appears iff total > 5.

  - `app/(main)/lists/ui/components/rails/__tests__/RecentlyVisitedRail.test.ts` — as above for `getVisitHistoryByUser`. Asserts: empty → `<div className="list-card-row-empty">No visits yet.</div>`; populated → mapped `<HistoryCard row>` children cap at 5 with the trailing `<MoreCard href="/lists/history">` iff total > 5; AND asserts the rail calls `getVisitHistoryByUser` with the over-fetch argument `{ limit: 50 }` (the home-digest-specific fetch bound — see Decision 5).

  **jsdom project (`.test.tsx` — RTL render):**

  - `app/(main)/__tests__/page.test.tsx` — the route shell. Mocks `./HomePage` (async server component, can't render in jsdom) to a sync stub; renders `<Page />`; asserts the outer `<main className="container">` wraps the `HomePage` stub, and the `<Suspense>` `fallback` is `<LoadingIndicator size="page" />`.

Renders `<CollapsibleRail name="x" title="T" seeAllHref="/h">child</CollapsibleRail>`; asserts: default-open (`localStorage` empty → `.rail` without `collapsed`, `aria-expanded="true"`, body rendered); clicking `.rail-toggle` writes `localStorage['home.rail.x.open'] = 'false'`, flips `aria-expanded`, adds `collapsed`, and unmounts the `.rail-body`; reload with `localStorage['home.rail.x.open'] = 'false'` renders collapsed; the `seeAllHref` renders a `<LinkButton variant="link">` "See all" anchor with `href="/h"`; omitting `seeAllHref` renders no "See all"; `headerExtra` renders into `.rail-header-extra`; the `localStorage` write throwing (quota) is swallowed (the `try/catch` branch).

## Impact

- **New files:** nine `.test.ts`/`.test.tsx` files under `__tests__/` directories, plus `capRail.test.ts`; one new source helper `app/(main)/lists/ui/components/rails/capRail.ts`. No shared `test-helpers` extracted on first authoring (the `@/db`→PGlite seed harness stays inline in the DB-integration tests until a second carve-out needs it — duplication audit §10.2 records the deferral).
- **Modified source:** the four rail components are refactored to call `capRail` (behavior preserved, locked by both the rail integration tests and `capRail.test.ts`). No other source change.
- **Modified config:** `eslint.config.mjs` gains the carve-out paths in the per-file `sonarjs/cognitive-complexity = error` override; `vitest.config.ts` gains per-file `thresholds` entries (one per executable file) referencing `COVERAGE_FLOOR`. `lib/dal.ts` is deliberately NOT enumerated (deferred finding 1). The node project's `esbuild` JSX setting may need `jsx: 'automatic'` so `.test.ts` files can import and prop-inspect the `.tsx` rail/page components without rendering — a one-line test-infra config addition within the carve-out's authority if the default transform does not handle it.
- **Modified specs:** `openspec/specs/home-digest/spec.md` gains a real `Purpose` and ONE ADDED requirement. The carve-out bookkeeping lives ONLY in this sub-proposal's archive directory (Tier 2); it does NOT roll into the parent accumulator or the active `testing-foundation` spec.
- **Modified governance:** three new checkboxes in `openspec/changes/test-coverage/tasks.md` (deferred findings 1–3). The §4.3 checkbox flips on archive of this sub-proposal, not at apply.
- **CI:** the existing four-gate workflow runs unchanged; the `test` job grows by the nine new files. The DB-integration tests boot PGlite per-suite (the harness already exists); the home/rail tests are prop-inspection (no render) so they are cheap.
- **Dependencies:** none added.
- **Downstream unblock:** the parent `test-coverage` §4.3 checkbox flips on archive. The DAL-against-PGlite integration pattern this carve-out establishes is reused by every later DAL carve-out (4.2/4.6/4.9/4.11/4.14). Sibling carve-outs that own the rails' reads (4.2/4.6/4.14) inherit the rail-capping behavior as known-good.
- **Risk:** medium. Notable risks (detailed in `design.md`):
  - **First DAL-against-PGlite test.** Swapping the module-level `@/db` const to a per-suite PGlite instance (`vi.mock('@/db', ...)` with a getter over a mutable holder set in `beforeEach`) is a new pattern. Mitigation: the `test/helpers/db.ts` `bootPglite` harness already exists and is exercised by `db.test.ts`; this carve-out only adds the connection-swap glue.
  - **Async-server-component prop-inspection in the node project.** Importing a `.tsx` component into a `.test.ts` (node project, no `react()` plugin) and inspecting the returned element relies on vitest's esbuild transforming the JSX. Mitigation: set the node project's `esbuild.jsx = 'automatic'` if the default does not; documented in `design.md` Decision 4.
  - **Scope boundary defensibility.** A reviewer might expect this carve-out to test the rails' DAL reads' sort orders. Mitigation: the boundary is stated explicitly (reads are sibling-owned; this carve-out tests the rails' capping/mapping/empty-state via integration against seeded data, and defers the sort-contract assertions + the discovered sort drift to the owning siblings).
  - **`lib/dal.ts` coverage deferral.** Not gating `dal.ts` per-file is a deviation from "enforce coverage floors ONLY on files in the carve-out". Mitigation: documented as deferred finding 1 with a governance checkbox; the deviation is forced by vitest's per-file (not per-function) gate against a shared 708-line module.
- **No runtime behavior change** other than the `capRail` extraction (behavior-preserving).
