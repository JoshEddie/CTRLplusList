## Why

`standardize-buttons` consolidated the button surface and absorbed the chips/text-link/toggle gaps along the way, but explicitly deferred three popover-or-radiogroup patterns whose ARIA models and keyboard semantics don't fit `<Button>`: dropdown menus, segmented controls, and form-input-shaped popover triggers. They're scattered across ~6 wrapper components today as parallel one-off implementations (`.menu-item` rows in `ListActionsMenu` and `UserAvatarPopover`; `.visibility-option` radio buttons in `VisibilityPicker`; `.view-toggle-btn` pair in `ItemsToolbar`; `.store-filter-trigger` / `.items-toolbar-filters-trigger` / `.if-lp-trigger` triggers across the filter popovers and list-selection picker). Each reimplements its own keyboard handling (arrow navigation, Home/End, Escape, focus return), click-outside detection, focus-trap concerns, and visual treatment, with the usual drift between copies â€” and the wrappers themselves are tangled with their internals because there's no primitive to extract. A single follow-up change introduces the three missing primitives plus re-shapes each wrapper into the StoreLinks pattern (wrapper owns state + orchestration; internals are system primitives), closing the gap left by `standardize-buttons` rather than letting these surfaces continue to diverge.

## What Changes

- Introduce a `<Menu>` primitive family at `app/ui/components/menu/`: `<Menu>` (popover container managing `role="menu"`, click-outside, Escape, focus return, arrow-key navigation between items), `<MenuItem>` (`<button role="menuitem">` with optional `icon` slot and `tone: 'default' | 'danger'`), `<MenuLinkItem>` (Next `<Link role="menuitem">` with the same tone API), and a shared `menuClasses()` builder.
- Introduce a `<SegmentedControl>` + `<SegmentedOption>` primitive family at `app/ui/components/segmented-control/`. `<SegmentedControl>` is a `role="radiogroup"` container managing arrow-key navigation between options. `<SegmentedOption>` is a `role="radio"` option with `aria-checked` semantics. Supports a `tone` prop (`'light' | 'on-dark'`) for the two surface contexts. Consumes `buttonClasses()` internals for focus-ring / hover-guard / min-touch consistency.
- Introduce a `<PopoverTrigger>` primitive at `app/ui/components/popover-trigger/`. Renders a `<button>` styled as a form-input-shaped dropdown (border, optional left icon, label, optional count badge, chevron right). Active state when filters/selections are applied. Used as the open-the-popover affordance for filter popovers and list-selection dropdowns. The popover *body* (the floating panel) stays page-scoped â€” only the trigger is unified, because the body's contents (search input + checklist for stores, two number inputs for price, listbox for list-selection) are too specialized to share.
- **Re-shape `ListActionsMenu.tsx`** into the StoreLinks pattern: keep it as the orchestrator that owns the open/close state and the delete-confirmation modal; replace its `.menu-item` rows with `<MenuItem>` / `<MenuLinkItem>` and its dropdown container with `<Menu>`. Remove `.menu-item`, `.menu-item-danger`, `.menu-dropdown` page-scoped CSS.
- **Re-shape `UserAvatarPopover.tsx`** similarly: orchestrator owns the avatar trigger and signed-in user header; replace `.avatar-popover-item` rows with `<MenuItem>` / `<MenuLinkItem>` and `.avatar-popover` container with `<Menu>`. Remove `.avatar-popover-item`, `.avatar-popover-divider`, `.avatar-popover-form` page-scoped CSS.
- **Re-shape `VisibilityPicker.tsx`**: orchestrator owns the state machine (private / unlisted / public transitions, optimistic updates, server-action coordination); replace the two `.visibility-option` `<button role="radio">` elements with `<SegmentedControl><SegmentedOption>` Ă— 2. The "Show in followers' feed" native checkbox stays as-is (it's a checkbox, not a button-family primitive). Remove `.visibility-toggle`, `.visibility-option`, `.visibility-option-label` page-scoped CSS.
- **Re-shape `ItemsToolbar.tsx` view-toggle**: replace `.view-toggle` + `.view-toggle-btn` pair with `<SegmentedControl><SegmentedOption>` Ă— 2 (grid / list). Same primitive, second caller. Remove `.view-toggle` / `.view-toggle-btn` page-scoped CSS.
- **Re-shape `StoreFilterPopover.tsx` and `PriceFilterPopover.tsx`**: orchestrators own their popover state (search query, local input state, apply/clear/done handlers, click-outside-with-apply for price); the `.store-filter-trigger` `<button>` becomes `<PopoverTrigger>`. The footer buttons (Clear, Done, Apply) migrate to `<Button variant="ghost" size="sm">` / `<Button variant="primary" size="sm">`. The popover panel itself stays page-scoped. Remove `.store-filter-trigger`, `.store-filter-clear`, `.store-filter-done` page-scoped CSS.
- **Re-shape `ItemsToolbar.tsx` filters-trigger** (the mobile "Filters" sheet trigger): `.items-toolbar-filters-trigger` migrates to `<PopoverTrigger>`. Same primitive, third caller. Remove `.items-toolbar-filters-trigger` page-scoped CSS.
- **Re-shape `ListSelection.tsx` dropdown trigger**: `.if-lp-trigger` migrates to `<PopoverTrigger>`. Fourth caller of the primitive. The `.if-lp-opt` `role="option"` listbox rows stay page-scoped â€” they're listbox options inside a `role="listbox"` container, a different ARIA pattern from menus, and abstracting a `<ListboxOption>` for one caller is premature. Remove `.if-lp-trigger` page-scoped CSS but keep `.if-lp` (wrapper) and `.if-lp-menu` / `.if-lp-opt` (listbox body) page-scoped.
- Document the boundary between `<Menu>` (action-list popover, `role="menu"`) and listbox (option-selection popover, `role="listbox"`) in design.md, with a forward-reference to a possible future `<Listbox>` primitive if a second listbox caller appears.
- Out of scope: a `<Listbox>` primitive (one caller â€” `ListSelection`'s dropdown rows). A `<Dialog>` primitive (the codebase already has `Modal.tsx` / `ConfirmDialog.tsx`; modal refactors are a separate change). A `<Tooltip>` primitive. Anything that requires React's `createPortal` for z-index escape (the existing menus and popovers all live inside their wrapper's DOM subtree â€” if a real "popover must escape overflow:hidden" need surfaces later, that's a follow-up).
- Dependencies: this change MUST land after `standardize-buttons`. The new primitives consume `buttonClasses()`, the `--btn-*` token surface, and the focus-ring / hover-guard / min-touch contract established there.

## Capabilities

### New Capabilities
- `menu-system`: Defines the visual and accessibility contract for popover menus â€” the `<Menu>` container, `<MenuItem>` / `<MenuLinkItem>` rows, role semantics (`role="menu"` / `role="menuitem"`), keyboard model (arrow keys, Home/End, Escape, focus return), tone variants (default / danger), the menu/listbox-vs-menu boundary decision, and the shared `menuClasses()` class builder.
- `segmented-control-system`: Defines the visual and accessibility contract for segmented controls â€” the `<SegmentedControl>` radiogroup container, `<SegmentedOption>` radio options, role semantics (`role="radiogroup"` / `role="radio"`), keyboard model (arrow keys, no tab between options), the two surface tones (`light` for white-card surfaces, `on-dark` for hero/nav surfaces), and the shared `segmentedClasses()` class builder.
- `popover-trigger-system`: Defines the visual and accessibility contract for form-input-shaped popover-opening buttons â€” the `<PopoverTrigger>` component, its props (`icon`, `label`, `count`, `active`, `chevron`), `aria-haspopup` + `aria-expanded` wiring, and the boundary between trigger (unified) and popover body (page-scoped).

### Modified Capabilities
- `button-system`: No requirement changes â€” but the new primitives consume `buttonClasses()` internals and the `--btn-*` token surface. This dependency is documented but doesn't modify the `button-system` contract.

## Impact

- **New components**: `app/ui/components/menu/` (Menu, MenuItem, MenuLinkItem, menuClasses, types, index). `app/ui/components/segmented-control/` (SegmentedControl, SegmentedOption, segmentedClasses, types, index). `app/ui/components/popover-trigger/` (PopoverTrigger, triggerClasses, types, index).
- **Wrapper re-shapes**: `ListActionsMenu.tsx`, `UserAvatarPopover.tsx`, `VisibilityPicker.tsx`, `ItemsToolbar.tsx` (view-toggle and filters-trigger), `StoreFilterPopover.tsx`, `PriceFilterPopover.tsx`, `ListSelection.tsx`. Each retains its state ownership and orchestration; only internals change.
- **CSS removed (after migration)**: `.menu-item`, `.menu-item-danger`, `.menu-dropdown` from `app/(main)/lists/ui/styles/list.css`; `.avatar-popover-item`, `.avatar-popover`, `.avatar-popover-divider`, `.avatar-popover-form`, `.avatar-popover-item-button` from auth styles; `.visibility-toggle`, `.visibility-option`, `.visibility-option-label` from list styles; `.view-toggle`, `.view-toggle-btn` from items styles; `.store-filter-trigger`, `.store-filter-clear`, `.store-filter-done` from `store-filter.css` (or wherever they live); `.items-toolbar-filters-trigger`, `.items-toolbar-filters-badge` from items-toolbar CSS; `.if-lp-trigger`, `.if-lp-trigger-label` from item-form CSS. **Kept**: `.menu-trigger` page-scoped circular override (already documented WCAG exception in `standardize-buttons`); `.store-filter-panel` / `.price-filter-panel` / `.store-filter-search` / `.store-filter-list` / `.store-filter-item` / `.store-filter-footer` (popover bodies â€” page-scoped); `.if-lp` / `.if-lp-menu` / `.if-lp-opt` (listbox body and options).
- **Dependency**: `standardize-buttons` must be merged before this change ships. The new primitives import from `app/ui/components/button/` (specifically `buttonClasses` for focus/hover/min-touch reuse) and consume the `--btn-*` token surface and the pressed-state CSS extended in `standardize-buttons`'s Decision 11.
- **a11y improvements**: every wrapper currently reimplements its own keyboard handling â€” some incompletely. `ListActionsMenu`'s dropdown has no arrow-key navigation; `VisibilityPicker`'s segmented control doesn't use `role="radiogroup"`'s expected arrow-key model; `UserAvatarPopover`'s items don't have arrow-key navigation. Migrating to the primitives fixes these systematically.
- **Risk**: large keyboard-model surface to get right in one change. Mitigation: build one primitive at a time, slice-migrate one wrapper, gut-check keyboard model in preview before sweeping the rest (same slice-and-sweep cadence as `standardize-buttons`).
- **Out of scope**: `<Listbox>` primitive (one caller); `<Dialog>` primitive (existing `Modal.tsx` already covers); `<Tooltip>`; portal-based escape from overflow:hidden contexts (no current caller needs it); restyling the popover body interiors beyond what the trigger and footer-button migrations naturally touch.
