## Why

Sub-proposal 4.12 of the `test-coverage` initiative — the PWA-shell carve-out named in the parent `test-coverage` change's `tasks.md` §4.12: "`app/manifest.ts` shape, `ServiceWorkerRegistration.tsx`. Service worker (`app/sw.ts`) is excluded from unit coverage; covered by 6.2." Two executable source files are in scope:

- `app/manifest.ts` — the `MetadataRoute.Manifest` factory that drives the installability contract (name, icons, theme, display mode).
- `app/ui/components/ServiceWorkerRegistration.tsx` — the `'use client'` component, mounted once in the root layout, that registers `/sw.js` at scope `/` after first paint.

`app/sw.ts` (the Serwist service-worker entry: precache + kill-switch) is **excluded** from this carve-out — its behavior (HTML-bypass, precache, `KILL_SW`) is browser-runtime and is covered by sub-proposal 6.2 `test-e2e-pwa-offline`. This carve-out tests only the two files whose contracts are observable in jsdom / node without a real service-worker runtime.

The `pwa-shell` capability already has a substantive active spec at `openspec/specs/pwa-shell/spec.md` (six requirements covering manifest shape, SW scope/precache, kill-switch, iOS Apple meta tags, and dev-disable). This sub-proposal therefore does NOT create a new capability spec — it (a) locks the existing R1 manifest-shape SHALL against a colocated test at the universal `COVERAGE_FLOOR`, (b) ADDS one SHALL for an invariant the source enforces today that no requirement currently locks (the `ServiceWorkerRegistration` client-side registration contract — path, scope, feature-detection guard, error-swallowing, null render), and (c) records the carve-out bookkeeping under `testing-foundation` Tier 2 per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep across every active `openspec/specs/*/spec.md`:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus the archived deltas rolled in from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, the six primitive carve-outs, `test-housekeeping`, and `test-app-frame`) — runner (vitest 4.x, two-project split: `.test.tsx`→jsdom, `.test.ts`→node), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule (write a test → refactor in place → `/* v8 ignore */` with rationale), `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Every requirement applies verbatim.
- `pwa-shell` (active spec) — six requirements. This carve-out asserts against R1 (manifest shape) directly and ADDS one SHALL for the `ServiceWorkerRegistration` component contract. R2 (SW serves only non-HTML), R3 (kill-switch), and R6 (dev-disable in `next.config`) govern `app/sw.ts` / `next.config` and are OUT of carve-out (6.2 territory). R4 (iOS Apple meta tags) governs `app/layout.tsx` — out of carve-out and coverage-excluded; see the noted drift in "What Changes".
- Cross-cutting design-system rule: `ServiceWorkerRegistration` and `manifest` render no interactive surfaces (the component returns `null`; the manifest is data), so no primitive-family spec is touched.
- Cache-tag rule does NOT apply: neither file performs a `'use cache'` server read or a `revalidateTag` mutation.

## What Changes

- **NEW** colocated test files under `__tests__/` directories mirroring the source layout. Two test files, one per executable source file:

  - `app/__tests__/manifest.test.ts` (**node project** — `manifest.ts` is a pure, DOM-free function, so its test runs under the node project, which also lets it read `public/icons/` from disk). Invokes `manifest()` once and asserts the returned object against the R1 contract: `name === 'Ctrl+List'`, `short_name === 'Ctrl+List'`, `description` equal to the root metadata description string (`'Create and share your lists with friends and family'`), `id === '/'`, `start_url === '/'`, `scope === '/'`, `display === 'standalone'`, `orientation === 'portrait'`, `background_color === '#25194e'`, `theme_color === '#25194e'`; the `icons` array contains exactly four entries — `192x192`/`any`, `192x192`/`maskable`, `512x512`/`any`, `512x512`/`maskable` — each with `type === 'image/png'`; the `src` of each icon (`/icons/icon-192.png`, `/icons/icon-512.png`) resolves to a file that EXISTS under `public/icons/` (asserted via `node:fs` `existsSync`, locking the R1 scenario clause "src values resolve to existing files under public/icons/"); and the manifest declares NO `gcm_sender_id` or any Web-Push field (per R1's negative clause). The single `manifest()` invocation gives 100% coverage of the pure factory.

  - `app/ui/components/__tests__/ServiceWorkerRegistration.test.tsx` (**jsdom project**). Renders `<ServiceWorkerRegistration />` after stubbing `navigator.serviceWorker`. Asserts: the component renders nothing (`container.firstChild === null`); when `'serviceWorker' in navigator` and `register` resolves, `navigator.serviceWorker.register` is called exactly once with `('/sw.js', { scope: '/' })` (locks the path + scope contract); when `register` REJECTS, the rejection is swallowed (no unhandled-rejection — the `.catch(() => {})` path is exercised and the test does not throw); when `serviceWorker` is absent from `navigator` (feature-detection guard false), `register` is never called and the effect no-ops; the registration fires inside the mount effect exactly once (not on rerender — the `[]` dep array). `navigator.serviceWorker` is stubbed via `Object.defineProperty(navigator, 'serviceWorker', { configurable: true, value: { register: vi.fn() } })` and reset in `afterEach`; the absent-API case is exercised by `delete (navigator as { serviceWorker?: unknown }).serviceWorker` (or redefining to `undefined`).

- **SOURCE REFACTOR (in-place, disposition (b))** — `app/manifest.ts`: `name: 'ctrl+List'` and `short_name: 'ctrl+List'` are corrected to `'Ctrl+List'` (capital C). This is a brand-casing drift: the manifest is the SOLE place in the app source using the lowercase `'ctrl+List'`; every other brand string — `app/layout.tsx`'s `title` / OpenGraph `title` / Twitter `title` / `apple-mobile-web-app-title`, and `AppLogo` / `Logo` / `SignInPage` / `UserMenu` image `alt` + `aria-label` — uses `'Ctrl+List'`, and the active `pwa-shell` spec R1 explicitly mandates `name: 'Ctrl+List'` / `short_name: 'Ctrl+List'`. The source manifest is the outlier (the name that surfaces on the OS install prompt / home-screen label would render with inconsistent casing). The first test draft locking `'ctrl+List'` would fail the assertion-substance audit (it would lock a brand bug); disposition (b) refactor-in-place fixes the source to match the authoritative brand + the existing spec, and the test asserts the FIXED value `'Ctrl+List'`. **No `pwa-shell` R1 spec modification** — the spec is already correct; only the source drifted to it.

- **SOURCE REFACTOR (in-place, disposition (b))** — `app/ui/components/ServiceWorkerRegistration.tsx`: the guard `if (typeof window === 'undefined' || !('serviceWorker' in navigator)) return;` has a dead operand. The check lives INSIDE a `useEffect` callback, which never executes during server rendering — `window` is therefore always defined when the guard runs, making the `typeof window === 'undefined'` operand unreachable in any environment (browser or jsdom). Default disposition (b): drop the dead operand, leaving `if (!('serviceWorker' in navigator)) return;`. Both branches of the remaining guard are then directly testable (stub present → register called; stub absent → no-op), yielding 100% branch coverage with no `/* v8 ignore */`. Alternative (c) — keep the operand and annotate `/* v8 ignore next -- SSR short-circuit; useEffect never runs server-side, window is always defined */` — is recorded in `design.md` as the fallback if a reviewer prefers retaining the defensive guard. Both behaviors are exercised by the new test.

- **NEW** ADDED requirement in the `pwa-shell` spec — the `ServiceWorkerRegistration` client-side contract (registers `/sw.js` at scope `/` when `'serviceWorker' in navigator`, no-ops otherwise, swallows registration rejection, renders `null`). The existing R2 prose describes the service worker's emission, scope, and precache policy and its scenario "SW is registered after first visit" describes the BROWSER outcome (`navigator.serviceWorker.controller` non-null) — but no requirement locks the React component's own contract (the path string, the scope option, the feature-detection guard, the error-swallow, the null render). See `design.md` Decision 3 and `specs/pwa-shell/spec.md`.

- **NEW** per-file thresholds in `vitest.config.ts` for the two files, each referencing the existing `COVERAGE_FLOOR` constant (no per-file numeric variation, per `test-housekeeping`'s single-constant rule). **`app/manifest.ts` is REMOVED from `vitest.config.ts`'s `coverage.exclude`** — it was excluded as a no-behavior config object, but R1's field-value contract is a substantive, testable shape; the carve-out elevates it to a covered file. `app/sw.ts` STAYS in `coverage.exclude` (out of carve-out; 6.2 covers it via e2e).

- **NEW** ESLint override in `eslint.config.mjs`: extend the existing per-file `sonarjs/cognitive-complexity = error` override array to include `app/ui/components/ServiceWorkerRegistration.tsx` and `app/manifest.ts`. Measured complexity at HEAD is trivial for both (≤2) — well under the ceiling of 15. The override locks the ceiling for future edits, matching the per-carve-out policy.

- **NO** test for `app/sw.ts` — out of carve-out; covered by 6.2 `test-e2e-pwa-offline`.
- **NO** test for `app/layout.tsx` — the call site that renders `<ServiceWorkerRegistration />` and declares the manifest/Apple-meta convention is a layout file, coverage-excluded, and out of carve-out.
- **NOTED DRIFT (out of scope, not fixed here):** `pwa-shell` R4's scenario asserts `<meta name="apple-mobile-web-app-status-bar-style" content="default">`, but `app/layout.tsx` declares `'black-translucent'`. This drift lives in `layout.tsx` (coverage-excluded, out of carve-out) and is left to whatever sub-proposal owns layout-metadata coverage; flagged here so the reviewer knows it was observed and consciously deferred, not missed.
- **NEW** `testing-foundation` carve-out bookkeeping record (sub-proposal-archive ONLY, per `test-coverage` design D13 two-tier rollup): the `pwa-shell` carve-out's two files have landed at the universal `COVERAGE_FLOOR`, the `sonarjs/cognitive-complexity` promotion to `error` is in effect for them, and `app/manifest.ts` has been removed from the coverage-exclude list. Recorded in `openspec/changes/test-pwa-shell/specs/testing-foundation/spec.md` ONLY — does NOT roll into the parent `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md`.

## Capabilities

### New Capabilities

None. The `pwa-shell` capability spec already exists with six substantive requirements.

### Modified Capabilities

- `pwa-shell`: ONE new requirement is ADDED (the `ServiceWorkerRegistration` client-side registration contract). No requirements are MODIFIED or REMOVED — R1 (manifest shape) is asserted as-is (the source is refactored to match it, not the reverse), and R2/R3/R4/R6 remain unchanged. The R1 manifest contract is now regression-locked by a colocated test.
- `testing-foundation`: a carve-out bookkeeping record is written to this sub-proposal's archive-only delta (Tier 2 per `test-coverage` design D13). No change to the active `testing-foundation` spec; no change to the parent `test-coverage` accumulator.

## Impact

- **New files:** two — `app/__tests__/manifest.test.ts` (node) and `app/ui/components/__tests__/ServiceWorkerRegistration.test.tsx` (jsdom). No shared `test-helpers` extracted (the two files share no setup — one stubs `node:fs`/none, the other stubs `navigator.serviceWorker`).
- **Modified config:** `vitest.config.ts` — `app/manifest.ts` removed from `coverage.exclude`; two new entries added to the `thresholds` map (`app/manifest.ts`, `app/ui/components/ServiceWorkerRegistration.tsx`), each pointing at `COVERAGE_FLOOR`. `eslint.config.mjs` — two paths added to the per-file `sonarjs/cognitive-complexity = error` override array.
- **Modified source:** `app/manifest.ts` (brand-casing fix ×2) and `app/ui/components/ServiceWorkerRegistration.tsx` (drop dead `typeof window` operand). Both in-place inside the carve-out; both behaviors locked by the new tests.
- **Modified specs:** `openspec/specs/pwa-shell/spec.md` gains ONE ADDED requirement (the `ServiceWorkerRegistration` client-side contract). No removals, no modifications to existing requirements. The carve-out bookkeeping requirement lives ONLY in this sub-proposal's `specs/testing-foundation/spec.md` (and, at archive, under `openspec/changes/archive/<date>-test-pwa-shell/`); it does NOT roll into the parent `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md`.
- **CI:** the existing four-gate workflow runs unchanged; the `test` job grows by the two new files (both tiny — `manifest.test.ts` is one factory invocation plus shape assertions; `ServiceWorkerRegistration.test.tsx` is a handful of renders with a stubbed `navigator.serviceWorker`).
- **Dependencies:** none added.
- **Downstream unblock:** the parent `test-coverage` change's checkbox for sub-proposal 4.12 flips on archive of this sub-proposal. The manifest-shape and SW-registration contracts being known-good supports 6.2 `test-e2e-pwa-offline`, which exercises the runtime side (`app/sw.ts` precache, HTML-bypass, kill-switch) and assumes the registration component and manifest are correct.
- **Risk:** low. Two small files, both testable as-shipped after the two trivial in-place refactors. Notable points: (a) `navigator.serviceWorker` is not implemented by jsdom, so the test must define it via `Object.defineProperty` with `configurable: true` and reset in `afterEach` — the same stubbing pattern `test-app-frame` used for `window.visualViewport`; (b) the manifest brand-casing refactor changes a user-visible string (the install-prompt / home-screen label) — but only to bring it into line with the brand used everywhere else, so it is a correction, not a behavior change a user relied on.
- **No runtime behavior change** other than the manifest brand-casing correction (`'ctrl+List'` → `'Ctrl+List'`) and the removal of dead SSR-guard code in the registration component (no observable effect — the removed branch was never reached).
