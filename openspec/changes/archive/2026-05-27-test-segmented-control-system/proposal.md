## Why

Sub-proposal 3.6 of the `test-coverage` initiative. With `test-button-system` (3.1), `test-chip-system` (3.2), `test-form-field-system` (3.3), `test-menu-system` (3.4), and `test-popover-trigger-system` (3.5) archived, the segmented-control family is the next primitive carve-out. The carve-out is `app/ui/components/segmented-control/` — two executable JSX components (`SegmentedControl.tsx`, `SegmentedOption.tsx`), one pure class composer module (`segmentedClasses.ts` exporting two functions), and two non-executable supporting files (`types.ts` — type-only; `index.ts` — re-exports only). The parent `test-coverage` change's tasks.md §3.6 line is explicit: "Elevate to `segmented-control-system` spec." That spec already exists at `openspec/specs/segmented-control-system/spec.md` (created by archiving `standardize-menus-and-controls`); this sub-proposal therefore does NOT create a new capability spec — it locks the existing SHALLs against colocated test files, elevates the small set of non-trivial *call-time* invariants the new tests assert that are not yet in the spec (notably the two class-composer fixed-token orders, the `useSegmentedContext` outside-provider throw, and the keydown listener's container-scoped attachment), and adds the carve-out bookkeeping under `testing-foundation` per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, `test-button-system`, `test-housekeeping`, `test-chip-system`, `test-form-field-system`, `test-menu-system`, `test-popover-trigger-system`) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Every requirement applies verbatim.
- `segmented-control-system` (active spec) — 6 active requirements covering: controlled `<SegmentedControl>` radiogroup primitive with `value` / `onChange` / `tone: 'light' | 'on-dark'` props; `<SegmentedOption>` rendering `<button role="radio">` with `aria-checked` reflecting active state and firing `onChange` on activation; roving-tabindex per the WAI-ARIA radiogroup pattern (active option `tabIndex=0`, inactive `-1`); arrow keys both move focus AND fire `onChange` to the new value with wrap-around at the ends; consumption of the `button-system` focus/hover contract; migration of `VisibilityPicker` and `ItemsToolbar` view-toggle to the primitive. The tests assert against these SHALLs as the authoritative contract. **No requirements are removed.** Four NEW invariants surfaced by reading source at HEAD are elevated alongside the existing requirements — see "Capabilities" below.
- `button-system` — `segmented-control-system`'s focus/hover requirement explicitly references the `--btn-focus-ring-color` token and `:focus-visible` model from `button-system` plus the `@media (hover: hover)` guard. Token-surface link only, not a transitive SHALL — `<SegmentedOption>` does not render a `<Button>`. No `button-system` SHALL is asserted by this carve-out's tests directly; the link is honored via `segmented-control.css` referencing the right tokens. No new test coverage of `button-system` invariants from here.
- `form-field-system`, `popover-trigger-system`, `menu-system`, `chip-system`, `loading-indicator-system` — unrelated; segmented-control primitives do not render through them. Note: `<PopoverTrigger>` is the closest visual analog (the same `tone: 'light' | 'on-dark'` surface-adaptation dimension), but no shared code path.
- `list-visibility`, `items-browser-chrome` — these capabilities consume `<SegmentedControl>` at call sites (`VisibilityPicker` private/shared toggle on the list hero; `ItemsToolbar` grid/list view-toggle) but their behavior contracts are not part of this carve-out. Capability-flow sub-proposals (4.5, 4.11) own those.
- Cross-cutting design-system rule applies: this proposal is a spec-modification, NOT a page-scoped one-off. Segmented controls are an interactive surface governed by a primitive-family spec; the spec edits here EXTEND the existing `segmented-control-system` spec with elevated call-time SHALLs surfaced during testing. No new tone, no new sibling primitive. Cache-tag rule does NOT apply (no server reads).

## What Changes

- **NEW** colocated test files (one `.test.tsx` per executable JSX component under jsdom; one `.test.ts` for the pure helper module under node):

  - `app/ui/components/segmented-control/__tests__/SegmentedControl.test.tsx` — renders a `<div role="radiogroup">` with class composed via `segmentedGroupClasses({ tone, extra: className })`; passes `aria-label` / `aria-labelledby` through to the container; provides the `SegmentedContext` to children so `<SegmentedOption>` instances reflect the active value and call `onChange` on activation; forwards a `ref` (via `useImperativeHandle` over an internal `useRef`) to the container `<div>` element; attaches one `keydown` listener to the container element (NOT to `document`) when mounted, and removes that listener on unmount; the keydown handler short-circuits when the pressed key is not one of `ArrowLeft`, `ArrowRight`, `ArrowUp`, `ArrowDown`; when no `[role="radio"]` descendants exist, the keydown handler short-circuits without calling `onChange` or `preventDefault`; when ArrowRight or ArrowDown is pressed, focus moves to the next option (wrap to first after the last) AND `onChange` is fired with the next option's `data-value`; when ArrowLeft or ArrowUp is pressed, focus moves to the previous option (wrap to last from the first) AND `onChange` is fired; when no option is currently `aria-checked="true"` (`currentIndex === -1`), ArrowRight selects the first option (`(-1 + 1) % len = 0`) and ArrowLeft selects the second-to-last (`(-1 - 1 + len) % len = len - 2`) — locks the no-current-selection recovery branch; when the resolved `next.dataset.value` is `undefined`, the handler short-circuits without firing `onChange` or `preventDefault`; the handler re-attaches when the `onChange` identity changes (verified by switching `onChange` while mounted, then asserting the new callback fires on a subsequent ArrowRight); the deps array is `[onChange]` only (no `value` dep) so a value-only change does NOT re-attach the listener — verified by spy-counting `addEventListener` calls; the `tone` prop flows into `segmentedGroupClasses` (asserted via exact-string class match for `tone="light"` and `tone="on-dark"`); the `className` prop is forwarded as `extra`; throws no error when rendered with zero `<SegmentedOption>` children (the `options.length === 0` short-circuit handles it).

  - `app/ui/components/segmented-control/__tests__/SegmentedOption.test.tsx` — renders a `<button type="button" role="radio">` with `data-value={value}` (the coupling attribute the parent's keydown handler reads via `next.dataset.value`); `aria-checked="true"` IFF the context's `value === this.value`, else `aria-checked="false"` (React serializes the boolean to the exact string); `tabIndex={0}` IFF active, else `tabIndex={-1}` (roving tabindex per the spec); class composed via `segmentedOptionClasses({ active, extra: className })` — asserted via exact-string class match including the `active` class when active; clicking the button invokes the context's `onChange` with this option's `value` exactly once — including when the option is ALREADY active (idempotent re-selection — the source has no `if (isActive) return` guard); pass-through `<button>` props (`disabled`, `aria-label`, `data-testid`, custom `id`) reach the underlying element; the `value`, `onChange`, and `aria-checked` HTML attributes from the parent `Omit<>` are NOT forwarded as raw DOM attributes (the prop type omits them; React does not pass them through); `ref` forwarding lands on the underlying `<button>`; rendering `<SegmentedOption>` outside a `<SegmentedControl>` throws the descriptive Error from `useSegmentedContext` ("`<SegmentedOption> must be rendered inside a <SegmentedControl>`") — locked via `expect(() => render(...)).toThrow(...)` with the exact-string message match.

  - `app/ui/components/segmented-control/__tests__/segmentedClasses.test.ts` — pure node test (`.test.ts` runs under the node project per `vitest.config.ts`): `segmentedGroupClasses({ tone: 'light' })` returns `'segmented tone-light'` (exact string, in order) — locks Decision 3a's "tone is ALWAYS emitted" contract (contrast with `popover-trigger`, which only emits a tone class for the non-default `on-dark`); `segmentedGroupClasses({ tone: 'on-dark' })` returns `'segmented tone-on-dark'`; `segmentedGroupClasses({ tone: 'light', extra: 'foo' })` returns `'segmented tone-light foo'`; `segmentedGroupClasses({ tone: 'on-dark', extra: '' })` returns `'segmented tone-on-dark'` (empty-string `extra` filtered); `segmentedGroupClasses({ tone: 'light', extra: undefined })` returns `'segmented tone-light'`; `segmentedOptionClasses({ active: true })` returns `'segmented-option active'`; `segmentedOptionClasses({ active: false })` returns `'segmented-option'` (false filtered by `.filter(Boolean)`); `segmentedOptionClasses({ active: true, extra: 'foo' })` returns `'segmented-option active foo'` (order: base, active, extra) — locks Decision 3b; `segmentedOptionClasses({ active: false, extra: 'foo' })` returns `'segmented-option foo'`; `segmentedOptionClasses({ active: false, extra: '' })` returns `'segmented-option'`; `segmentedOptionClasses({ active: false, extra: undefined })` returns `'segmented-option'`.

- **NO** test for `index.ts` — re-export only, zero executable statements after TS erasure. Excluded under the existing `app/ui/components/*/index.ts` glob in `vitest.config.ts`'s `coverage.exclude` (no change needed).
- **NO** test for `types.ts` — type-only (`SegmentedTone = 'light' | 'on-dark'`), zero executable statements after TS erasure. Already implicitly excluded (no runtime entry); confirm absence from the report.
- **NO** test for `segmented-control.css` — CSS files are not part of the JS coverage report.
- **NEW** ESLint override in `eslint.config.mjs`: extend the existing per-file `sonarjs/cognitive-complexity = error` override array to include `SegmentedControl.tsx`, `SegmentedOption.tsx`, and `segmentedClasses.ts`. Measured complexity at HEAD: `SegmentedControl.tsx` is the only non-trivial file (keydown handler with key-filter, options-empty short-circuit, currentIndex resolution, forward/backward modulo, undefined-value short-circuit) — expected in the 8–12 range, under the ceiling of 15. `SegmentedOption.tsx` is 1–2 (single-return JSX with one inline conditional for the active state). `segmentedClasses.ts` is 1–2 (two `.filter(Boolean).join(' ')` expressions). The override locks the ceiling at 15 for future edits.
- **NEW** per-file thresholds in `vitest.config.ts` for the three executable files, each referencing the existing `COVERAGE_FLOOR` constant (one entry each — NO per-file numeric variation, per `test-housekeeping`'s single-constant rule).
- **MODIFIED** `segmented-control-system` spec — ADDS four new requirements that elevate call-time invariants the new tests lock against and that are currently implicit in the source:
  - `segmentedGroupClasses({ tone, extra })` SHALL compose the wrapper class string as `['segmented', `tone-${tone}`, extra].filter(Boolean).join(' ')` with `tone-${tone}` ALWAYS emitted (so `tone="light"` produces `'segmented tone-light'`, NOT `'segmented'`) — locks the always-emit-tone contract that distinguishes this composer from `triggerClasses` (which only emits a tone class for the non-default `on-dark`), and the fixed token order that `segmented-control.css` selectors like `.segmented.tone-on-dark .segmented-option.active` depend on.
  - `segmentedOptionClasses({ active, extra })` SHALL compose the option class string as `['segmented-option', active && 'active', extra].filter(Boolean).join(' ')` in fixed token order: `'segmented-option'`, then `'active'` (when truthy), then `extra` (when truthy) — locks the composer's output shape so CSS selectors and `extra`-via-`className` passthrough remain stable.
  - `useSegmentedContext()` SHALL throw a descriptive Error (message exactly `'<SegmentedOption> must be rendered inside a <SegmentedControl>'`) when invoked outside a `<SegmentedControl>` ancestor — locks the developer-error safety net that protects against silently-broken consumers (e.g. an option mistakenly rendered in a `<Menu>` or directly on a page would otherwise read `value: undefined` from a null context and short-circuit unpredictably).
  - `<SegmentedControl>`'s `keydown` listener SHALL be attached to the container `radiogroup` element, NOT to `document` — locks the scoped-listener contract that lets multiple segmented controls coexist on the same page without arrow-key interference, and that prevents the segmented control from intercepting arrow keys when focus is elsewhere on the page (e.g. text input, scrollable region, other widget). The listener SHALL be removed on unmount; the deps array SHALL be `[onChange]` so the listener re-attaches on `onChange` identity change but NOT on every `value` change.
  These elevations are documented in `design.md` Decision 3.
- **NEW** `testing-foundation` carve-out bookkeeping record (sub-proposal-archive ONLY, per `test-coverage` design D13 two-tier rollup): the segmented-control-system primitive carve-out has landed at the universal `COVERAGE_FLOOR`, the `sonarjs/cognitive-complexity` promotion to `error` is in effect for the three files, and subsequent sub-proposals consuming `<SegmentedControl>` / `<SegmentedOption>` / `segmentedClasses` inherit the assumption that those modules are tested and complexity-locked. Recorded in `openspec/changes/test-segmented-control-system/specs/testing-foundation/spec.md` ONLY (and at archive in `openspec/changes/archive/<date>-test-segmented-control-system/specs/testing-foundation/spec.md`) — does NOT roll into the parent's `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md` (which is created at test-coverage's archive time from the parent accumulator).
- **NEW** four-audit findings recorded in `tasks.md`. Carve-out is small (three files; the largest is `SegmentedControl.tsx` at 118 LOC with the keydown effect handler as the only non-trivial path). Anticipated dispositions: assertion-substance pass on the new tests; duplication may surface a shared `Harness` helper for rendering `<SegmentedControl>` + N `<SegmentedOption>`s with various tones / values — kept inline if used only within one file (precedent: `test-menu-system`, `test-popover-trigger-system`); complexity audit expected to pass on all three (≤ ~12 each, below ceiling 15); testability audit expected to be straightforward (the only document-level concern is the keydown listener, which is container-scoped — fired via `fireEvent.keyDown(container, …)` or `userEvent.keyboard(...)` after focusing the active option).
- **NO source refactors expected** beyond a potential parameter-default precedent like `test-menu-system` §1.4 / §7.4 and `test-popover-trigger-system` §1.4 / §5.4 (each found that a destructured-with-no-default function threw on no-args calls per their spec scenarios). `segmentedGroupClasses` and `segmentedOptionClasses` already have destructured parameters with no defaults and no `= {}` — but no existing or new spec scenario asserts the no-args call (the spec only describes the `tone` and `active` arguments, which are always present in real consumers since they are required props). The need only surfaces if a new spec scenario is added that calls the function with no args; none is planned here. Any audit finding requiring source change is recorded with disposition in `tasks.md` (refactor-in-place commits become part of this change).

- **Risk:** low. Three small files, all testable as-shipped. Realistic miss: failing to author the `NoCheckedOption_ArrowRight_SelectsFirst` and `NoCheckedOption_ArrowLeft_SelectsSecondToLast` tests, since v8 reports the line as "covered" once `currentIndex === -1` is reached at least once — the explicit assertions that the wrap-around math produces the right index in the no-checked case are the only way to lock the contract. Secondary risk: the keydown-listener-deps-array assertion (`[onChange]` only, not `[value]`) is easy to skip because it has no user-visible failure mode until someone changes the deps to `[value, onChange]` and the listener re-attaches on every selection, doubling event handlers if cleanup is also miswritten — the test spy-counts `addEventListener` to lock it. Tertiary risk: the `useSegmentedContext` throw is easy to write as a positive-path-only test ("works inside a provider"); the negative test (`expect(() => render(<SegmentedOption value="x"/>)).toThrow(...)`) with exact-string message match is the lock. Quaternary risk: the always-emit-tone difference between `segmentedGroupClasses` and `triggerClasses` is easy to forget when authoring the test — a contributor steeped in the `popover-trigger` precedent could write `expect(segmentedGroupClasses({ tone: 'light' })).toBe('segmented')` (false expectation); the test names and the new spec SHALL surface the difference explicitly.
- **No runtime behavior change.**
