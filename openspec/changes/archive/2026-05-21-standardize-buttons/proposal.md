## Why

Button styling is fragmented across three parallel systems (`.btn` variants, `.form-shell-btn-*`, and one-off page-scoped classes), with diverging radii, padding, font sizes, and mobile-scaling rules. The fragmentation produces visible inconsistency between pages and — more critically — at least one WCAG 2.5.8 touch-target failure on mobile (`form-shell-btn-*` shrinks to ~30px high under 1000px) plus several a11y gaps (no enforced `:focus-visible`, unguarded `:hover` that sticks on iOS, icon-only buttons without confirmed `aria-label`, the bookmark button missing `aria-pressed`). Beyond CSS, the existing `<Button>` component is a className passthrough that enforces nothing, so every call site hand-writes `className="btn primary"` and nothing prevents future drift. A single token-backed system with two real components (`<Button>` and `<LinkButton>`) sharing a canonical class builder, plus migrating every call site through it, closes the visual, a11y, and structural gaps in one change rather than leaving an open seam.

## What Changes

- Introduce shared button design tokens in `global.css` (`--btn-radius`, `--btn-padding-y`, `--btn-padding-x`, `--btn-font-size`, `--btn-font-weight`, `--btn-min-height: 44px`, `--btn-min-width: 44px`, `--btn-focus-ring-color`, plus `-sm` siblings for legitimately-small contexts like pagination/chips).
- Reconcile `.btn` and `.form-shell-btn-*` to consume the same tokens so the two systems become visually identical, then **remove** `.form-shell-btn-*` entirely once nothing references it.
- Enforce a 44×44 CSS-pixel minimum touch target on every standard button via `min-height` / `min-width` on the shared base.
- Fix the mobile-shrink regression in `form-shell.css:632-634` (currently reduces padding/font under 1000px).
- Add a canonical `:focus-visible` ring style on the shared button base.
- Guard all button `:hover` rules with `@media (hover: hover)` to prevent sticky-hover on iOS touch.
- **Merge `nav` and any list-hero button treatment into a single `on-dark` variant.** The current `.nav` class becomes `on-dark`; bookmark/follow buttons (currently using `.secondary`, which is visually wrong on a purple surface) migrate to `on-dark`. This is an intentional visual change for bookmark/follow.
- Build `<Button>` ([app/ui/components/Button.tsx](app/ui/components/Button.tsx)) as a real component with `variant` (`primary | secondary | ghost | danger | on-dark`), `size` (`sm | md`), `isLoading` (sets `disabled` + `aria-busy` + spinner, does not swap label), and `pressed` (when defined, sets `aria-pressed` and triggers pressed-state styling — orthogonal to variant).
- Add `<LinkButton>` as a sibling component that renders Next `<Link>` with the same visual treatment, accepts `variant` and `size` but not `isLoading` (links don't have native disabled semantics; if a real caller needs it later we add it).
- Extract a canonical class builder `buttonClasses({ variant, size, pressed, extra })` consumed by both `<Button>` and `<LinkButton>` so there is one source of truth for class composition.
- **Migrate every call site** from `className="btn primary"` (and `.form-shell-btn-*`, and the page-scoped one-offs that are absorbed) to `<Button variant=…>` or `<LinkButton variant=…>`. Page-scoped classnames that are legitimately small (pagination, chips) migrate to `size="sm"` with the WCAG 2.5.8 spacing exemption documented inline.
- **Bring the store-links row into the system.** `StoreLinks.tsx` currently renders bare `<a>` / `<button>` elements styled by `.buy-link` / `.buy-link--extra` / `.buy-link-more` — a fourth parallel button system. Migrate the row's internals to `<LinkButton size="sm">` (primary + extras) and `<Button variant="ghost" size="sm">` (the `+N` expand toggle), keeping `StoreLinks.tsx` as the orchestrator that owns expand/collapse choreography. *(Update: the expand/collapse choreography piece is superseded by `replace-storelinks-expand-with-popover` — the `+N` chip is now a `<Menu>` trigger, no inline animation. The Button/LinkButton migration described here remains valid; only the choreography is replaced.)* No new `<StoreLink>` component is introduced — the row's wrapper *is* `StoreLinks`, since these primitives are never reused outside it. Remove `.buy-link*` rules from `app/(main)/items/ui/styles/store-links.css` once nothing references them. **Open question:** `size="sm"` must serve both pagination (~32px feels right) and store-link chips (currently ~24px) — if visual review of item rows shows `sm` is too tall and densifies lists unacceptably, the fallback is either adding a `size="xs"` tier or shrinking `sm` itself. The decision is deferred to design.md and resolved during migration, not before.
- Audit and fix icon-only buttons (`DeleteListButton`, `DeleteItemButton`, `BookmarkButton`, `ListActionsMenu` trigger, `Pagination`) for missing `aria-label`.
- Remove `.bookmark-button` and `.follow-button` color/layout classes from `following-and-history.css` — absorbed into `variant="on-dark"` + `pressed`.
- **Add `variant="link"`** for text-button affordances (no border, no background, primary-color text, underline on hover) — covers `ImageUrlInput`'s "Can't find a URL? Search for an image" disclosure and `CollapsibleRail`'s "See all" link, both currently page-scoped one-offs.
- **Extend pressed-state CSS to every variant**, not just `on-dark`. `SpoilerToggle` today switches between `.btn primary` (on) and `.btn secondary` (off) — which violates Decision 3 (variant ≠ toggle state). Migrating it to `<Button variant="secondary" pressed={…}>` requires `.btn.secondary[aria-pressed="true"]` styling to exist; add the equivalent for `primary`, `ghost`, `danger` too so future toggles can pick any variant.
- **Add a `<Chip>` primitive** at `app/ui/components/chip/`. Chips are label + remove-× affordances — two elements wrapped together, not a single button. Two existing call sites: `.items-toolbar-chip` active-filter chips (`ItemsToolbar.tsx`) and `.if-lp-chip` selected-list chips (`itemform/ListSelection.tsx`). `<Chip>` consumes `buttonClasses()` internally for focus-ring / hover-guard / min-touch consistency but has its own API (`onRemove`, label children). Remove `.items-toolbar-chip` and `.if-lp-chip` styling from page-scoped CSS after migration.
- **Delete `app/ui/components/Form/FormButton.tsx`** — a 54-line `<button>` wrapper with its own SVG spinner and "Loading..." label that duplicates the new `<Button isLoading>` semantics. Migrate any callers to `<Button variant="primary" isLoading={…}>`.
- **Delete `app/(main)/lists/ui/components/EditListButton.tsx`** — a className-passthrough `<button>` that two call sites (`ListDetails` hero and `ListActionsMenu` dropdown) feed different classes into. The two call sites want different primitives (`<Button variant="on-dark">` in hero; a menu-item row in dropdown); inline its `useState` at the call sites and let each pick the right primitive. The dropdown call site retains its `.menu-item` styling temporarily (see "Out of scope" below).
- **Migrate `SpoilerToggle`** from variant-switching (`.btn primary` ↔ `.btn secondary`) to a single variant with `pressed`. The ON/OFF state-badge stays as page-scoped children inside the button.
- Out of scope — **deferred to follow-up change `standardize-menus-and-controls`**: the `<Menu>` + `<MenuItem>` family (replacing `.menu-item` rows in `ListActionsMenu` and `UserAvatarPopover`), the `<SegmentedControl>` primitive (replacing `VisibilityPicker`'s Private/Shared toggle and `ItemsToolbar`'s grid/list view toggle), and the `<PopoverTrigger>` primitive (replacing `.store-filter-trigger` / `.items-toolbar-filters-trigger` / `.if-lp-trigger`). These are popover-or-radiogroup patterns with shared keyboard/focus concerns that warrant their own focused design exercise after this change lands. Their wrapper components (`ListActionsMenu`, `VisibilityPicker`, `StoreFilterPopover`, `PriceFilterPopover`, `UserAvatarPopover`, `ListSelection`) stay as-is in this change.
- Out of scope — **documented exemptions, not deferred**: `.gsi-material-button` (Google sign-in vendor styling). `AppNav` nav items (`.app-nav-item`, `.app-nav-toggle`) — these are navigation items, a separate DS primitive, not buttons; flagged in design.md. `CollapsibleRail`'s `.rail-toggle` (whole-heading expand affordance) — a heading-button, not a pill button. `.if-lp-opt` listbox options inside `ListSelection`'s dropdown — `role="option"` listbox items, not buttons. The list-hero's `--hero-gradient` surface (separate change). Building new variants beyond `link`.

## Capabilities

### New Capabilities
- `button-system`: Defines the visual and accessibility contract for all interactive button elements in the app — variants (purely visual, including `link` for text-button affordances), sizes, the orthogonal `pressed` toggle prop (with pressed-state CSS for every variant), minimum touch targets, focus indication, hover-on-touch behavior, loading state semantics, the two-component split (`<Button>` for `<button>`, `<LinkButton>` for `<Link>`), the shared `buttonClasses()` class builder, the token surface that styles consume, and the sibling `<Chip>` primitive for label + remove-× affordances that consume the same focus/hover/min-touch contract.

## Impact

- **CSS**: `app/ui/styles/button.css` (rewritten to consume tokens, adds spinner + pressed-state CSS for **every variant** not just `on-dark`, adds `.btn.link` variant), `app/ui/styles/form-shell.css` (`.form-shell-btn-*` removed after call-site migration), `app/ui/styles/global.css` (tokens added). Page-scoped CSS files: `.bookmark-button` / `.follow-button` color/layout rules removed from `app/(main)/lists/ui/styles/following-and-history.css`; `.buy-link` / `.buy-link--extra` / `.buy-link-more` color/layout/sizing rules removed from `app/(main)/items/ui/styles/store-links.css` (the row's flex/gap/expand-animation rules stay — only the button-shaped chip styling is absorbed); `.items-toolbar-chip` removed from items-toolbar CSS once chips migrate to `<Chip>`; `.if-lp-chip` (the chip body, not the dropdown trigger) removed from item-form CSS once chips migrate. The page-scoped small-button classes (`.items-page-btn`, `.choose-items-new-btn`, `.list-hero-btn`, `.menu-trigger`, `.if-search-link`, `.rail-see-all`) are removed once their call sites migrate (or stay if a documented exemption is required).
- **Components**: New directory `app/ui/components/button/` containing `Button.tsx`, `LinkButton.tsx`, `buttonClasses.ts`, `types.ts`. New directory `app/ui/components/chip/` containing `Chip.tsx`, `chipClasses.ts` (consumes `buttonClasses()` internals). Deleted: `app/ui/components/Form/FormButton.tsx` (callers migrate to `<Button isLoading>`); `app/(main)/lists/ui/components/EditListButton.tsx` (callers inline its `useState`). Icon-only button components get `aria-label` audits: `DeleteListButton`, `DeleteItemButton`, `BookmarkButton`, `ListActionsMenu`, `Pagination`. `BookmarkButton` and the follow button additionally gain `pressed={…}` for `aria-pressed` semantics. `SpoilerToggle` migrates from variant-switching to `<Button pressed={…}>` with one variant.
- **Call sites**: All ~47 `className=".*btn.*"` usages across ~22 files migrate to `<Button>` or `<LinkButton>`, plus the store-link row in `app/(main)/items/ui/components/StoreLinks.tsx` (3 element types: primary buy-link, extra buy-links, +N expand toggle), plus the two `<Chip>` call sites (active-filter chips in `ItemsToolbar`, selected-list chips in `ListSelection`), plus the two inlining sites for `EditListButton` (`ListDetails` and `ListActionsMenu`). This is in scope — the change is not done until they are migrated.
- **Adjacent inconsistency, deliberately not fixed here**: The list hero uses `--hero-gradient` (`linear-gradient(135deg, #4a35c5 0%, #7855f0 100%)`) while the nav uses solid `--primary-color` (`#7324ce`). This is itself a drift inconsistency of the same shape as the button drift this change addresses, but the surface decision belongs to `redesign-home-and-tokens` or a follow-up change. The `on-dark` button variant is designed against the solid nav purple as the canonical surface; it works on the gradient endpoints (close enough in hue/value) but if the gradient stays *and* contrast verification flags an issue at `#7855f0`, the fix is at the surface, not the button.
- **Out of scope** (recap): `.gsi-material-button` (Google brand requirement). The list-hero gradient surface (separate change). Building new variants beyond the reconciled set + `link`. Menus / segmented controls / popover triggers (follow-up change `standardize-menus-and-controls`). Nav items and rail-toggle headings (documented exemptions per the "What Changes" list).
- **Risk**: The 44px floor will make some currently-compact buttons taller. Visual review needed on dense layouts (item rows, toolbar, action menus, pagination, store-link chips) — these are the candidates for `size="sm"` with documented exemption. The store-link row is the densest case: if `size="sm"` lands at ~32px to serve pagination well, item rows will grow noticeably; the resolution path (drop `sm` smaller, or add `xs`) is decided during migration based on actual visual review, not pre-committed. The bookmark/follow visual change (from `.secondary` light fill to `on-dark` transparent treatment) needs design-intent confirmation before flipping — cross-check `redesign-home-and-tokens/design.md` to see if it already planned this.
