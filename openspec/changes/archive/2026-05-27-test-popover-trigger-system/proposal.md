## Why

Sub-proposal 3.5 of the `test-coverage` initiative. With `test-button-system` (3.1), `test-chip-system` (3.2), `test-form-field-system` (3.3), and `test-menu-system` (3.4) archived, the popover-trigger family is the next primitive carve-out. The carve-out is `app/ui/components/popover-trigger/` plus the component-scoped hook at `app/ui/hooks/usePopoverDismiss.ts` — one executable JSX component (`PopoverTrigger.tsx`), one pure class composer (`triggerClasses.ts`), one React hook (`usePopoverDismiss.ts`), and two non-executable supporting files (`types.ts` — type-only; `index.ts` — re-exports only). The parent `test-coverage` change's tasks.md §3.5 line is explicit: "Elevate to `popover-trigger-system` spec." That spec already exists at `openspec/specs/popover-trigger-system/spec.md` (created by archiving `standardize-menus-and-controls`); this sub-proposal therefore does NOT create a new capability spec — it locks the existing SHALLs against colocated test files, elevates the small set of non-trivial *call-time* invariants the new tests assert that are not yet in the spec (notably `PopoverTrigger`'s count-badge zero-suppression gate, the inline chevron's `aria-hidden="true"` accessibility contract, the `triggerClasses` composer's fixed token order, and `usePopoverDismiss`'s null-ref short-circuit), and adds the carve-out bookkeeping under `testing-foundation` per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, `test-button-system`, `test-housekeeping`, `test-chip-system`, `test-form-field-system`, `test-menu-system`) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Every requirement applies verbatim.
- `popover-trigger-system` (active spec) — 7 active requirements covering: form-input-styled `<button>` primitive (label + optional icon + optional count badge + chevron); no-variant-prop / `tone: 'light' | 'on-dark'` surface adaptation; focus + 44px touch contract via `button-system` tokens; migration of filter popovers (`StoreFilterPopover`, `PriceFilterPopover`, `ItemsToolbar` mobile-filters trigger) and `ListSelection` list-picker trigger to `<PopoverTrigger>`; popover-body content stays page-scoped; shared dismiss behavior via `usePopoverDismiss({ open, onClose, ref })` hook at `app/ui/hooks/usePopoverDismiss.ts`; list-detail hero status pill consumes `<PopoverTrigger tone="on-dark">` with state-encoding icon. The tests assert against these SHALLs as the authoritative contract. **No requirements are removed.** Four NEW invariants surfaced by reading source at HEAD are elevated alongside the existing requirements — see "Capabilities" below.
- `button-system` — `popover-trigger-system`'s focus/touch requirement explicitly references "reuses the `--btn-focus-ring-color` token and `:focus-visible` model from `button-system`, guards `:hover` with `@media (hover: hover)`, and meets the 44×44 minimum touch target." This is a token-surface link, not a transitive SHALL — `<PopoverTrigger>` does not render a `<Button>`. No `button-system` SHALL is asserted by this carve-out's tests directly; the link is honored via `popover-trigger.css` referencing the right tokens (`--field-focus-ring-color`, `--field-focus-ring-width`, the `@media (hover: hover)` guard, the `--field-min-height` floor which is ≥ 44px in `global.css`).
- `form-field-system` — `<PopoverTrigger>` reuses field tokens (`--field-padding-y`, `--field-padding-x`, `--field-min-height`, `--field-radius`, `--field-border-color`, `--field-font-size`) to stay visually consistent with text inputs and native selects. Same token-surface link; no transitive SHALL asserted. The carve-out tests do not re-verify field-token visual outcomes — those are covered by `form-field-system` itself.
- `chip-system`, `menu-system`, `segmented-control-system`, `loading-indicator-system` — unrelated; popover-trigger primitives do not render through them. Note: `<SegmentedControl>` is the closest visual analog (`tone: 'light' | 'on-dark'` is the same surface-adaptation dimension), but no shared code path.
- `items-browser-chrome`, `list-item-management`, `list-hero-header`, `list-visibility` — these capabilities consume `<PopoverTrigger>` at call sites (filter popovers in items toolbar; list-picker in item-form; list-hero status pill) but their behavior contracts are not part of this carve-out. Capability-flow sub-proposals (4.5, 4.7, 4.9, 4.11) own those.
- Cross-cutting design-system rule applies: this proposal is a spec-modification, NOT a page-scoped one-off. Popover triggers are an interactive surface governed by a primitive-family spec; the spec edits here EXTEND the existing `popover-trigger-system` spec with elevated call-time SHALLs surfaced during testing. No new tone, no new sibling primitive. Cache-tag rule does NOT apply (no server reads).

## What Changes

- **NEW** colocated test files (one `.test.tsx` per executable JSX component under jsdom; one `.test.ts` for the pure helper under node; one `.test.tsx` for the hook under jsdom since `usePopoverDismiss` uses `document.addEventListener`):

  - `app/ui/components/popover-trigger/__tests__/PopoverTrigger.test.tsx` — renders a `<button type="button" class="popover-trigger">` containing the children in DOM order: optional `icon` slot (any ReactNode; renders before label), `<span class="popover-trigger-label">{label}</span>`, optional `<span class="popover-trigger-count">{count}</span>` (rendered ONLY when `count !== undefined && count > 0`), inline `<svg class="popover-trigger-chevron" aria-hidden="true" …>` chevron; `count === 0` renders NO badge span (the `count > 0` gate suppresses the zero case — distinct from `count={undefined}` which also renders no badge); `count={undefined}` renders NO badge span; `count={3}` renders the badge with text content `'3'`; `tone="on-dark"` adds `tone-on-dark` to the class string via `triggerClasses`; `tone="light"` (and tone omitted, default `'light'`) does NOT add `tone-on-dark`; `active={true}` adds `active` to the class string; `active={false}` (and omitted) does NOT; the `className` prop is forwarded as `extra` to `triggerClasses` and appended after `active`; `ref` forwarding lands on the underlying `<button>`; explicit `type` prop (e.g. `type="submit"`) overrides the `'button'` default; pass-through `<button>` props (`onClick`, `disabled`, `aria-label`, `aria-haspopup`, `aria-expanded`, `data-testid`) reach the underlying button; clicking it invokes `onClick` exactly once. The chevron is rendered as a strict `<svg aria-hidden="true" width="10" height="6" viewBox="0 0 10 6">` with one `<path d="M1 1l4 4 4-4" stroke="currentColor" …>` — the `aria-hidden` attribute is asserted explicitly so a regression that drops it (announcing the path data to screen readers) fails by name.

  - `app/ui/components/popover-trigger/__tests__/triggerClasses.test.ts` — pure node test (`.test.ts` runs under the node project per `vitest.config.ts`): `triggerClasses()` (no args, relies on `tone` parameter default `'light'`) returns `'popover-trigger'`; `triggerClasses({})` returns `'popover-trigger'`; `triggerClasses({ tone: 'light' })` returns `'popover-trigger'` (default tone does not add a tone class); `triggerClasses({ tone: 'on-dark' })` returns `'popover-trigger tone-on-dark'` (exact string, in order); `triggerClasses({ active: true })` returns `'popover-trigger active'`; `triggerClasses({ active: false })` returns `'popover-trigger'` (false filtered by `.filter(Boolean)`); `triggerClasses({ tone: 'on-dark', active: true })` returns `'popover-trigger tone-on-dark active'` (order: base, tone, active); `triggerClasses({ tone: 'on-dark', active: true, extra: 'foo' })` returns `'popover-trigger tone-on-dark active foo'` (order: base, tone, active, extra); `triggerClasses({ extra: '' })` returns `'popover-trigger'` (empty-string `extra` filtered); `triggerClasses({ extra: undefined })` returns `'popover-trigger'` (explicit undefined filtered).

  - `app/ui/hooks/__tests__/usePopoverDismiss.test.tsx` — uses `@testing-library/react`'s `renderHook` under jsdom: `Closed_NoListenersAttached` — spy on `document.addEventListener`; mount with `open: false`; assert `addEventListener` NOT called for `mousedown` or `keydown` (the `if (!open) return` short-circuits the effect body); `Open_ListenersAttached` — spy; mount with `open: true`; assert `addEventListener` called twice (once for `mousedown`, once for `keydown`); `OutsideMousedown_OnCloseCalled` — mount with `open: true` and a real `ref` pointing at a sibling DOM element; dispatch `MouseEvent('mousedown', { bubbles: true })` on `document.body` (target outside the ref'd element); `onClose` called once; `InsideMousedown_OnCloseNotCalled` — dispatch `mousedown` whose `target` is a descendant of `ref.current`; `onClose` NOT called (the `ref.current.contains(target)` check returns true → the negation short-circuits); `NullRef_OutsideMousedown_OnCloseNotCalled` — mount with `open: true` and a `ref` whose `current` is `null` (e.g. fresh `useRef(null)` never attached); dispatch `mousedown` on `document.body`; `onClose` NOT called (the `ref.current && …` short-circuit early-exits before any `contains` call) — **Spec delta SHALL** — Decision 3d; `EscapeKey_OnCloseCalled` — dispatch `KeyboardEvent('keydown', { key: 'Escape' })` on `document`; `onClose` called once; `OtherKey_OnCloseNotCalled` — dispatch `keydown` with `key === 'a'`; `onClose` NOT called; `Unmount_ListenersCleanedUp` — spy on `document.removeEventListener`; unmount; assert `removeEventListener` called twice (once per listener); `ClosedAfterOpen_ListenersCleanedUp` — rerender with `open: false` after `open: true`; spy on `removeEventListener`; assert the cleanup ran (both listeners removed); `OnCloseIdentityChange_ListenersReattached` — rerender with a different `onClose` reference while open; cleanup runs (old listeners removed) and new listeners are attached pointing at the new `onClose` (verified by dispatching a key and asserting the new callback fires, not the old one) — exercises the `[open, onClose, ref]` deps array.

- **NO** test for `index.ts` — re-export only, zero executable statements after TS erasure. Excluded under the existing `app/ui/components/*/index.ts` glob in `vitest.config.ts`'s `coverage.exclude` (no change needed).
- **NO** test for `types.ts` — type-only (`PopoverTriggerTone = 'light' | 'on-dark'`, `PopoverTriggerOwnProps`, `PopoverTriggerProps`), zero executable statements after TS erasure. Already implicitly excluded (no runtime entry); confirm absence from the report.
- **NO** test for `popover-trigger.css` — CSS files are not part of the JS coverage report.
- **NEW** ESLint override in `eslint.config.mjs`: extend the existing per-file `sonarjs/cognitive-complexity = error` override array to include each of the three files (`PopoverTrigger.tsx`, `triggerClasses.ts`, `usePopoverDismiss.ts`). Measured complexity at HEAD: all three files are 1–2 (PopoverTrigger is a single-return JSX function with two inline conditionals; triggerClasses is a single `.filter(Boolean).join(' ')`; usePopoverDismiss is one `useEffect` with two event handlers and one `!open` short-circuit). The override locks the ceiling at 15 for future edits.
- **NEW** per-file thresholds in `vitest.config.ts` for the three executable files, each referencing the existing `COVERAGE_FLOOR` constant (one entry each — NO per-file numeric variation, per `test-housekeeping`'s single-constant rule).
- **MODIFIED** `popover-trigger-system` spec — ADDS four new requirements that elevate call-time invariants the new tests lock against and that are currently implicit in the source:
  - `<PopoverTrigger>` count badge SHALL render IFF `count !== undefined && count > 0` (both `count === 0` AND `count === undefined` SHALL produce no badge span) — locks the zero-suppression gate the source enforces but the spec does not state. The existing scenario shows `count={3}` rendering a "3" badge; this requirement makes the absence cases explicit.
  - `<PopoverTrigger>`'s inline chevron `<svg>` SHALL carry `aria-hidden="true"` so screen readers do not announce the path data — locks the AT-decoration contract the source enforces but the spec does not state.
  - `triggerClasses({ active, tone, extra })` SHALL compose the wrapper class string in fixed token order: `'popover-trigger'`, then `'tone-on-dark'` (when `tone === 'on-dark'`), then `'active'` (when truthy), then `extra` (when truthy), joined by single spaces with falsy tokens filtered — locks the composer's output shape so CSS selectors like `.popover-trigger.tone-on-dark.active` remain stable across both row primitives that share this base class.
  - `usePopoverDismiss({ open, onClose, ref })` SHALL no-op the outside-click handler when `ref.current` is `null` — the `ref.current && !ref.current.contains(target)` guard prevents an outside click from firing `onClose` before the ref has attached (e.g. the popover-open frame between `setState(true)` and ref-callback resolution). Locks the mount-race safety the source enforces but the spec does not state.
  These elevations are documented in `design.md` Decision 3.
