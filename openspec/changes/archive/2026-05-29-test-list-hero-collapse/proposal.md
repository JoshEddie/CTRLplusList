## Why

Sub-proposal 4.8 of the `test-coverage` initiative (parent `tasks.md` §4.8, GitHub issue #47) — the `list-hero-collapse` capability carve-out. The capability already has a substantive active spec at `openspec/specs/list-hero-collapse/spec.md` (six requirements covering the collapse toggle, the collapsed strip, the `ListActionsMenu` `prependedItems`/`isOwner` extension, the `?hero=closed` URL-state model, and the share-URL hero-param exclusion). This sub-proposal therefore does NOT create a new capability spec — it (a) locks the existing SHALLs against colocated test files at the universal `COVERAGE_FLOOR`, (b) MODIFIES the scenarios whose menu-item labels and visibility-row ordering describe authoring-time wording that the source never matched (the `Share` → `Share List` label and the `Just me` → `Hidden` row label + the internally-inconsistent row ordering — see Decision 3 in `design.md`), (c) ADDS one new SHALL for an invariant the source enforces today that no requirement currently locks (keyboard operability of the collapsed strip via Enter/Space), and (d) adds the carve-out bookkeeping under `testing-foundation` Tier 2 per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, `test-housekeeping`, and the primitive-family carve-outs) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule (write the test OR `/* v8 ignore */` with named rationale — never lower the floor), `<State>_<Behavior>` `it()` shape, three-role `describe()` convention (owner / authenticated-non-owner / unauthenticated where applicable), the four-audit obligation (assertion-substance, duplication, complexity, testability), the invariant-elevation audit, the assertion-substance bar, the `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy, and the NextAuth-as-network-boundary mocking allowance. Every requirement applies verbatim.
- `list-hero-collapse` (active spec) — 6 active requirements. The tests assert against these SHALLs as the authoritative contract. Two requirements' scenarios MUST be MODIFIED for label/ordering drift (R3 visibility rows; the menu-content scenarios in R2/R3 — see "What Changes"). One NEW invariant surfaced by reading source at HEAD is ADDED (collapsed-strip keyboard activation).
- `list-hero-header` (active spec) — OWNS the expanded-hero composition and the `ListDetails.tsx` composer that decides which prepended items to pass. This carve-out does NOT test `ListDetails.tsx`, `ShareButton.tsx`, `VisibilityPicker.tsx`, `BookmarkButton.tsx`/`BookmarkContainer.tsx`, or the `ListHeroSection.tsx`/`page.tsx` route shells — those belong to 4.7 `test-list-hero-header` and the route's own flow. The collapse carve-out tests only the modules whose primary purpose is the collapsed state and its kebab extension. See Decision 1.
- `list-visibility` (active spec, **HIGH stakes**, 4.11) — OWNS the three-state visibility semantics (`setListVisibility`'s Private/Shared composition + feed signaling). `VisibilityMenuItems` and the `VISIBILITY_ROWS` table render the picker UI and invoke `setListVisibility`, but the action's behavior contract is NOT part of this carve-out — the server action is mocked at the module boundary in these client-component unit tests. The optimistic-update + rollback-on-failure behavior of `VisibilityMenuItems` is tested here (it lives in this carve-out's file) but is NOT elevated to a `list-hero-collapse` SHALL; the action's normative contract stays in `list-visibility`.
- `following` (active spec, 4.2) and `list-collections` (bookmarks, 4.6) — OWN the follow/unfollow and bookmark action contracts that `FollowMenuItem` / `BookmarkMenuItem` invoke. Same treatment: the menu-item components are tested here (block-gating, disclosure gating, optimistic rollback), the underlying actions (`followUser`/`unfollowUser`/`bookmarkList`/`unbookmarkList`) and DAL reads (`isFollowing`/`isBlocked`/`getBookmarkStatus`/`viewerHasAnyFollows`) are mocked at the module boundary.
- `menu-system` (active spec, 3.4) — the `Menu`, `MenuItem`, `MenuItemRadio`, `MenuLinkItem` primitives are tested under `test-menu-system` (already carved out). This carve-out renders through the real primitives (does NOT mock them) so the integration is exercised, but asserts no `menu-system` SHALL directly.
- `button-system` (active spec, archived 3.1) — `ListActionsMenu`'s kebab trigger is a `<Button variant="on-dark">`; token-surface link only, no `button-system` SHALL asserted here.
- Cross-cutting design-system rule applies: this proposal is a spec-modification, NOT a page-scoped one-off. The collapse toggle, collapsed strip, and kebab extension are governed by the `list-hero-collapse` capability spec; the label/ordering corrections EXTEND/MODIFY that existing spec. No new sibling capability is created.
- Cache-tag rule: `HeroCollapsedItemsContainer.tsx` performs server-side DAL reads (`getBookmarkStatus`, `isFollowing`, `isBlocked`, `viewerHasAnyFollows`). These reads consume the `'use cache'` tags owned by `following` / `list-collections` / the block-relationship surface; this carve-out does NOT change any read or add any mutation, so no `revalidateTag` obligation is introduced. The DAL is mocked at the module boundary in the container's unit test.
