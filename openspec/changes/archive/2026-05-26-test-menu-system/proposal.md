## Why

Sub-proposal 3.4 of the `test-coverage` initiative. With `test-button-system` (3.1), `test-chip-system` (3.2), and `test-form-field-system` (3.3) archived, the menu family is the next primitive carve-out. The carve-out is `app/ui/components/menu/` — four executable JSX components (`Menu.tsx`, `MenuItem.tsx`, `MenuItemRadio.tsx`, `MenuLinkItem.tsx`) plus one pure class composer (`menuClasses.ts`) and two non-executable supporting files (`types.ts` — type-only union; `index.ts` — re-exports only). The parent `test-coverage` change's tasks.md §3.4 line is explicit: "Elevate to `menu-system` spec." That spec already exists at `openspec/specs/menu-system/spec.md` (created by archiving `standardize-menus-and-controls`); this sub-proposal therefore does NOT create a new capability spec — it locks the existing SHALLs against colocated test files, elevates the small set of non-trivial *call-time* invariants the new tests assert that are not yet in the spec (notably `Menu`'s outside-click anchor-ignore behavior, `Menu`'s `preventScroll: true` initial-focus guard, `MenuItemRadio`'s `onClick` → `defaultPrevented` short-circuit before `onSelect`, and the shared `menuItemClasses` composer contract), and adds the carve-out bookkeeping under `testing-foundation` per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, `test-button-system`, `test-housekeeping`, `test-chip-system`, `test-form-field-system`) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation (per `test-housekeeping`), universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule (write the test OR `/* v8 ignore */` with named rationale — never lower the floor), `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Every requirement applies verbatim.
- `menu-system` (active spec) — 8 active requirements covering Menu popover container (controlled `open`/`onClose`, `role="menu"`, Escape and outside-click dismissal), MenuItem (`role="menuitem"`, icon + tone), MenuLinkItem (Next `<Link role="menuitem">`, same icon + tone API, visually identical to MenuItem), arrow-key navigation across mixed row types (`[role^="menuitem"]` selector, wrap, Home/End, skip `aria-disabled="true"`), focus/touch contract (`--btn-focus-ring-color`, `@media (hover: hover)` guard), trigger ARIA wiring (`aria-haspopup="menu"` / `aria-expanded={open}` on the caller's trigger, optional `anchorRef`), MenuItemRadio (`role="menuitemradio"`, `aria-checked={checked}`, trailing `✓` when checked, optional description), and the migration history (ListActionsMenu/UserAvatarPopover migration; legacy CSS class removal). The tests assert against these SHALLs as the authoritative contract. **No requirements are removed.** A small number of NEW invariants surfaced by reading the source at HEAD are elevated alongside the existing requirements — see "Capabilities" below.
- `button-system` — the menu primitive's focus/touch contract requirement explicitly references "consume the `--btn-focus-ring-color` token and `:focus-visible` model established by `button-system`" and the `@media (hover: hover)` sticky-hover decision. This is a token-surface link, not a transitive SHALL — `<MenuItem>` does not render a `<Button>`. No `button-system` SHALL is asserted by this carve-out's tests directly; the link is honored via the CSS file referencing the right tokens.
- `chip-system`, `form-field-system`, `popover-trigger-system`, `segmented-control-system`, `loading-indicator-system` — unrelated; menu primitives do not render through them.
- `app-frame`, `list-collections`, `list-item-management`, `list-visibility`, `following`, `home-digest`, `items-browser-chrome` — these capabilities consume menu primitives at call sites (e.g., `ListActionsMenu`, `UserAvatarPopover`, the items-toolbar more-menu, list-visibility selector menu) but their behavior contracts are not part of this carve-out. Capability-flow sub-proposals (4.x) own those.
- Cross-cutting design-system rule applies: this proposal is a spec-modification, NOT a page-scoped one-off. Menus are an interactive surface governed by a primitive-family spec; the spec edits here EXTEND the existing `menu-system` spec with elevated call-time SHALLs surfaced during testing. No new tone, no new sibling primitive. Cache-tag rule does NOT apply (no server reads).

## What Changes

- **NEW** colocated test files under `app/ui/components/menu/__tests__/` (per `test-housekeeping`'s relocated convention). One `.test.tsx` per executable JSX component (under jsdom) and one `.test.ts` for the pure helper (under node):

  - `Menu.test.tsx` — `open={false}` returns `null` (no DOM produced); `open={true}` renders a `<div role="menu" class="menu-popover">` with the provided `aria-label` / `aria-labelledby`, containing the children; `className` is appended to `menu-popover` as a space-separated token via the `.filter(Boolean).join(' ')` composition; `ref` is forwarded to the popover `<div>` via `useImperativeHandle` (the ref points at `localRef.current` after mount); Escape key (dispatched on `document`) invokes `onClose` once AND calls `anchorRef.current?.focus()` (verified by spying on `focus` on the anchor element); outside-click (`mousedown` outside both the popover and the anchor) invokes `onClose` AND calls anchor focus; click *inside* the popover does NOT invoke `onClose`; click on the *anchor* element does NOT invoke `onClose` (verified explicitly — this is the dismiss-then-reopen guard); arrow-key navigation: `ArrowDown` from any item focuses the next `[role^="menuitem"]:not([aria-disabled="true"])` (covers MenuItem, MenuItemRadio, MenuLinkItem); `ArrowUp` focuses previous; `Home` focuses first; `End` focuses last; wrap behavior: `ArrowDown` from the last item focuses the first; `ArrowUp` from the first item focuses the last; navigation skips items with `aria-disabled="true"` (the CSS selector enforces this); `preventDefault()` is called on every handled key (verified by passing a custom KeyboardEvent and checking `defaultPrevented`); on open, the first non-disabled item receives focus with `preventScroll: true` (the `preventScroll` argument is captured via `vi.spyOn(HTMLElement.prototype, 'focus')`); a re-mount toggling `open={false} → open={true}` re-runs the effect (subsequent listeners cleaned up); `useEffect` cleanup removes both `mousedown` and `keydown` listeners on unmount and on `open={false}` transition.
  - `MenuItem.test.tsx` — renders a native `<button type="button" role="menuitem" class="menu-item">` containing `icon` followed by `children` in DOM order; `tone="danger"` adds `tone-danger` to the class string via `menuItemClasses`; `tone="default"` (and tone omitted) does NOT add `tone-danger`; `className` prop is forwarded as `extra` to `menuItemClasses` and appended after the tone token; `ref` forwarding lands on the underlying `<button>`; explicit `type` prop overrides the `'button'` default (e.g., `type="submit"`); pass-through `<button>` props (`onClick`, `aria-disabled`, `disabled`, `aria-label`, `data-testid`) reach the underlying button; clicking it invokes `onClick` exactly once.
  - `MenuLinkItem.test.tsx` — renders a Next `<Link role="menuitem" class="menu-item">` (under the test harness, `next/link` resolves to an `<a href>` element with `role="menuitem"`); `tone="danger"` adds `tone-danger` via the shared `menuItemClasses`; `tone="default"` (and omitted) does NOT add it; `className` is forwarded as `extra`; `href` reaches the underlying anchor element; `icon` and `children` render in DOM order; `ref` forwarding lands on the anchor element. The test requires the foundation's existing `next/link` mock (no new mock added — see Decision 3).
  - `MenuItemRadio.test.tsx` — renders `<button type="button" role="menuitemradio" aria-checked={checked} class="menu-item menu-item-radio">`; `className` is appended as a third space-separated token via `.filter(Boolean).join(' ')`; `ref` forwarding lands on the underlying button; `icon` prop renders inside `<span class="menu-item-radio__icon">` only when truthy (omitted prop → no icon span in the tree); `children` render inside `<span class="menu-item-radio__label">`; `description` renders inside `<span class="menu-item-radio__description">` only when truthy (omitted → no description span in the tree); the indicator span (`<span class="menu-item-radio__indicator" aria-hidden>`) ALWAYS renders for the stable-slot contract — when `checked={true}` its text content is `'✓'`, when `checked={false}` its text content is `''` (empty string); the `onClick` handler is called first with the click event; when the consumer's `onClick` calls `e.preventDefault()` (or otherwise sets `defaultPrevented`), `onSelect` is NOT called; when `onClick` does not prevent default, `onSelect` is called exactly once; when no `onClick` prop is provided, `onSelect` is called exactly once (the `onClick?.(e)` optional chain is exercised); `aria-checked` reflects `checked` as the string `"true"` / `"false"` (jsdom DOM attribute representation); the explicit `type` prop overrides the `'button'` default; pass-through props (`disabled`, `aria-disabled`, `aria-label`, `data-testid`) reach the button.
  - `menuClasses.test.ts` — pure node test (`.test.ts` runs under the node project per `vitest.config.ts`): `menuItemClasses()` (no args) returns `'menu-item'`; `menuItemClasses({})` returns `'menu-item'`; `menuItemClasses({ tone: 'default' })` returns `'menu-item'` (default tone does not add a tone class); `menuItemClasses({ tone: 'danger' })` returns `'menu-item tone-danger'` (exact string, in order); `menuItemClasses({ tone: 'default', extra: 'foo' })` returns `'menu-item foo'`; `menuItemClasses({ tone: 'danger', extra: 'foo' })` returns `'menu-item tone-danger foo'` (order is `menu-item`, then `tone-danger`, then `extra`); `menuItemClasses({ extra: '' })` returns `'menu-item'` (empty-string `extra` filtered by `.filter(Boolean)`); `menuItemClasses({ extra: undefined })` returns `'menu-item'` (explicit `undefined` filtered).

- **NO** test for `index.ts` — re-export only, zero executable statements after TS erasure. Excluded under the existing `app/ui/components/*/index.ts` glob in `vitest.config.ts`'s `coverage.exclude` (no change needed).
- **NO** test for `types.ts` — type-only (a single exported union `MenuItemTone = 'default' | 'danger'`), zero executable statements after TS erasure. Already implicitly excluded (no runtime entry); confirm absence from the report.
- **NO** test for `menu.css` — CSS files are not part of the JS coverage report.
- **NEW** ESLint override in `eslint.config.mjs`: extend the existing per-file `sonarjs/cognitive-complexity = error` override array to include each of the five files (`Menu.tsx`, `MenuItem.tsx`, `MenuItemRadio.tsx`, `MenuLinkItem.tsx`, `menuClasses.ts`). Measured complexity at HEAD: `Menu.tsx` is the highest at ~8–10 (two `useEffect` bodies each with a keyboard-event if/else-if ladder; the arrow-key navigation effect drives the ceiling); the four other files are 1–2. The override locks the ceiling at 15 for future edits.
- **NEW** per-file thresholds in `vitest.config.ts` for the five executable files, each referencing the existing `COVERAGE_FLOOR` constant (one entry each — NO per-file numeric variation, per `test-housekeeping`'s single-constant rule).
- **MODIFIED** `menu-system` spec — ADDS four new requirements that elevate call-time invariants the new tests lock against and that are currently implicit in the source:
  - `<Menu>` outside-click handler SHALL ignore clicks on the configured `anchorRef.current` (a click that originates inside the anchor element does NOT call `onClose`) — locks the dismiss-then-reopen guard the source enforces but the spec does not state.
  - `<Menu>` SHALL focus the first non-disabled item on open with `{ preventScroll: true }` — locks the offscreen-menu safety the source's inline comment documents but the spec does not state. The arrow-key navigation requirement already exists; this is the *initial* focus contract.
  - `<MenuItemRadio>` SHALL invoke the consumer's `onClick` prop first with the click event, and SHALL invoke `onSelect` AFTER `onClick` IFF `e.defaultPrevented` is false — this is the "consumer can cancel selection by preventing default" contract the source implements (`onClick?.(e); if (!e.defaultPrevented) onSelect();`) but the spec does not state. The existing "activating it invokes `fn`" scenario covers the no-onClick happy path; this requirement makes the cancel path explicit.
  - `menuItemClasses({ tone, extra })` SHALL compose the wrapper class string as `'menu-item'` then `'tone-danger'` (when `tone === 'danger'`) then `extra` (when truthy), joined by single spaces with falsy tokens filtered — locks the shared composer used by both `MenuItem` and `MenuLinkItem` so the two row types remain visually identical at the class-string level, which the existing "MenuLinkItem and MenuItem are visually identical" scenario asserts only at the visual level. This requirement makes the composer contract explicit and testable.
  These elevations are documented in `design.md` Decision 3.
- **NEW** `testing-foundation` carve-out bookkeeping record (sub-proposal-archive ONLY, per `test-coverage` design D13 two-tier rollup): the menu-system primitive carve-out has landed at the universal `COVERAGE_FLOOR`, the `sonarjs/cognitive-complexity` promotion to `error` is in effect for the five files, and subsequent sub-proposals consuming any menu primitive inherit the assumption that those modules are tested and complexity-locked. Recorded in `openspec/changes/test-menu-system/specs/testing-foundation/spec.md` ONLY (and at archive in `openspec/changes/archive/<date>-test-menu-system/specs/testing-foundation/spec.md`) — does NOT roll into the parent's `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md` (which is created at test-coverage's archive time from the parent accumulator).
- **NEW** four-audit findings recorded in `tasks.md`. Carve-out is medium-sized (five files; the largest is `Menu.tsx` at ~118 LOC with two non-trivial `useEffect` bodies driving the complexity ceiling). Anticipated dispositions: assertion-substance pass on the new tests; duplication likely produces a `__tests__/test-helpers.ts` covering the "render `<Menu open={true}>` with a few `<MenuItem>` children and dispatch a keyboard event" harness (≥3 use sites: the arrow-key, Home/End, and skip-disabled tests all share this shape); complexity audit expected to pass on all five files (Menu.tsx is the only one above ~2; measured ~8–10 against ceiling 15); testability audit expected to surface one or two patterns (the document-level event-listener cleanup; the `useImperativeHandle` ref-forwarding test path).
- **NO source refactors expected.** The five files are testable as-shipped; every branch surfaces as observable DOM, observable callback shape, or observable side effect on a spied function (`focus`, `addEventListener`/`removeEventListener`).

## Capabilities

### New Capabilities

None. The `menu-system` spec already exists.

### Modified Capabilities

- `menu-system`: ADDS four requirements (Menu outside-click anchor-ignore, Menu initial-focus `preventScroll: true`, MenuItemRadio onClick→defaultPrevented short-circuit before onSelect, `menuItemClasses` composer contract) that elevate call-time invariants the new tests lock against. No existing requirements are removed or behavior-changed.
- `testing-foundation`: a carve-out bookkeeping record is written to this sub-proposal's archive-only delta (see "What Changes" above and `test-coverage` design D13). No change to the active `testing-foundation` spec; no change to the parent `test-coverage` accumulator.

## Impact

- **New files:** four `.test.tsx` files (one per executable component) + one `.test.ts` file (for the pure helper) under `app/ui/components/menu/__tests__/`. If duplication surfaces during the §7 audit, a `test-helpers.tsx` joins them under the same `__tests__/` directory (excluded from coverage via the existing `**/__tests__/**` glob). No new shared helpers in `test/helpers/` (the foundation already provides RTL + jest-dom + jsdom shims + the `next/link` mock).
- **Modified config:** `eslint.config.mjs` gains five paths in the per-file override array. `vitest.config.ts` gains five entries in the `thresholds` map, each pointing at `COVERAGE_FLOOR`.
- **Modified source:** none expected. Any audit finding requiring source change is recorded with disposition in `tasks.md`; refactor-in-place commits become part of this change.
- **Modified specs:** `openspec/specs/menu-system/spec.md` gains four ADDED requirements (Menu outside-click anchor-ignore, Menu initial-focus preventScroll, MenuItemRadio onClick→defaultPrevented gate, menuItemClasses composer contract). No removals. The carve-out bookkeeping requirement lives ONLY in this sub-proposal's archive directory (`openspec/changes/archive/<date>-test-menu-system/specs/testing-foundation/spec.md`); it does NOT roll into the parent `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md` (per `test-coverage` design D13 two-tier rollup).
- **CI:** the existing four-gate workflow runs unchanged; the `test` job runtime grows by the five new files (four jsdom render tests + one node test; expected sub-second per file on average, slightly slower for `Menu.test.tsx` due to document-event-listener setup/teardown).
- **Dependencies:** none added.
- **Downstream unblock:** the parent `test-coverage` change's checkbox for sub-proposal 3.4 flips on archive. Capability-flow sub-proposals that render menu primitives at their call sites (4.1 `test-app-frame` for `AppMenu.tsx` / the user-avatar popover; 4.6 `test-list-collections` insofar as ListCard renders the ListActionsMenu; 4.9 `test-list-item-management` for any list-action menus; 4.11 `test-list-visibility` for the visibility-selector menu) benefit from each menu primitive being a known-good module with its contract elevated in the spec.
- **Risk:** moderate. Five files vs. ten for form-field, but one file (`Menu.tsx`) carries non-trivial document-level event-listener wiring and `useEffect`-driven focus management. Realistic miss: failing to model the `preventScroll: true` argument on the initial-focus call, since v8 reports the line as "covered" once the focus call runs at all — the explicit assertion (`vi.spyOn(HTMLElement.prototype, 'focus')` with `expect(spy).toHaveBeenCalledWith({ preventScroll: true })`) is the only way to lock the contract. The new spec requirement (Decision 3b) is explicit on this. Secondary risk: the outside-click anchor-ignore path (Decision 3a) requires constructing a click on the anchor element while the menu is open and asserting `onClose` is NOT called — easy to author wrong by clicking inside the popover instead of on the anchor. The test names this case `OutsideClick_AnchorIgnored` so a future regression that drops the `anchorRef?.current?.contains(target)` early-return fails by name. Tertiary risk: the `MenuItemRadio` `onClick?.(e); if (!e.defaultPrevented) onSelect()` gate — easy to miss the no-onClick path (where the optional chain short-circuits and `defaultPrevented` is still false on the synthetic event); the test names this as `NoOnClick_OnSelectCalled` so the optional chain's absent-callback branch is locked.
- **No runtime behavior change.**
