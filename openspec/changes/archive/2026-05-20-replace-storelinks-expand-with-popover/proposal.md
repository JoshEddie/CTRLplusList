## Why

The store-links row on item cards uses an in-row "expand on hover" choreography for extra stores: the primary buy-link sits next to a `+N` pill, and on hover the extras' `max-width` animates from 0 to natural-width inside a `flex-wrap: wrap` container. When the expanded chips' combined width exceeds the card's inner width — which happens routinely at 2-col viewports and for any item with 2+ extras — the row wraps to a second line, growing the card by ~32px. Because `.item-grid` uses `grid-auto-rows: min-content`, **every card in the same grid row is pulled to the taller height**, producing a visible vertical jolt that the user reads as the page shifting beneath them.

Replacing the in-row expand with a popover-anchored menu (consuming the `<Menu>` + `<MenuLinkItem>` primitives from `standardize-menus-and-controls`) eliminates the height jump by construction: the chip row stays single-row at all times and the extras live in a floating panel that doesn't participate in the card's layout. As a bonus the popover scales cleanly to many stores (5, 10, 20) where the current expand pattern would have overflowed the card.

## What Changes

- **Remove the expand-collapse choreography** from `StoreLinks.tsx`: delete `expanded` state, `collapseTimer`, `COLLAPSE_DELAY_MS` and the `cardRef`/`collapseBoundaryRef` plumbing in `Item.tsx`. The "+N" chip becomes a **menu trigger** (`<Button variant="ghost" size="sm" aria-haspopup="menu" aria-expanded={open}>`) that opens a `<Menu>` containing `<MenuLinkItem>` rows for each extra store.
- **Restructure `.storeLinks` layout** from flex-wrap to CSS grid:
  - Single store (no extras): `grid-template-columns: 1fr` — primary buy-link stretches to fill the card width.
  - Multiple stores (extras present): `grid-template-columns: 1fr auto` — primary buy-link stretches, `+N` trigger sizes to content.
  - `flex-wrap: wrap` is removed entirely; the row is always exactly one line tall.
- **Drop `overflow: hidden` from `.item-container`** (Option 2 of the explored alternatives — chosen over portaling). Move the rounded-corner clip onto the surfaces that need it: `border-top-left-radius` / `border-top-right-radius` on `.item-image-container` (already has `overflow: hidden`); `border-bottom-left-radius` / `border-bottom-right-radius` on `.purchased-banner` for the case where it's the bottom-most element. This keeps the visual treatment identical while letting the popover panel render outside the card boundary without clipping. **BREAKING (visual)**: any future card-internal content that depended on `overflow: hidden` for clipping must opt in at its own surface.
- **Delete the `.storeLinks-link--extra` machinery** entirely from `app/(main)/items/ui/styles/store-links.css`: the `max-width: 240px ↔ 0` transition, the `.storeLinks:not(.is-expanded)` / `.storeLinks.is-expanded` rules, the `prefers-reduced-motion` carve-outs, and the `is-expanded` class itself. Extras are no longer rendered in the chip row at all.
- **Reuse the `<Menu>` primitive** (from `standardize-menus-and-controls`) as-is for the popover panel: arrow-key navigation between menu items, Escape to close, focus return, click-outside dismissal via `usePopoverDismiss`. The wrapper (`StoreLinks.tsx`) owns the open/close state and the anchor ref. **Hover-to-open is preserved** as an additional affordance on top of the menu's click-to-open default: `onMouseEnter` on the `+N` trigger sets `open=true`; `onMouseLeave` from the trigger ∪ panel boundary schedules a 220ms grace close (same delay as today's `COLLAPSE_DELAY_MS`). Click and keyboard activation use `<Menu>`'s built-in model.
- Update the openspec change `standardize-buttons` (where §8.6 of tasks.md describes the chip-dimension override and §10.3 of tasks.md describes the `.storeLinks-link--extra` animation hooks) with a referencing comment noting that this change supersedes that pattern. The chip dimension override resolution is moot: the wrap problem is gone, so the design-system 32px `sm` chip is fine without a page-scoped height override.

## Impact

- **Components**: `app/(main)/items/ui/components/StoreLinks.tsx` (significant rewrite — remove expand state, add menu wiring, restructure JSX), `app/(main)/items/ui/components/Item.tsx` (remove `cardRef` and `collapseBoundaryRef` prop passing).
- **CSS**: `app/(main)/items/ui/styles/store-links.css` — replace `.storeLinks` flex-wrap layout with grid; delete `.storeLinks-link--extra`, `.storeLinks:not(.is-expanded)`, `.storeLinks.is-expanded`, and the `prefers-reduced-motion` carve-outs that reference removed selectors. `app/(main)/items/ui/styles/item.css` — remove `overflow: hidden` from `.item-container`; add `border-top-{left,right}-radius` to `.item-image-container`; add `border-bottom-{left,right}-radius` to `.purchased-banner`.
- **Dependencies**: `standardize-menus-and-controls` MUST be implemented and merged before this change ships. We consume `<Menu>`, `<MenuLinkItem>`, and `usePopoverDismiss` directly; building a throwaway hand-rolled popover here would be pointless when the primitive is in flight.
- **Cross-change coordination**: a follow-up edit to `openspec/changes/standardize-buttons/proposal.md` and `tasks.md` is required — add a comment at the §8.6 and §10.3 entries noting that `replace-storelinks-expand-with-popover` supersedes the expand-collapse-with-chip-dimension-override approach. The migration in §8.6 (Button/LinkButton primitives for the row's three button types) stays valid; only the expand-choreography piece is replaced.

- **Row-view redesign (scope expansion)**: in addition to the popover work above, this change redesigns the list/sortable row layout (`.item-list .item-container`, `.sortable-item .item-container`). New row anatomy: `[image] [name top / price+leader-dots bottom] [tall buy-link pill] [+N popover trigger] [right-column action area]`. The right column adapts per viewer: owner → edit/archive icons (or kebab `<Menu>` at <400px); non-owner unclaimed → "Claim" button; anyone claimed/spoiler → a wide claimed-state pill that absorbs the buy-link + +N + action slots (cols 3–5 of the inner grid). The `.purchased-banner` footer is hidden in row view (its info is folded into the wide pill); it remains in grid view. Grid (card) view is explicitly untouched.

- **Mobile horizontal-card layout (scope expansion)**: at viewport widths <600px the row layout reflows from the desktop D3 shape into a vertically-stacked horizontal card — image upper-left spanning two rows, title on row 1 right-of-image, price on row 2 right-of-image, description full-width on row 3, action row (buy-link + `+N` left, viewer-aware right-actions right) full-width on row 4. The earlier attempt to compress the 5-column desktop grid into mobile widths produced unreadable cramming (~80px for the name column at 380px viewport, with item names truncating to 2–3 characters and prices clipping); the horizontal-card pattern is the agreed answer that keeps list view available on mobile rather than forcing grid. Kebab `<Menu>` for owner actions still engages at <400px. Breakpoints: D3 ≥600px (two-row content + leader dots, tall pill); M1 <600px (horizontal-card stack, no leader dots, compact pill); M1-kebab <400px adds the owner-actions kebab swap.

- **Choose-items migration (scope expansion)**: `.choose-items-row` at `/lists/[id]/choose-items` is the third parallel row-layout system in the codebase (alongside `.item-list` and `.sortable-item`). It was missed during the WCAG row-layout reunification work and represents drift. This change migrates choose-items to share the same row primitive — at both desktop and mobile breakpoints — with the checkbox in the leading slot (analogous to how sortable-item puts the drag handle in the leading slot). Choose-items adopts the shared row's title/price-with-leader/description-row/store-chip-row anatomy, with its existing selection state modifiers (`.is-on`, `.is-removing`) layered on as additive background-color rules. **Description is rendered in choose-items rows** (new behavior; was previously omitted). `.choose-items-row`'s bespoke 4-column grid template, hover background, color tokens, and dimensions are deleted in favor of the shared rules.

- **Sortable owner buy-link chips kept visible**: `.sortable-item .storeLinks` continues to render at all viewports for parity with the items library view. Owners may want quick access to where they originally added a link from any surface that lists their items. Earlier scoping considered stripping these for the sortable view but the decision is to retain them.
- **Risk**: the `overflow: hidden` removal on `.item-container` is the highest-risk piece — if any other card-internal element silently depended on the container's clipping (e.g. a hover transform that would now reveal an artifact past the rounded corner), it surfaces here. Mitigation: visual review at each viewport (mobile 2-col, tablet 3-col, desktop 4/6-col) and in both card states (active, purchased, with/without purchased-banner footer).
- **Out of scope**: any redesign of the popover panel's internal content (it's a `<Menu>` of `<MenuLinkItem>` rows — same shape as `ListActionsMenu` post-`standardize-menus-and-controls`); any change to the primary buy-link's variant or sizing (stays `<LinkButton variant="primary" size="sm">`); the `Purchase` button rendered as children of `StoreLinks` (untouched); the `claim-counter` and `purchased-banner` semantics (untouched beyond the corner-radius addition).
