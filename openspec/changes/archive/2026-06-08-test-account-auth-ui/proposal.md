## Why

Sub-proposal **9.3** of the `test-coverage` initiative ([issue #113](https://github.com/JoshEddie/CTRLplusList/issues/113)) — a coverage-gap follow-up discovered at the §7.1 close-out audit (`openspec/changes/test-coverage/tasks.md` §9.3).

The auth & account-menu **client UI** is at **0% unit coverage** today. The sign-in *flow* is e2e-covered by 6.1 (`test-e2e-critical-flows`, [issue #55](https://github.com/JoshEddie/CTRLplusList/issues/55)) — the guest spec navigates to `/sign-in` and asserts the logo + "Sign in with Google" button — but the **components** that render that surface, and the account-menu (avatar popover) the signed-in viewer touches, have no unit tests. Every live file below is absent from `vitest.config.ts` `thresholds` (verified against the live map) and has no colocated test.

A spec-grep (every active `spec.md` grepped for the component names, file paths, DOM strings, and behaviors these files touch) surfaced a **dead-code cascade** that reshapes the issue's named carve-out. The issue names `app/ui/components/AuthPage.tsx` (exporting `AuthProvider`) as the primary file, but that component has **zero importers** repo-wide — it is the **old chrome entry point**, superseded by `AppFrame` (the `1.0: migrate auth and shared chrome to new primitives` commit). The live account-chrome path is `app/(main)/layout.tsx → AppFrame → User → UserMenu → UserAvatarPopover` (signed in) / `SignedOutMenu` (signed out); the live sign-in path is `app/(auth)/sign-in/page.tsx → SignInPage → AuthContainer + SignInButton`. `AuthProvider` renders neither. Deleting it transitively orphans `AppMenu.tsx → Logo.tsx`/`Nav.tsx` (each imported *only* by the previous link), which are already floored + tested under the archived carve-out **4.1** (`test-app-frame`). Per the operator's close-out decision (recorded in §9.3 authoring), this carve-out disposes the **entire old-chrome cluster by deletion** and unit-covers the **live** auth/account UI.

Inherited constraints (binding SHALLs cited as-is — none re-owned here):

- **`testing-foundation`** (active accumulator at `openspec/changes/test-coverage/specs/testing-foundation/spec.md`, plus every prior sub-proposal's archived deltas) — governs verbatim: the vitest 4.x runner with the jsdom/node two-project split (`.test.tsx` → jsdom, `.test.ts` → node), four-gate pre-merge, `__tests__/` colocation, the universal per-file `COVERAGE_FLOOR` (`lines:98 / statements:98 / branches:95 / functions:100`) referenced from the single `vitest.config.ts` constant, the no-backdoor disposition rule (`/* v8 ignore */` with a named rationale, never lower the floor), the assertion-substance bar, the `<State>_<Behavior>` `it()` shape and three-role `describe()` convention, the four-audit + invariant-elevation obligations, and the `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Two load-bearing mocking allowances apply: **(1) NextAuth is the network boundary** — `@/lib/auth`'s `auth()` is mocked, never the modules it wraps; **(2) `next/navigation`'s `redirect()` is mocked to throw a sentinel** for RSC guards (the `FollowingPage.test.tsx` / `ChooseItemsBody.test.tsx` precedent).
- **`menu-system`** (active) — **already specs `UserAvatarPopover.tsx`**: its requirement *"UserAvatarPopover uses Menu primitive"* binds the popover to render as a `<Menu>` containing the signed-in user header (name + email), a `MenuLinkItem` to `/settings/connections` ("Connections"), and a submit `MenuItem` ("Sign out") inside a `signOutUser` form. This carve-out **LOCKS** that requirement at the component level (rendered through the **real** `Menu`/`MenuItem`/`MenuLinkItem`, already floored) — no `menu-system` requirement is modified.
- **`e2e-critical-flows`** (active) — its *"Sign-in surface"* requirement specs that "the AuthPage sign-in UI renders … the 'Sign in with Google' button" for a session-less visitor. The prose name "AuthPage" is **legacy terminology for the sign-in surface**; the e2e test (`e2e/auth.guest.spec.ts`) navigates to `/sign-in`, which renders `SignInPage` — **not** the dead `AuthPage.tsx` file. Deleting `AuthPage.tsx` therefore does not touch the e2e flow. Per the issue's coordination note, the two tiers are kept non-redundant: **e2e** asserts the real navigation + rendered button for the logged-out viewer; this **unit** carve-out asserts the *branch logic* the e2e cannot cheaply reach (the `SignInPage` redirect-when-already-authenticated guard, the `UserMenu` signed-in/out surface choice, the action-dispatch wiring). The stale "AuthPage" prose name is recorded as an audit observation (see `design.md`), not modified here.
- **`app-frame`** (active) — describes the **live** `AppFrame`/`AppNav` gradient nav (brand lockup, primary pills, viewer avatar at right). It never names `AppMenu`/`Logo`/`Nav` (the old primitives), so deleting the old-chrome cluster requires **no** `app-frame` delta — the spec already documents the surviving path.
- **Primitive-family specs rendered *through*, not re-owned** — `menu-system` (`Menu`/`MenuItem`/`MenuLinkItem` in `UserAvatarPopover`), `button-system` (`Button`/`buttonClasses` in `SignInButton`/`UserMenu`). Both are already tested and floored; this carve-out renders through the **real** primitives and asserts no primitive SHALL directly.

Cache-tag note: none of these files own a server-side read or mutation tag. The two server actions they dispatch (`signInUser` / `signOutUser`) are owned and floored by **4.13** (`test-server-endpoint-authorization`, `app/actions/user.ts`); they are **module-mocked** at the boundary in the component tests (the established `vi.mock('@/app/actions/...')` pattern), so the tests assert *which action the UI wires to a form*, not the auth side-effects.
