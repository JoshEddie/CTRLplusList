## Why

Sub-proposal 4.7 of the `test-coverage` initiative — a capability-flow carve-out for the `list-hero-header` capability, the list-detail hero panel. The parent `test-coverage` change's `tasks.md` §4.7 names the carve-out as "`list-hero-header` capability including contrast invariants." The capability already has a **substantive active spec** at `openspec/specs/list-hero-header/spec.md` (created by archiving `redesign-list-hero`, eight requirements covering single-panel gradient composition, identity/controls zones, the occasion eyebrow, visibility-picker placement, owner/viewer/preview action sets, and a WCAG-AA contrast requirement). Its `Purpose` is still the post-archive `TBD` placeholder. So this sub-proposal (a) locks the existing SHALLs against colocated test files at the universal `COVERAGE_FLOOR`, (b) fills in the spec `Purpose` and MODIFIES one requirement (the visibility-picker-placement SHALL gains a scenario forbidding an empty `.list-hero-share-wrapper` on viewer/preview views — locking the source fix below), (c) fixes two source findings the assertion/duplication audits surface (a nested empty `.list-hero-share-wrapper`, and visibility-derivation logic copy-pasted between `ListDetails.tsx` and `ShareButton.tsx`), (d) lands an enforceable automated check for the contrast requirement (R8) — the "including contrast invariants" deliverable — via an sRGB relative-luminance helper that reads the gradient + text-color tokens straight from the CSS, and (e) adds the carve-out bookkeeping under `testing-foundation` Tier 2 per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep across every active spec (not just the capability folder):

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from the foundation + every primitive-family + `test-app-frame` carve-out) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy, shared helpers under `test/helpers/`, NextAuth-as-network-boundary mocking allowance. Every requirement applies verbatim.
- `list-hero-header` (active spec) — 8 active requirements. The tests assert against these SHALLs as the authoritative contract. **No requirements are added or removed**; the source is brought into conformance where it has drifted (the nested share-wrapper), and the contrast requirement (R8) gains an automated enforcer. The `Purpose` placeholder is filled. Layout-only requirements (the single continuous gradient panel, side-by-side-vs-stacked zones, `space-between` height-matching, the hairline divider) are CSS contracts not exercisable in jsdom — they are covered by DOM-structure assertions (which elements/classes exist, DOM order, owner/viewer/preview composition) plus the contrast check; full-layout verification is e2e territory (6.x). See `design.md` Decision 7.
- `list-visibility` (active spec) — owns the `<VisibilityPicker>` internal composition and the three-state vocabulary. The hero spec governs only the picker's **placement**. `VisibilityPicker` is therefore OUT of this carve-out (mocked in `ListDetails` tests); its behavior is owned by 4.11 `test-list-visibility`. The shared visibility-derivation fallback (`list.visibility ?? (list.shared ? LINK : OWNER)`) is extracted to `lib/visibility.ts` (a `test-pure-libs`/2.1 surface) and tested here as a duplication-audit fix.
- `following` (active spec) — owns `<Avatar>`, the linked owner-name, and the Follow button placement/colocation in the viewer byline. `Avatar` and `FollowContainer` are OUT (mocked); their behavior is owned by 4.2 `test-following`. The hero locks only that the byline group renders them in order.
- `list-collections` (active spec) — owns the bookmark collection; `<BookmarkContainer>` / `BookmarkButton` are OUT (mocked), owned by 4.6.
- `list-hero-collapse` (active spec) — owns `HeroCollapseShell`, `HeroCollapsedItems`, `HeroCollapsedItemsContainer`. These are OUT (4.8). `HeroCollapseShell` is mocked to a transparent children-passthrough in `ListDetails` tests so the hero's own composition is asserted without exercising the collapse machinery.
- `button-system`, `menu-system`, `segmented-control-system`, `form-field-system` — `ListDetails`/`ShareButton`/`EditListAction` consume `<Button>` / `<LinkButton>` / `<Menu>` via public APIs. Those primitives are already tested under 3.1–3.6 (archived). Token-surface link only; no primitive-family SHALL is asserted from this carve-out. Per the cross-cutting design-system rule, NO page-scoped override of `.btn` / `.menu-item` is introduced (the hero spec's own R "no page-scoped primitive override" requirement is honored).
- `list-item-management` (4.9, pending) — owns `ListActionsMenu` (the kebab invokes `deleteList` from `app/actions/lists.ts`) and `EditListAction`'s downstream `ListFormContainer`/`ListForm`. `ListActionsMenu` is OUT of this carve-out (mocked); see `design.md` Decision 2 for the boundary rationale and the recommendation that 4.9 claim it. `EditListAction` itself (the hero's owner-edit trigger button) IS in carve-out; its modal child `ListFormContainer` is mocked.

No server-side read is added or modified by this change; `ListDetails` is an async server component but performs no `await` and reads no `'use cache'` surface (it receives all data via props from `ListHeroSection`). No `cacheTag` / `revalidateTag` coordination applies.
