## Why

Sub-proposal 4.2 of the `test-coverage` initiative (parent `tasks.md` §4.2; GitHub issue #41). This is the **first capability-flow carve-out that exercises the data layer end-to-end** — a server action that mutates the follow graph (`app/actions/follows.ts`) and the DAL reads that observe it (`getFollowingByUser`, `getFollowersOfUser`, `getFollowingFeedUsers`, plus the three follow-graph predicates `isFollowing` / `isBlocked` / `viewerHasAnyFollows` consumed by the in-carve-out `FollowContainer`), wired up to the page UI under `app/(main)/following/` and the follow/profile UI components under `app/(main)/users/ui/components/`.

No DAL function or server action is tested anywhere in the repo today — only the pglite harness (`test/helpers/db.ts`) and the `next/cache` mock (`test/helpers/next-cache.ts`) exist as plumbing the `test-foundation` spike landed. This carve-out is therefore the one that **establishes the canonical DAL + server-action integration test pattern** (mock `@/db` → a migrated pglite instance, no-op `cacheTag` / capture `updateTag` via `mockNextCache()`, mock `@/lib/auth` at the NextAuth boundary). That pattern is a Tier 1 `testing-foundation` contribution — every downstream data-layer sub-proposal (4.3 home-digest, 4.9 list-item-management, 4.11 list-visibility, 4.13 server-endpoint-authorization, 4.14 visit-history, 4.15 user-actions) inherits it.

Inherited constraints surfaced by spec-grep across all active specs:

- **`testing-foundation`** (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus the archived deltas from `test-foundation-spike`, `test-foundation`, `test-housekeeping`, `test-pure-libs`, and the primitive-family + `test-app-frame` carve-outs) — every requirement applies verbatim: vitest 4.x with the jsdom (`.test.tsx`) / node (`.test.ts`) two-project split, the four-gate pre-merge, `__tests__/` colocation, the universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, the no-backdoor disposition rule (test it OR `/* v8 ignore */` with a named rationale — never lower the floor), the `<StateUnderTest>_<ExpectedBehavior>` `it()` shape, the three-role `describe()` convention, the four-audit obligation, the invariant-elevation audit, the assertion-substance bar (no tautologies / execute-for-coverage), the `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy, the rate-limited-external-service prohibition, and the NextAuth-as-network-boundary mocking allowance.
- **`following`** (active spec, `openspec/specs/following/spec.md`) — 12 active requirements. The tests assert against these SHALLs as the authoritative contract. Directly relevant to this carve-out: "Users SHALL follow and unfollow other users" (idempotent follow, no-self-follow, block-gated follow, unfollow no-op), "First follow by a viewer SHALL surface a disclosure dialog" (the `viewerHasAnyFollows`-derived signal → `FollowDisclosureDialog`), "List pages SHALL expose a follow affordance … hidden when blocked" (the `FollowContainer` returns `null` when either-direction block), and "Follow-graph mutations SHALL NOT use interactive transactions" (the neon-http sequential-statement constraint + the inline `after()` last-seen write in `FollowingPage`). **One latent invariant is elevated** (see Capabilities): the issue's "partial unique index backstop" language is inaccurate — the actual de-dup backstop for follow-toggle races is the **composite primary key `(follower_id, followee_id)`** on `user_follows` (`db/schema.ts:137`), reached via `onConflictDoNothing()` in `followUser`. No requirement currently names the mechanism; this change adds one.
- **`button-system`** — `FollowButton`, `FollowDisclosureDialog`, and `ProfileHeader` render through the `Button` / `LinkButton` primitives (already tested + complexity-locked under `test-button-system`). Token-surface link only; the primitives are NOT re-tested here and are NOT mocked — rendering through the real primitive asserts the integration. No `button-system` SHALL is asserted from this carve-out's tests, and no `button-system` spec change is proposed.
- **`list-collections`** — `FollowingPage` renders `<ListCollectionsNav />`, `PublicListsGrid` / `UserCardGrid` render `<ListCard>` / `<MoreCard>`. These are owned by `list-collections` (sub-proposal 4.6) and are out of this carve-out; they are module-mocked to stubs where rendering them would pull in out-of-carve-out behavior, OR rendered as real children where they are pure pass-throughs (decided per-file in `design.md` Decision 7).
- **`home-digest`** — `getFollowingFeedUsers` ALSO powers the home page's Following rail (sub-proposal 4.3). It matches the issue's explicit `getFollowing*` glob AND backs the in-carve-out `/following` page, so this change owns its tests; 4.3 inherits it as already-tested. The boundary is stated in `design.md` Decision 1.
- **Cross-cutting design-system rule** applies: no page-scoped one-off classes are introduced. Every interactive surface in the carve-out already routes through a governed primitive.
- **Data-freshness rule** applies: `isFollowing` / `isBlocked` / `viewerHasAnyFollows` are `'use cache'` reads tagged `user_follows` / `user_blocks`; the mutations that must revalidate them all live in the in-carve-out `app/actions/follows.ts` (`updateTag('user_follows')`, `updateTag('user_blocks')`) and the inline `after()` write in `FollowingPage`. Tests assert the `updateTag` calls fire (and only fire on the success path).

## What Changes

- **NEW** colocated test files under `__tests__/` directories mirroring source. Two run under the **node** project (`.test.ts`, DAL + server action against pglite); the rest run under **jsdom** (`.test.tsx`, components):

  - `app/actions/__tests__/follows.test.ts` (node) — the highest-value file. Boots a migrated pglite DB per test, mocks `@/db` to point at it, `mockNextCache()`, and mocks `@/lib/auth`'s `auth()` to control the viewer session. Asserts for **`followUser`**: inserts a `user_follows(follower=viewer, followee=target)` row; is idempotent on a second call (no duplicate, no throw — the composite-PK `onConflictDoNothing()` backstop); rejects self-follow (`Invalid`, no row); rejects when either direction is blocked (`Blocked`, no row); returns `Unauthorized` when `auth()` yields no session; fires `updateTag('user_follows')` exactly once on success and NOT on the unauthorized / self / blocked early-returns; returns `{ success:false, error:'Failed' }` and does NOT call `updateTag` when the insert throws (DB error injected). **`unfollowUser`**: deletes the matching row; is a no-op success when no row exists; `Unauthorized` with no session; `updateTag` on success only. **`removeFollower`**: deletes the `(follower=other, followee=viewer)` row (direction inverted from unfollow); `Unauthorized`; no-op success. **`blockUser`**: inserts the `user_blocks` row AND deletes both-direction `user_follows` rows; the block-first ordering is asserted (a follow row inserted by a racing `followUser` after the block-insert is still gated by the block check); rejects self-block; `Unauthorized`; fires BOTH `updateTag('user_follows')` and `updateTag('user_blocks')` exactly once each on success and neither when a statement throws; idempotent re-block cleans up leftover follow rows. **`unblockUser`**: deletes the `user_blocks` row; `Unauthorized`; `updateTag('user_blocks')` on success only. No code path calls `db.transaction(...)` or `tx.*` (asserted by the absence of a transaction surface on the mocked `db`).

  - `lib/__tests__/dal.following.test.ts` (node) — boots pglite, mocks `@/db` + `mockNextCache()`, seeds users / follows / blocks / public lists directly, asserts: **`getFollowingByUser`** returns the viewer's followees joined with `{id,name,image}`, newest-follow-first (`created_at DESC`), empty array for a user who follows no one; **`getFollowersOfUser`** returns followers (inverted direction) with the same join + ordering; **`getFollowingFeedUsers`** returns followees with `latest_shared_at = MAX(shared_at)` over their `public` lists, `new_count` counting public lists shared after `GREATEST(last_seen_following_at, follow_created_at)`, `new_count` coerced from the Postgres count-string to a `number`, ordered by `MAX(shared_at) DESC`, a followee with no public lists surfacing `latest_shared_at = null` / `new_count = 0`, and the `GREATEST` clamp suppressing a followee's pre-existing public lists when `last_seen_following_at` is null but the follow is newer; **`isFollowing`** true/false; **`isBlocked`** true/false (directional); **`viewerHasAnyFollows`** true when ≥1 follow, false when zero (drives the first-follow disclosure). The cache scope is exercised through `mockNextCache` (no-op `cacheTag`).

  - `app/(main)/users/ui/components/__tests__/FollowButton.test.tsx` (jsdom) — label is `"Following"` when `following`, `"Follow {userName}"` when not + name present, `"Follow"` when name null; renders `<FaCheck>` when following and `<FaPlus>` when not; `aria-disabled` reflects `pending`; `aria-label` equals the label; `pressed` reflects `following`; `onClick` fires on click; renders through the real `Button` primitive (variant pass-through).

  - `app/(main)/users/ui/components/__tests__/FollowControls.test.tsx` (jsdom) — the client state machine. Mocks `@/app/actions/follows` (`followUser`/`unfollowUser`), `next/navigation` `useRouter`, and `react-hot-toast`. ## Capabilities

### New Capabilities

None. Both touched capabilities (`following`, `testing-foundation`) already have substantive active specs / accumulators.

### Modified Capabilities

- `following`: ONE requirement is ADDED — the composite primary key `(follower_id, followee_id)` on `user_follows` is the de-dup backstop for follow-toggle races (reached via `onConflictDoNothing()`), correcting the issue's "partial unique index" phrasing and locking the mechanism the idempotent-follow scenario depends on. ## Impact

- **New files:** two node-project `.test.ts` files (`app/actions/__tests__/follows.test.ts`, `lib/__tests__/dal.following.test.ts`) and eleven jsdom `.test.tsx` files (one per executable UI module + the two `following/` route files). A shared follow-graph seeding helper under `test/helpers/` MAY be added if the §duplication audit surfaces 3+ reuse sites.
- **Modified config:** `eslint.config.mjs` gains the carve-out's executable paths in the per-file `sonarjs/cognitive-complexity = error` override array. `vitest.config.ts` gains one `thresholds` entry per executable file, each pointing at `COVERAGE_FLOOR`.
- **Modified source:** one in-carve-out refactor — extract the triplicated `initialsOf` helper (disposition recorded in `tasks.md` §audits). No other source changes; the action / DAL / components are testable as-shipped.
- **Modified specs:** `openspec/specs/following/spec.md` gains one ADDED requirement (composite-PK dedup backstop). The Tier 1 `testing-foundation` DAL/action-harness requirement is written to this sub-proposal's `specs/testing-foundation/spec.md` and rolls into `openspec/changes/test-coverage/specs/testing-foundation/spec.md` at apply. The Tier 2 carve-out bookkeeping lives ONLY in the sub-proposal archive.
- **CI:** the existing four-gate workflow runs unchanged; the node project gains its first DAL/action suites (each boots a fresh pglite per test — fast, in-process, no external DB). The image-search rate-limit prohibition is not engaged (no external service touched).
- **Dependencies:** none added — `@electric-sql/pglite` and the pglite drizzle driver are already present from the foundation spike.
- **Downstream unblock:** the parent `test-coverage` checkbox for §4.2 flips on archive. The Tier 1 pglite DAL/action harness pattern unblocks every subsequent data-layer carve-out (4.3, 4.9, 4.11, 4.13, 4.14, 4.15) — they inherit the established pattern rather than re-deriving it.
- **Risk:** medium. Notable risks:
  - **First-of-its-kind DAL/action harness.** `vi.mock('@/db')` must resolve to a single shared pglite instance across the module-under-test's static `import { db } from '@/db'`. The hoisting of `vi.mock` requires a mutable holder pattern; pglite boot is async while `vi.mock`'s factory is sync. Mitigation + the exact pattern are in `design.md` Decision 2; if the static-import binding cannot be swapped cleanly, the fallback is a thin `getDb()` indirection in the carve-out (a testability refactor inside scope).
  - **`'use cache'` directive under vitest.** The cached DAL reads (`isFollowing` etc.) carry the Next `'use cache'` directive; under the node project (no Next compiler) it is an inert string and `cacheTag` is no-op'd by `mockNextCache()`. Verified pattern; documented in Decision 3.
  - **jsdom `<dialog>` support.** `HTMLDialogElement.showModal()` / `close()` are not fully implemented by jsdom; `FollowDisclosureDialog.test.tsx` stubs them and asserts on the calls + `open` attribute rather than real top-layer behavior. Documented in Decision 6.
  - **Async server-component rendering** (`FollowContainer`, `FollowingPage`). Same React 19 + RTL consideration as `test-app-frame`; fallback is direct invocation (`const tree = await FollowContainer(props); render(tree)`). Documented in Decision 4.
  - **The composite-PK spec correction is a behavioral claim worth defending.** The issue text says "partial unique index"; the schema says composite PK. The ADDED SHALL fixes the record and the idempotency test locks the `onConflictDoNothing()` path; a future migration that drops the PK would fail the test loudly.
  - **`getFollowingFeedUsers` ownership overlap with home-digest (4.3).** Claimed here per the explicit glob; 4.3 must NOT re-add a duplicate suite. Boundary noted in Decision 1 and in this proposal's "NO test" list.
- **No runtime behavior change** other than the `initialsOf` extraction (behavior-preserving, locked by the new tests).
