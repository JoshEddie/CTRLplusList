## Why

Sub-proposal 3.1 of the `test-coverage` initiative. With `test-foundation`, `test-foundation-spike`, and `test-pure-libs` archived, the button family is the lowest-risk primitive carve-out to land next: it is the first cross-cutting design-system primitive whose `buttonClasses` composer was already tested in 2.1, and the two thin React shells (`Button.tsx`, `LinkButton.tsx`) consume that composer plus a handful of accessibility-bearing prop translations (`isLoading` → `disabled` + `aria-busy` + spinner; `pressed` → `aria-pressed`). Landing this slice exercises the foundation against JSX-rendering primitive components for the first time, validates the per-file 90% Primitive-component floor, and unblocks the remaining `3.x` family sub-proposals which all follow the same shape.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (governing `test-coverage` change + archived `test-foundation` + `test-foundation-spike` + `test-pure-libs`) — runner, layout, mocking rules, fixture sources, coverage floors (90% per file for primitive components), complexity threshold, four-gate pre-merge, four-audit obligation, assertion-substance bar, two-part `<State>_<Behavior>` Vitest `it()` shape, three-role `describe()` convention. Every requirement applies verbatim.
- `button-system` (active spec) — authoritative on every observable contract `<Button>` / `<LinkButton>` carry: focus indicator via `:focus-visible`, 44 CSS-pixel touch floor at the standard size, `sm` opt-out floor, hover-on-touch guard via `@media (hover: hover)`, **loading state SHALL set `disabled` + `aria-busy="true"` and render a spinner alongside (not replacing) the children** (Button only — LinkButton does not support loading), **toggle state is orthogonal to variant and emitted as `aria-pressed` only when `pressed` is defined (undefined → attribute omitted)**, six variants `primary | secondary | ghost | danger | on-dark | link`, two sizes `sm | md` (default `md`), shared `buttonClasses({ variant, size, extra })` composer (covered by 2.1, NOT re-tested here), icon-only buttons SHALL carry an accessible name (component-level: the *requirement* lives at the call site, but the components SHALL accept and forward `aria-label`). Tests SHALL exercise behavior already locked by these SHALLs.
- `list-visibility`, `following`, `server-endpoint-authorization`, `list-item-management` — these specs *use* `<Button>` / `<LinkButton>` at call sites (bookmark, follow, claim, delete) but their authorization / state semantics are not part of this carve-out. The components are agnostic to which domain prop they carry. No transitive SHALLs apply.

Cross-cutting design-system rule applies (the change touches an interactive surface): the carve-out is the primitive itself; no page-scoped className overrides are introduced, no new variant/size is added. Cache-tag rule does NOT apply (no server reads).

## What Changes

- **NEW** test files (in a `__tests__/` directory colocated with the carve-out, per TESTING.md's location convention):
  - `app/ui/components/button/__tests__/Button.test.tsx` — jsdom + RTL render assertions covering: (a) every variant × size combination forwards the correct class string via `buttonClasses` (rendered element's `className` is asserted); (b) `isLoading={true}` sets `disabled`, `aria-busy="true"`, renders the `.btn-spinner` span, wraps `children` in `.sr-only` (loading-state contract from `button-system` Requirement: "Loading state disables the button and announces busy status"); (c) `isLoading={false}` / omitted renders `children` directly with no spinner, no `aria-busy`; (d) `pressed={true}` emits `aria-pressed="true"`, `pressed={false}` emits `aria-pressed="false"`, `pressed` omitted emits NO `aria-pressed` attribute (toggle-state contract from `button-system` Requirement: "Toggle state is orthogonal to variant and uses aria-pressed"); (e) `type` default is `"button"` (NOT browser default `"submit"`), explicit `type="submit"` passes through; (f) `disabled` passes through, AND `disabled || isLoading` short-circuits via OR (a `disabled={false} isLoading={true}` button is still disabled); (g) `className` is forwarded as `extra` to `buttonClasses` and appears on the rendered element; (h) `forwardRef` resolves to the rendered `<button>` element (asserted via `ref.current.tagName === 'BUTTON'`); (i) arbitrary `<button>` HTML attributes (`onClick`, `aria-label`, `data-*`, `name`, `value`) pass through unchanged; (j) `onClick` does NOT fire when `isLoading={true}` (the `disabled` attribute prevents the event per HTML).
  - `app/ui/components/button/__tests__/LinkButton.test.tsx` — jsdom + RTL covering: (a) every variant × size combination produces the correct class string on the rendered `<a>`; (b) `href` is forwarded to the Next `<Link>` and appears as `href` on the rendered `<a>`; (c) `pressed={true|false|undefined}` mirrors the Button toggle-state contract; (d) `className` forwarded as `extra`; (e) `forwardRef` resolves to the rendered `<a>` element (`ref.current.tagName === 'A'`); (f) arbitrary anchor attributes (`onClick`, `aria-label`, `data-*`, `target`, `rel`) pass through; (g) `children` rendered as the link's content; (h) **LinkButton does NOT accept `isLoading`** — the TypeScript type does not include it, AND no spinner span is ever rendered. This is an explicit non-feature locked by the `button-system` spec ("`<LinkButton>` does NOT support a loading state in this change"); the test SHALL assert the absence as a contract guard.
- **NO** test for `index.ts` (re-export only, zero executable statements after TS erasure; same class as `lib/types.ts` in 2.1).
- **NO** test for `types.ts` (type-only, same class as `lib/types.ts` in 2.1; both files SHALL be added to `vitest.config.ts`'s `coverage.exclude` patterns).
- **NEW** ESLint override in `eslint.config.mjs`: promote `sonarjs/cognitive-complexity` from `warn` to `error` for the two carve-out files (`Button.tsx`, `LinkButton.tsx`). Both files are tiny (53 / 35 LOC, single rendered tree, no branching beyond `isLoading ? … : children` and `pressed === undefined`) — measured complexity is ~3 on `Button`, ~1 on `LinkButton`. Override locks the ceiling.
- **NEW** per-file coverage thresholds in `vitest.config.ts` for the two new files at 90% lines / 90% statements / 90% functions, with a relaxed branches floor (declared at audit time based on what the v8 reporter counts on JSX) — same pattern `test-pure-libs` used for `use-media-query.ts`.
- **NEW** four-audit findings recorded in `tasks.md`. Carve-out is small; expected dispositions are "no finding" for duplication and complexity, "no finding" or "minor in-place fix" for testability, full assertion-substance pass on the new tests.
- **NO source refactors expected.** Both components were authored as a pair in `standardize-buttons`; their JSX is minimal and testable as-shipped.
- **NO MODIFIED capability requirements** beyond the standard `testing-foundation` bookkeeping requirement (carve-out completion at the 90% floor, complexity locked at error, colocated tests, exclusions for `types.ts` + `index.ts`). The `button-system` spec is unchanged: every behavior the tests exercise is *already* a SHALL there. The assertion audit will run an explicit "non-elevation" pass per the testing-foundation invariant-elevation requirement and record one-line rationale per non-elevation in `tasks.md`.

## Capabilities

### New Capabilities

None.

### Modified Capabilities

- `testing-foundation`: adds a single requirement recording that the button-system primitive carve-out (`Button.tsx`, `LinkButton.tsx`) has landed at the 90% per-file Primitive-component floor, that the `sonarjs/cognitive-complexity` promotion to `error` is in effect for those files, that `types.ts` and `index.ts` are added to the `coverage.exclude` patterns as type-only / re-export-only respectively, and that subsequent sub-proposals consuming `<Button>` or `<LinkButton>` inherit the assumption that those modules are tested and complexity-locked. This is the standard archive-time bookkeeping mechanism every test-* sub-proposal uses to keep `testing-foundation`'s authority over coverage state honest.

## Impact

- **New files:** two test files (one per non-excluded carve-out file). No new test helpers — every test uses RTL + jsdom (already configured), the existing `test/helpers/setup.ts` (which loads `@testing-library/jest-dom/vitest` matchers), and a small inline `vi.mock('next/link', …)` to render `<a>` directly in the LinkButton test (Next's `<Link>` would otherwise expect a Router context).
- **Modified config:** `eslint.config.mjs` gains two entries in the per-file override array. `vitest.config.ts` gains `app/ui/components/button/types.ts` and `app/ui/components/button/index.ts` in `coverage.exclude`, plus per-file thresholds for `Button.tsx` and `LinkButton.tsx`.
- **Modified source:** none expected. Any audit finding requiring source change is recorded with disposition in `tasks.md`; refactor-in-place commits become part of this change.
- **CI:** the existing four-gate workflow runs unchanged; the `test` job runtime grows by the two new files (jsdom render assertions, expected sub-second per file at this size).
- **Dependencies:** none added. RTL (`@testing-library/react`), `@testing-library/jest-dom`, jsdom — all installed by `test-foundation`.
- **Downstream unblock:** the parent `test-coverage` change's checkbox for sub-proposal 3.1 flips on archive. `test-misc-primitives` (3.8) and every capability-flow sub-proposal that asserts on button-rendered DOM benefit from the components being a known-good primitive.
- **Risk:** low. Components are thin, behavior is fully observable from the rendered DOM, no DB, no server actions, no cross-cutting refactors. Realistic miss: failing to model the `pressed={false}` vs `pressed={undefined}` distinction (the production code uses `pressed === undefined ? undefined : pressed`, which means `pressed={false}` MUST emit `aria-pressed="false"`, NOT omit the attribute). The `button-system` spec is explicit on this and the test names it as a discrete case.
- **No runtime behavior change.**
