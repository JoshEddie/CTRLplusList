## Why

The form/input layer fails the same WCAG floors the button system used to fail, in more places and with more severity. Every input on the site sits at `min-height: 37px` (below the 44px touch target floor), is bordered with `#d1d5db` (~1.6:1 contrast against white, failing the 3:1 non-text UI floor of WCAG 1.4.11), shows focus only via a 1px border-color swap with `outline: none` (failing WCAG 2.4.11 focus-visible robustness), and reports errors in `#ef4444` at 12–13px (~3.76:1, failing the 4.5:1 normal-text floor of WCAG 1.4.3). Native `<select>` elements in the items toolbar and pagination bypass the system entirely and inherit UA defaults that are smaller, differently-styled, and visually orphaned from the inputs they sit beside.

Underneath the WCAG failures sits a deeper structural problem: the form layer has fragmented into parallel partial solutions in exactly the shape the button drift took. There are **three** error display patterns (`.input-error`, `.form-error`, `.error-message`) used inconsistently, including one that hides the error text behind a tooltip icon (`FormError` via `TooltipWrapper`) — a measurable accessibility regression because errors should be persistent, screen-reader-announced, and `aria-describedby`-linked to the input, not gated behind a hover/focus interaction. There are existing `<FormInput>` / `<FormTextarea>` / `<FormLabel>` components in `Form.tsx` but most call sites bypass them (`<input className="form-input">` appears in `ItemNameInput`, `ImageUrlInput`, `StoreInput`, `PurchaseFlowContainer`, and more), so the components enforce nothing. There are at least five page-scoped input variants (`.items-search-input`, `.store-filter-search`, `.guest-name-input`, `.position-input`, `.page-size-select select`) that re-implement the input look from scratch and drift on borders, heights, and fonts. Labels are inconsistently associated with inputs: `<FormLabel>` has no `htmlFor`, several inputs rely on `aria-label` only, the required-state asterisk is sometimes a literal `*` in the label text and sometimes a CSS pseudo-element with `content: 'Required'` (untranslatable, weak AT semantics). `QuantityLimitSelect` is named "Select" but is a number input plus a checkbox. `FormSelect` is a dynamic-imported `react-select` shipped to a single call site (`ListForm` occasion picker) while four native `<select>` call sites elsewhere have no shared wrapper at all.

A WCAG-only patch would mean bumping a few token values and adding a focus ring — fast, but it leaves every structural problem above intact, and the next round of drift starts immediately. A real fix needs both: meet the WCAG floors _and_ collapse the fragmentation into a single field system that future call sites flow through, the same shape of fix `standardize-buttons` applied to its surface.

### Revision (post-implementation learning)

The first implementation attempt revealed an architectural flaw that this proposal previously enabled: shipping primitives that own their chrome via a class (`.field-control` on the `<input>` itself) while also accepting `className` passthrough left every primitive subvertable from the outside. The first page-scoped decorator that wrapped an input (`.items-search`) had to **strip the primitive's chrome from outside** to make its layout work — defeating the "one visual contract" pillar from the inside. Four visual chromes emerged within the first sweep. This revision adopts the chrome-owner pattern from `budget_eddiefamily`'s `app/components/FormField/` (read it for a working reference): the **parent wrapper owns the chrome via grid-based slots**, child inputs render with `background: transparent; border: none; height: 100%; width: 100%` and inherit chrome from the parent. Decorations live **inside** the wrapper via grid slots (`icon` + `iconPosition` props), not outside via decorator wrappers. There is no `className` passthrough to the underlying input — structurally impossible to subvert. Specialized layouts (search-with-trailing-clear, value-with-prefix-and-suffix) are **sibling primitives** with their own grid templates, not modifiers on a single primitive. This is the principle the proposal should have specified from the start: **deviation shouldn't be possible**.

## What Changes

- Introduce shared field design tokens in `global.css` (`--field-radius`, `--field-padding-y`, `--field-padding-x`, `--field-font-size`, `--field-min-height: 44px`, `--field-border-color`, `--field-border-color-hover`, `--field-border-color-focus`, `--field-border-color-error`, `--field-focus-ring-color`, `--field-error-color`, `--field-label-font-size: 14px`, `--field-placeholder-color`). No `-sm` size siblings — the 44px floor is enforced everywhere (the previous proposal's `sm` opt-out reflected an unwillingness to challenge the toolbar's dense layout; the corrected design treats the toolbar visual as the thing to fix, not the floor to compromise).
- Bump `--field-border-color` from `#d1d5db` to a value reaching ≥3:1 against the field background (WCAG 1.4.11 non-text UI). Bump `--field-error-color` from `#ef4444` to a value reaching ≥4.5:1 against the field background (WCAG 1.4.3 normal text). Bump `--field-label-font-size` to 14px from the current 12px.
- Enforce the 44×44 CSS-pixel minimum touch target on every text input, textarea, native select wrapper, and checkbox/radio control via `min-height` on the shared chrome owner. The single token applies everywhere — no opt-out.
- Replace the `outline: none + 1px border-color shift` focus treatment with a canonical `:focus-visible` ring on the chrome-owner via `box-shadow` so the focus indicator is ≥2px and reaches ≥3:1 against adjacent colors (WCAG 2.4.11 AA).
- Build **`<FormField>`** at `app/ui/components/field/FormField.tsx` as the internal chrome owner (not exported for direct caller use). `<FormField>` renders a `<div class="form_field">` with CSS-grid layout owning the visual chrome (border, background, padding, focus-within, min-height). Accepts `icon: ReactNode` and `iconPosition: 'left' | 'right'` (default `'left'`) which switch the grid template (`auto 1fr` vs `1fr auto`, or `1fr` when no icon). Accepts `label`, `description`, `error`, `required`, `disabled`. Generates `useId()` and wires `htmlFor`/`id`/`aria-describedby`/`aria-invalid`/`aria-required` automatically. Renders `<FieldError>` when `error` is present.
- Build public field-type wrappers as thin shells over `<FormField>`:
  - **`<TextField>`** — `<input class="form_field_input">` for `text | email | url | tel | password | search`. Search is acceptable here (no clear button); for search + clear, use `<SearchField>`.
  - **`<TextareaField>`** — `<textarea class="form_field_textarea">`. Multi-line; the grid switches to top-aligned-icon when this child is present.
  - **`<SelectField>`** — native `<select class="form_field_select">` with custom chevron in the trailing icon slot. Accepts `options: { value, label }[]` or `children` for `<option>` markup.
  - **`<DateField>`** — `<input type="date">` with no chrome override (UA picker stays).
  - **`<DatalistField>`** — `<input type="text" list={useId}>` plus a sibling `<datalist>` populated by an `options: ReactNode` prop. Free-text with suggestions.
  - **`<PriceField>`** — locked `$` icon (left), cents-as-integer math, formatted-string display. The pattern is copied from `budget_eddiefamily`'s `NumericField` (`type="money"` path): `inputMode="numeric"` with manual digit parsing → cents → divide-by-100 for display, avoiding the buggy mobile `inputMode="decimal"` UX. Use this where `StoreInput`'s `$`-prefixed price input lives; the field replaces the `.if-price-wrap` + `.if-dollar` + `.if-price-in` hand-rolled prefix layout.
- Build **`<SearchField>`** as a **sibling** primitive (not a `<FormField>` wrapper) at `app/ui/components/field/SearchField.tsx`. Its own grid is `auto 1fr auto` (search icon, input, clear button); it reuses the `.form_field` chrome CSS for border/background/focus-within but adds its own class for the 3-column grid. The clear button is auto-rendered when `value` is non-empty AND `onClear` is provided. Discriminated union: `{ onClear, trailing: never }` xor `{ onClear: never, trailing }` so callers can swap the trailing slot for a count badge or other affordance without conflict.
- Build **`<CheckboxField>`** as a separate primitive at `app/ui/components/field/CheckboxField.tsx`. Native `<input type="checkbox">` with `appearance: none`, custom 24×24 box, white check on primary fill when checked. Wraps in a `<label>` with `min-height: 44px` to satisfy 2.5.8 via the label-as-hit-area pattern. Standalone use only (form-context checkboxes use `<CheckboxField label="…">` directly, never inside `<FormField>` — checkboxes don't fit the field-row grid).
- Build **`<FieldError>`** at `app/ui/components/field/FieldError.tsx` as the single canonical error display: persistent (not tooltip-gated), no `role="alert"` (the `aria-describedby` linkage handles announcement on focus; `role="alert"` re-announces on every render and is hostile), color-token-driven, automatically `id`-linked to the input via `<FormField>`'s `aria-describedby` chain.
- **No `className` passthrough to the underlying input.** Field-type wrappers omit `className` from the spread input/select/textarea props (`Omit<ComponentPropsWithRef<"input">, "className" | "disabled" | "type">` etc). The wrapping `<FormField>` _does_ accept a `className` for the outer div — strictly for layout positioning by a parent (`flex: 1` etc), never for chrome.
- **All inputs sit on white surfaces in this app.** Single visual contract (no `data-surface` palette switching). If a future surface adds dark backgrounds (notably the list hero's `--hero-gradient`), this proposal explicitly does NOT widen the primitive — those cases call for a sibling primitive or stay page-scoped, decided when the case arises.
- **Delete the tooltip-as-error pattern in `FormError`** ([app/ui/components/Form/Form.tsx:79](app/ui/components/Form/Form.tsx#L79)). Error content gated behind interaction is a 3.3.1 violation. All callers migrate to `<FormField error={…}>` or any field-type wrapper's `error` prop.
- **Delete the CSS pseudo-element required indicator** (`.required::after { content: 'Required'; }` in [form.css:27](app/ui/styles/form.css#L27)). `<FormField required>` renders the required affordance as real DOM text + sets `aria-required="true"` on the underlying input.
- **Audit and normalize the required state** across every form. Literal `*` characters in label markup (`<FormLabel>Name*</FormLabel>` in `ItemNameInput`) are removed; the `required` prop is the single mechanism.
- **Migrate every input/textarea/select/checkbox call site** to the new primitives. The migration absorbs and deletes every page-scoped input class whose visual treatment is fully redundant with the standard chrome: `.items-search-input`, `.items-search` wrapper border/background, `.store-filter-search`, `.guest-name-input`, `.position-input`, `.page-size-select select`, `.items-sort` chrome, `.if-price-wrap` (replaced by `<PriceField>`), `.if-dollar`, `.if-price-in`. Each migration replaces a bare `<input className="form-input">` with the appropriate field-type wrapper.
- **Replace native `<select>` usages** in `PageSizeSelect`, `ItemsToolbar` (3 selects: sort, purchases, show), and `QuantityLimitSelect`'s structural pattern with `<SelectField>`. The opened state delegates to UA (mobile pickers, free a11y, no JS weight).
- **Replace `<input type="checkbox">` usages** in `StoreFilterPopover` (per-store checkboxes), `VisibilityPicker` (in-feed toggle), and `QuantityLimitSelect`'s unlimited toggle with `<CheckboxField>`.
- **Decide `<FormSelect>` (react-select) fate in design.md.** Default decision: delete `<FormSelect>` + `<SelectWrapper>` + `select.css`, migrate the single caller (`ListForm` occasion picker) to `<DatalistField>` with the six common occasions as suggestions. Fallback to reskinned `<FormSelect>` only if datalist UX is verified unacceptable in Safari.
- **Delete `FormGroup`, `FormLabel`, `FormInput`, `FormTextarea`, `FormError`, `FormDescription`, `Form`** in [app/ui/components/Form/Form.tsx](app/ui/components/Form/Form.tsx) after call-site migration. Replaced by the new field-type wrappers + a bare `<form>` element at call sites.
- **Audit label associations** across the entire form layer. Every input gets a real `<label htmlFor>` via `<FormField>`'s rendered label; `aria-label`-only inputs persist only where there is genuinely no visual label (the items-toolbar search input's visible position alongside the "Items" header is the canonical case).
- **Migrate the items-toolbar search input** to `<SearchField onClear={…}>`. The bespoke `.items-search` wrapper and its chrome-stripping CSS overrides are deleted entirely — `<SearchField>` owns the icon-input-clear layout natively.
- **Migrate `StoreInput`'s store/price/link row inputs** — store name and link become `<TextField aria-label>`, price becomes `<PriceField aria-label>` (icon-locked `$`, cents-as-integer). The row's `.if-store-row` grid layout stays, the `.if-price-wrap` prefix-layout dies.
- **Migrate `ImageUrlInput`** — URL field becomes `<TextField type="url" trailing={thumb}>` if a trailing-thumb slot proves needed, or `<TextField type="url">` with the thumbnail rendered below as a sibling preview block. Design.md picks; the criterion is "does the thumb belong inside the input row or below it?" — the case is decided during migration, not pre-committed.
- **Centralize field icons** in `app/ui/components/field/field-icons.tsx` exporting a `FIELD_ICONS` constant (`name`, `date`, `link`, `email`, `search`, etc.) — same pattern as `budget_eddiefamily`'s `field-icons.tsx`. Callers prefer `icon={FIELD_ICONS.name}` over re-importing the icon component at every call site, so the icon for a kind can be swapped in one place. Optional — the `icon` prop on field wrappers still accepts any `ReactNode`.
- Out of scope — **deferred to follow-up changes**:
  - **`<RadioGroup>` / standalone `<Radio>` primitives** — no current call site uses radio inputs; defer until needed.
  - **`<ReorderField>` / number-stepper primitive** — `ReorderInputGroup`'s two-button-plus-input stepper is specialized to the lists page. The inner input migrates to `<TextField type="number">`, but the stepper layout stays page-scoped (single call site doesn't earn a primitive).
  - **The `<input type="file">` in `ImageSearch`** — no shared file-input primitive; the existing markup gets a minimal accessibility pass (label-association + focus ring) but stays page-scoped.
  - **Slider, switch, combobox primitives** — no call sites; defer.
  - **Dark/on-image surfaces** — the list hero's translucent-white-on-purple `<CheckboxField>` (currently styled via page-scoped overrides on the `<Checkbox>` primitive in the prior implementation) stays as a page-scoped override of the standard primitive's color tokens. If a second on-dark surface emerges, that's the moment to introduce a sibling primitive or a `data-surface` mechanism — not before.
- Out of scope — **documented exemptions, not deferred**:
  - The `<input>` inside `ImageSearch` (search-in-modal) becomes `<SearchField>`, but the result-grid radio-style image cards stay as-is.
  - `<input type="hidden">` everywhere — not user-facing.

## Capabilities

### New Capabilities

- `form-field-system`: Defines the visual and accessibility contract for all interactive form-field elements in the app. The chrome lives on a single `<FormField>` wrapper that owns the visual grid via `icon` + `iconPosition` slots; field-type wrappers (`<TextField>`, `<TextareaField>`, `<SelectField>`, `<DateField>`, `<DatalistField>`, `<PriceField>`) render transparent child elements that inherit chrome from the wrapper. Specialized layouts (`<SearchField>`) are sibling primitives that reuse the chrome CSS but provide their own grid template. `<CheckboxField>` is a separate primitive (checkbox doesn't fit the field-row grid). `<FieldError>` is the single canonical error display, automatically `id`-linked via `<FormField>`'s `aria-describedby` chain. `className` is NOT forwarded to underlying inputs — only to the outer wrapper for layout positioning. There is no size variant — the 44px floor applies universally. The capability covers minimum touch targets, focus indication, contrast floors for borders / placeholders / error / label text, label/input association, required/invalid ARIA semantics, the centralized `FIELD_ICONS` registry, and the rules governing when a new field kind is added vs. when a sibling primitive is justified.

The `<input type="file">` in `ImageSearch` — same logic; file-input primitive deferred. The `react-icons` size mixing inside inputs — flagged but not fixed; icon-sizing belongs to a future icon-system change.
- **Risk**: The 44px floor will make some currently-compact inputs taller. The items-toolbar filter sheet, store-filter popover's per-store checkbox list, and reorder-input stepper are the candidates. Without the `sm` opt-out from the prior proposal, these need a layout response — likely additional horizontal space in the toolbar (rearranged grid) or accepting taller stacks in the filter sheet (which already scrolls). Mitigation: visual review during the sweep; if a critical layout breaks, raise it to the user before introducing an `sm` size that re-enables drift.
- **Risk**: The border-color bump changes the look of every form on the site at once. This is a deliberate visual change (the current border is barely visible by design accident, not intent). Mitigation: screenshot before/after in the PR description, call out the WCAG 1.4.11 driver, link to the contrast ratio measurement.
- **Risk**: Deleting `Form.tsx`'s sub-components AND the prior implementation's `field/` directory is a big diff. Mitigation: structure the migration in commits by call-site family (item-form components; list-form; toolbar; popovers; purchase modal), run the build between each so type errors surface incrementally.
- **Risk**: Native `<SelectField>` styling is more constrained than custom JS selects. Mitigation: the WCAG and visual goals are about the _closed_ state matching the input chrome; the _open_ state is delegated to UA. If a future design demands a styled-open menu, that's the moment to revisit.
- **Risk**: The `FormSelect`/react-select decision affects the occasion picker's UX. Default deletion path means losing react-select's search-as-you-type behavior. Mitigation: `<DatalistField>` provides typed-suggestion behavior natively; verify Safari renders the datalist usefully before merge. Fallback is reskinned `<FormSelect>` consuming the new tokens.
- **Risk**: The chrome-owner pattern requires the child input class names (`form_field_input`, `form_field_select`, `form_field_textarea`) to be applied by the wrappers and never spread into via `className`. Mitigation: enforced by omitting `className` from the wrappers' input-prop types — TypeScript rejects any attempt to pass it. The only `className` accepted is on the outer wrapper for layout positioning.
