## Why

Sub-proposal 3.7 of the `test-coverage` initiative. With `test-button-system` (3.1), `test-chip-system` (3.2), `test-form-field-system` (3.3), `test-menu-system` (3.4), `test-popover-trigger-system` (3.5), and `test-segmented-control-system` (3.6) archived, the loading-indicator family is the next primitive carve-out. The carve-out is `app/ui/components/LoadingIndicator.tsx` (one executable component file at 20 LOC — the smallest primitive carve-out in the initiative so far) and `app/ui/components/loading-indicator.css` (pure CSS — not part of the JS coverage report). The parent `test-coverage` change's tasks.md §3.7 line is explicit: "Elevate to `loading-indicator-system` spec." That spec already exists at `openspec/specs/loading-indicator-system/spec.md` (created by archiving `unify-loading-indicators` and extended by subsequent route-segment changes); this sub-proposal therefore does NOT create a new capability spec — it locks the existing SHALLs against a colocated test file, elevates the small set of non-trivial *call-time* invariants the new tests assert that are not yet in the spec (notably the exact class-string composition that the CSS selectors depend on), and adds the carve-out bookkeeping under `testing-foundation` per the parent's D13 two-tier rollup.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active delta in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation`, `test-foundation-spike`, `test-pure-libs`, `test-button-system`, `test-housekeeping`, `test-chip-system`, `test-form-field-system`, `test-menu-system`, `test-popover-trigger-system`, `test-segmented-control-system`) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Every requirement applies verbatim.
- `loading-indicator-system` (active spec) — 16 active requirements covering: the single shared `<LoadingIndicator>` primitive as the only `<Suspense>` fallback and the only rendered output of route-segment `loading.tsx` files; the fixed enum of size variants (`inline | rail | form | page`) with declared min-heights (`2rem / 200px / 400px / 60vh`) and spinner diameters (`16px` for `inline`, `32px` otherwise); fallback placement inside the same layout container the suspended content will occupy; pure-CSS spinner consuming `var(--primary-color)`; accessibility shape (`role="status"`, `aria-live="polite"`, visually-hidden "Loading…" label, `aria-hidden="true"` on the spinner); the `(main)` route group's no-layout-Suspense rule; the `MainShell.tsx` non-existence; per-page `<main className="container [variant?]">` placement; per-route Suspense placement contracts for `/items/[id]`, `/lists/[id]/edit`, `/lists/[id]/choose-items`, `/user/[id]`, `/settings/connections`, and `/lists/[id]`; the no-duplicate-chrome rule on fallbacks. The tests assert against the primitive-shape SHALLs as the authoritative contract (every other requirement is a placement contract owned by consuming routes, out of scope for a primitive-family carve-out). **No requirements are removed.** Two NEW invariants surfaced by reading source at HEAD are elevated alongside the existing requirements — see "Capabilities" below.
- `button-system`, `form-field-system`, `menu-system`, `popover-trigger-system`, `chip-system`, `segmented-control-system` — unrelated; `LoadingIndicator` does not render through any primitive family. The component is the only primitive in the app that renders no `<Button>`, no `<input>`, no menu, no popover, no segmented control — a leaf primitive whose only DOM output is two `<span>` elements inside a `<div>`. No transitive SHALL applies.
- Capability-flow consumers (`home-digest`, `list-collections`, `list-hero-collapse`, `list-hero-header`, `list-item-management`, `items-browser-chrome`, `following`, etc.) consume `<LoadingIndicator>` at call sites (e.g. `<Suspense fallback={<LoadingIndicator size="rail" />}>` on the home rails, `size="form"` on the edit-list form, `size="page"` on the items grid). Those call-site placements are owned by the existing route-Suspense SHALLs in the `loading-indicator-system` spec (lines 156–256 of the active spec) — NOT by this carve-out. The carve-out exercises the PRIMITIVE, not the placement contracts.
- Cross-cutting design-system rule applies: this proposal is a spec-modification, NOT a page-scoped one-off. Loading indicators are an interactive surface (well, a non-interactive surface — but the cross-cutting rule covers every shared design-system primitive) governed by a primitive-family spec; the spec edits here EXTEND the existing `loading-indicator-system` spec with elevated call-time SHALLs surfaced during testing. No new size variant, no new sibling primitive. Cache-tag rule does NOT apply (no server reads).

## What Changes

- **NEW** colocated test file (one `.test.tsx` for the executable JSX component under jsdom):

  - `app/ui/components/__tests__/LoadingIndicator.test.tsx` — renders a `<div role="status" aria-live="polite">` with class composed as `'loading-indicator loading-indicator--<size>'` (exact-string, fixed token order); each of the four `size` variants (`'inline'`, `'rail'`, `'form'`, `'page'`) produces its corresponding variant class (`loading-indicator--inline`, `loading-indicator--rail`, `loading-indicator--form`, `loading-indicator--page`) — asserted via exact-string class match for all four; the outer `<div>` is a `<div>` (not a `<section>`, not a `<span>`), confirmed by `element.tagName === 'DIV'`; the outer `<div>` carries `role="status"` and `aria-live="polite"` exactly; the outer `<div>` contains exactly two child `<span>` elements; the first child is a `<span class="loading-indicator__spinner" aria-hidden="true">` with no text content; the second child is a `<span class="sr-only">Loading…</span>` (exact-string text match, including the U+2026 horizontal ellipsis character — NOT three ASCII dots); the visually-hidden label is announced via `role="status"` + `aria-live="polite"` on the parent (asserted by `getByRole('status')` reading the accessible name and returning `"Loading…"`); the spinner span is hidden from the accessibility tree via `aria-hidden="true"` (asserted by `getByRole('status').children[0].getAttribute('aria-hidden') === 'true'`).

- **NO** test for `loading-indicator.css` — CSS files are not part of the JS coverage report. The min-heights (`inline: 2rem`, `rail: 200px`, `form: 400px`, `page: 60vh`) and spinner diameters (`inline: 16px`, others: 32px) declared in the active spec are CSS-only behaviors verified by the class names being applied to the right DOM element. jsdom does not fully resolve CSS variables or `min-height` declarations from a stylesheet, so a `getComputedStyle` assertion would be tautological-against-jsdom rather than load-bearing. The existing spec scenarios (e.g. "Size variants resolve to declared min-heights") remain spec-only; the carve-out's tests lock the CLASS APPLICATION that the CSS selectors target, which is the testable component-level contract.
- **NO** test for the `import './loading-indicator.css';` side-effect import — the import has no runtime observable behavior in jsdom (CSS does not apply). The `tsc --noEmit` and `next build` gates verify the import resolves; the carve-out's coverage focuses on the JSX render path.
- **NEW** ESLint override in `eslint.config.mjs`: extend the existing per-file `sonarjs/cognitive-complexity = error` override array to include `app/ui/components/LoadingIndicator.tsx`. Measured complexity at HEAD: 1 (a single `return` statement with one inline template-literal class composition, no conditionals, no loops, no early returns). The override locks the ceiling at 15 for future edits.
- **NEW** per-file threshold in `vitest.config.ts` for the one executable file, referencing the existing `COVERAGE_FLOOR` constant (one entry — NO per-file numeric variation, per `test-housekeeping`'s single-constant rule).
- **MODIFIED** `loading-indicator-system` spec — ADDS two new requirements that elevate call-time invariants the new tests lock against and that are currently implicit in the source:
  - The `<LoadingIndicator>` primitive SHALL compose its outer `<div>` class as the exact string `'loading-indicator loading-indicator--<size>'` in fixed token order — `'loading-indicator'` always first, `'loading-indicator--<size>'` second, no third token (the primitive accepts no `className` prop). This locks the CSS selector contract: `loading-indicator.css` uses BOTH the base `.loading-indicator` selector (for the flex layout shared across all sizes) AND the variant-specific `.loading-indicator--<size>` selector (for the min-height per size and the spinner-diameter-by-variant compound selectors like `.loading-indicator--inline .loading-indicator__spinner`). A rewrite that drops the base class, reorders the tokens, or merges them into a single hyphen-joined class (e.g. `loading-indicator-inline`) would silently break the shared-layout CSS or the variant-scoped spinner-diameter CSS.
  - The primitive's DOM SHALL be exactly: an outer `<div>` containing exactly two child `<span>` elements in fixed order — first a `<span class="loading-indicator__spinner" aria-hidden="true">` (no text content), then a `<span class="sr-only">Loading…</span>` (with the exact U+2026 horizontal ellipsis character, NOT three ASCII dots). The element types and the child order are load-bearing: the CSS uses descendant selectors like `.loading-indicator__spinner` and relies on the spinner being a self-closing visual; the screen-reader announcement relies on the sr-only label being the only readable text inside the `role="status"` region.
  These elevations are documented in `design.md` Decision 3.
- **NEW** `testing-foundation` carve-out bookkeeping record (sub-proposal-archive ONLY, per `test-coverage` design D13 two-tier rollup): the loading-indicator-system primitive carve-out has landed at the universal `COVERAGE_FLOOR`, the `sonarjs/cognitive-complexity` promotion to `error` is in effect for the one file, and subsequent sub-proposals consuming `<LoadingIndicator>` inherit the assumption that the module is tested and complexity-locked. Recorded in `openspec/changes/test-loading-indicator-system/specs/testing-foundation/spec.md` ONLY (and at archive in `openspec/changes/archive/<date>-test-loading-indicator-system/specs/testing-foundation/spec.md`) — does NOT roll into the parent's `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md` (which is created at test-coverage's archive time from the parent accumulator).
- **NEW** four-audit findings recorded in `tasks.md`. Carve-out is the smallest in the initiative so far: one file, 20 LOC, complexity 1, no conditionals, no event handlers, no context, no refs, no useEffect, no useState. Anticipated dispositions: assertion-substance pass on the new tests (every assertion names a specific DOM attribute, element type, exact class string, or exact text content); duplication audit expected to find no extractable patterns (single-file test); complexity audit expected to pass trivially (complexity 1, well below ceiling 15); testability audit expected to be straightforward (single render-and-assert path per variant; no document-level state, no async, no refs).
- **NO source refactors expected.** Every observable branch of the single render path is reachable by varying the `size` prop across its four enum values. There are no early returns, no conditionals, no parameter defaults. The component's behavior is fully determined by its props.

## Impact

- **New files:** one `.test.tsx` for `LoadingIndicator`. Lives at `app/ui/components/__tests__/LoadingIndicator.test.tsx`. No `test-helpers.tsx` anticipated — single-file carve-out with no inter-test duplication to extract.
- **Modified config:** `eslint.config.mjs` gains one path in the per-file override array. `vitest.config.ts` gains one entry in the `thresholds` map, pointing at `COVERAGE_FLOOR`.
- **Modified source:** none expected. The component is a pure render with no conditionals, so every branch v8 measures is reachable by the four-variant sweep. Any audit finding requiring source change is recorded with disposition in `tasks.md`; refactor-in-place commits become part of this change.
- **Modified specs:** `openspec/specs/loading-indicator-system/spec.md` gains two ADDED requirements (exact class-string composition; exact DOM child structure). No removals. The carve-out bookkeeping requirement lives ONLY in this sub-proposal's archive directory (`openspec/changes/archive/<date>-test-loading-indicator-system/specs/testing-foundation/spec.md`); it does NOT roll into the parent `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md` (per `test-coverage` design D13 two-tier rollup).
- **CI:** the existing four-gate workflow runs unchanged; the `test` job runtime grows by one new file (jsdom; expected sub-second — the test file is the smallest in the initiative because there are no event dispatches, no spies, no rerenders).
- **Dependencies:** none added.
- **Downstream unblock:** the parent `test-coverage` change's checkbox for sub-proposal 3.7 flips on archive. Capability-flow sub-proposals that render `<LoadingIndicator>` at their call sites (most §4 sub-proposals — 4.1 `test-app-frame`, 4.3 `test-home-digest`, 4.6 `test-list-collections`, 4.7 `test-list-hero-header`, 4.8 `test-list-hero-collapse`, 4.9 `test-list-item-management`, 4.10 `test-list-metadata`, 4.11 `test-list-visibility`, 4.5 `test-items-browser-chrome`, 4.15 `test-user-actions`) benefit from `<LoadingIndicator>` being a known-good module with its primitive contract elevated in the spec. The placement contracts in the existing `loading-indicator-system` spec (e.g. "`/user/[id]` SHALL render two independent section Suspenses") are exercised by the capability-flow sub-proposals that own those routes.
- **Risk:** very low. One small file, no executable branches beyond the props sweep, no async, no events, no spies. Realistic miss: the `"Loading…"` text assertion using three ASCII dots instead of the U+2026 horizontal ellipsis character — a contributor copy-pasting from a non-Unicode-aware editor could write `expect(...).toHaveTextContent('Loading...')` (three dots) which would silently fail to match the source's single-character `…` — the test naming and Decision 3 explicitly call out the ellipsis encoding. Secondary risk: writing an `aria-hidden` assertion that uses the boolean attribute form (`hasAttribute('aria-hidden')`) instead of the string-value form (`getAttribute('aria-hidden') === 'true'`); the React serialization produces the literal string `"true"`, not the bare attribute, and a `hasAttribute`-only check would pass against the wrong serialization (`aria-hidden=""` or `aria-hidden="false"`). Tertiary risk: the four-variant class-string sweep is easy to author as a parameterized `it.each` loop, which `testing-foundation`'s `<State>_<Behavior>` `it()` shape rule discourages in favor of four discrete `it()` blocks (one per variant) so a single-variant regression produces a single-named failing test instead of an opaque "parameterized test failed at index 2" output.
- **No runtime behavior change.**
