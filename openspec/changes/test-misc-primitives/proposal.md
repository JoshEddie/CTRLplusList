## Why

Sub-proposal 3.8 of the `test-coverage` initiative. With 3.1–3.7 archived (button, chip, form-field, menu, popover-trigger, segmented-control, loading-indicator), the four genuinely-ungoverned primitives at the top of `app/ui/components/` are the remaining primitive carve-out: `ConfirmDialog.tsx`, `TooltipWrapper.tsx`, `Empty.tsx`, `FormShell.tsx`. Unlike the prior primitive families, these four do not live in per-family sub-folders and do not have governing family-system specs today — the parent's tasks.md §3.8 is explicit: "Creates minimal family specs for each as part of this proposal." This sub-proposal therefore creates four NEW capability specs (`confirm-dialog-system`, `tooltip-system`, `empty-state-system`, `form-shell-system`), each minimal — only the call-time invariants the colocated tests lock against — and adds colocated tests at the universal `COVERAGE_FLOOR`. The parent's strict-cap rule is honored: a fifth genuinely-ungoverned primitive (e.g. `AuthPage.tsx`, `AppLogo.tsx`, `Logo.tsx`) gets its own sub-proposal added to the parent's tasks.md §3.x, not bundled here.

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active accumulator at `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas from `test-foundation*`, `test-pure-libs`, `test-housekeeping`, and the six archived primitive carve-outs) — runner (vitest 4.x), four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule, `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, four-audit obligation, invariant-elevation audit, assertion-substance bar, `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Every requirement applies verbatim.
- `button-system` (active) — `ConfirmDialog`, `Empty`, and `FormShellFooter` all render through `<Button>` / `<LinkButton>`; the focus, hover-on-touch, touch-target, and variant-as-visual contracts flow through verbatim. No transitive SHALL is re-asserted by these tests — the button-system tests already lock those contracts. The new family specs reference the `button-system` token surface but do not duplicate its requirements.
- `menu-system` (active) — `ConfirmDialog` is invoked from `ListActionsMenu` (an active call site) but the dialog itself is not a menu; no menu SHALL flows through the dialog primitive.
- `form-field-system` (active) — explicitly removed `TooltipWrapper` as a vehicle for field-error text (`spec.md:333`: "The system SHALL NOT use a tooltip, popover, or any hover/focus-gated UI to convey field error text"). The new `tooltip-system` spec MUST honor this exclusion — the family scope is "decorative / informational hover hints," not field validation. `<FormField>` does NOT render through `<FormShell>` and vice versa; `FormShell` is the modal-form chrome, not the field primitive.
- No existing `confirm-dialog-system`, `tooltip-system`, `empty-state-system`, or `form-shell-system` capability spec exists today (verified via `ls openspec/specs/`). These are NEW capabilities.
- Cross-cutting design-system rule applies: each of the four primitives is an interactive surface (modal, hover hint, empty-state CTA, modal form chrome) governed by a primitive-family spec. The four new specs created here cover that governance. No page-scoped one-off classes are introduced.
- Cache-tag rule does NOT apply — none of the four primitives reads server data.

## What Changes

- **NEW** four minimal capability specs (one per primitive):
  - `confirm-dialog-system` — locks `<ConfirmDialog>`'s isOpen short-circuit, overlay + content DOM shape, three-button row (optional tertiary → Cancel → Confirm), each button's onClose composition (tertiary and Confirm BOTH close after acting; Cancel closes), and the `tertiary.variant` default of `'primary'`.
  - `tooltip-system` — locks `<TooltipWrapper>`'s wrapper-class composition (`'tooltip-container'` alone when `className` is falsy; `` `tooltip-container ${className}` `` with a single joining space when truthy — no trailing whitespace in either branch), the `showTooltip` default-true + conditional tooltip span, and the family's exclusion from error-text duty (cross-link to `form-field-system`).
  - `empty-state-system` — locks `<Empty>`'s capitalization of `type`, the purchase-vs-other title/description branching, the purchase-vs-other CTA branching (purchase → no CTA; non-purchase with `setShowNewItem` → `<Button>`; non-purchase without → `<LinkButton href={`/${type}s/new`}>`).
  - `form-shell-system` — locks `<FormShell>`'s overlay-click-dismiss-on-self-target-only contract, the three variant class compositions (`'form-shell'` / `'form-shell form-shell-wide'` / `'form-shell form-shell-split'`), the close-button aria-label + dismiss invocation, and `useDismiss`'s three-branch resolution (onClose wins → `router.back()` when `window.history.length > 1` → `router.push(closeHref)` fallback). Same SHALLs cover `<FormShellFooter>`'s Cancel + Submit row, the `isPending`-via-`isLoading` passthrough, and the `deleteSlot` insertion point.
- **NEW** colocated test files under `app/ui/components/__tests__/` (one `.test.tsx` per executable file):
  - `app/ui/components/__tests__/ConfirmDialog.test.tsx`
  - `app/ui/components/__tests__/TooltipWrapper.test.tsx`
  - `app/ui/components/__tests__/Empty.test.tsx`
  - `app/ui/components/__tests__/FormShell.test.tsx` (covers both `FormShell` and `FormShellFooter` plus the internal `useDismiss`)
- **NEW** per-file thresholds in `vitest.config.ts` for the four files, each referencing `COVERAGE_FLOOR` (per `test-housekeeping`'s single-constant rule).
- **NEW** ESLint override in `eslint.config.mjs`: extend the per-file `sonarjs/cognitive-complexity = error` override array to include all four files, with the carve-out comment header `// test-misc-primitives (sub-proposal 3.8) — locked at universal COVERAGE_FLOOR.`. Anticipated complexity at HEAD: `FormShell.tsx`'s `useDismiss` has the deepest branching (three conditions: `onClose` short-circuit, `window.history.length > 1` guard, `closeHref` fallback) — expected ≤ 8. `ConfirmDialog.tsx` is a single render with one conditional (`isOpen` short-circuit, optional tertiary slot) — expected ≤ 4. `Empty.tsx` has two conditional branches (purchase-vs-other for text; setShowNewItem-or-LinkButton for CTA) — expected ≤ 5. `TooltipWrapper.tsx` is a single render with one conditional (`showTooltip`) — expected ≤ 2. All four well below the ceiling of 15.
- **NO** test for `ServiceWorkerRegistration.tsx`, `AuthPage.tsx`, `AppLogo.tsx`, `Logo.tsx`, `Header.tsx`, `Nav.tsx`, `AppNav.tsx`, `AppMenu.tsx`, `AppFrame.tsx`, `ListCard.tsx`, `ListCardRow.tsx`, `MoreCard.tsx`, `ListCollectionsNav.tsx`, `LoadingIndicator.tsx` — these belong to capability-flow sub-proposals (4.1 app-frame, 4.6 list-collections, 4.12 pwa-shell) or the loading-indicator carve-out (3.7), per the parent's mapping. The strict-cap rule prevents bundling them here.
- **NO** source refactors anticipated. `Empty.tsx`'s `purchase`-branch and `LinkButton` fallback are reachable at HEAD only from no current call site (the only consumer is `ItemsPage.tsx` with `type="item"` + `setShowNewItem`), but both branches are reachable from the component's prop surface — tested via direct render with the appropriate props. No dead-code removal is in scope; if the four-audit surfaces a finding requiring source change, it's recorded with disposition in `tasks.md`.
- **NEW** `testing-foundation` carve-out bookkeeping record (sub-proposal-archive ONLY, per `test-coverage` design D13 two-tier rollup): the four primitives are tested at the universal `COVERAGE_FLOOR` and complexity-locked at error. Recorded in `openspec/changes/test-misc-primitives/specs/testing-foundation/spec.md` ONLY (and at archive in `openspec/changes/archive/<date>-test-misc-primitives/specs/testing-foundation/spec.md`). Does NOT roll into the parent's `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md`.

## Capabilities

### New Capabilities

- `confirm-dialog-system`: governs `<ConfirmDialog>` — modal confirmation primitive with overlay, title, message, optional tertiary action, Cancel (ghost), and Confirm (danger) buttons.
- `tooltip-system`: governs `<TooltipWrapper>` — decorative hover-hint primitive. Explicitly excludes form-field error text duty (delegated to `form-field-system`'s inline `<FieldError>`).
- `empty-state-system`: governs `<Empty>` — empty-collection state with capitalized title, type-aware description, and optional inline-create CTA OR link-to-create fallback.
- `form-shell-system`: governs `<FormShell>` and `<FormShellFooter>` — modal-form chrome with overlay-dismiss, variant-aware width, header-close button, and Cancel/Submit footer. Includes the `useDismiss` internal helper's three-branch resolution.

### Modified Capabilities

- `testing-foundation`: a carve-out bookkeeping record is written to this sub-proposal's archive-only delta (Tier 2 per `test-coverage` D13). No change to the active `testing-foundation` spec; no change to the parent `test-coverage` accumulator.

## Impact

- **New files:** four `.test.tsx` files under `app/ui/components/__tests__/`. If duplication crosses two or more files during the §5 audit, a `__tests__/test-helpers.tsx` joins the directory (excluded from coverage via the existing `**/__tests__/**` glob). Precedent across prior carve-outs: helpers stay inline unless reused across files.
- **New specs:** four new active specs under `openspec/specs/<name>/spec.md`, created by archiving this sub-proposal: `confirm-dialog-system`, `tooltip-system`, `empty-state-system`, `form-shell-system`. Each has a Purpose paragraph (NEW specs SHOULD ship with Purpose written, not "TBD" — the `test-menu-system` / `test-segmented-control-system` "TBD" precedent only applies to specs that were ALREADY "TBD" before the carve-out started; specs created here are net-new and ship with full Purpose).
- **Modified config:** `eslint.config.mjs` gains four paths in the per-file override array. `vitest.config.ts` gains four entries in the `thresholds` map, each pointing at `COVERAGE_FLOOR`.
- **Modified source:** none expected. Any audit finding requiring source change is recorded with disposition in `tasks.md`; refactor-in-place commits become part of this change.
- **CI:** the existing four-gate workflow runs unchanged; the `test` job runtime grows by the four new files (all jsdom; expected sub-second each — `FormShell.test.tsx` is the heaviest because of the `useRouter` mock + `window.history` manipulation, but still small).
- **Dependencies:** none added.
- **Downstream unblock:** the parent `test-coverage` change's checkbox for sub-proposal 3.8 flips on archive. Capability-flow sub-proposals that render these primitives at their call sites — `ConfirmDialog` from 4.6 `test-list-collections` (`DeleteListButton`, `ListActionsMenu`) and 4.9 `test-list-item-management` (`DeleteItemButton`); `TooltipWrapper` from 4.9 (`ModalButtons`); `Empty` from 4.5 `test-items-browser-chrome` (`ItemsPage`); `FormShell` from 4.6 (`ListForm`) and 4.9 (`ItemForm`) — benefit from these modules being known-good with their contracts locked in active specs.
- **Risk:** low–medium. Four small files (largest is `FormShell.tsx` at 107 LOC), all testable in jsdom. Realistic miss: failing to author the `useDismiss` three-branch coverage — specifically the `window.history.length > 1` true-branch (`router.back()` fires) AND the false-branch fallthrough to `router.push(closeHref)`, which require independent jsdom `window.history` manipulation across two tests. Secondary risk: the `<ConfirmDialog>` tertiary-action close-composition (tertiary's `onClick` AND `onClose` BOTH fire on tertiary click) is easy to assert as either-side-only; the test must spy both and assert both. Tertiary risk: the `<Empty>` `LinkButton`-fallback path (non-purchase + no `setShowNewItem`) is not exercised at any call site at HEAD; the test must render this path explicitly to lock the `href={`/${type}s/new`}` composition. Quaternary risk: the `<TooltipWrapper>` wrapper-class composition originally used a template literal that emitted a trailing space when no `className` was passed (`` `tooltip-container ${className || ''}` `` → `'tooltip-container '`); this carve-out trims the wart by switching to a conditional `className ? `tooltip-container ${className}` : 'tooltip-container'`, and the tests lock both branches' exact-string output so any future regression-to-wart fails loudly.
- **No runtime behavior change.**
