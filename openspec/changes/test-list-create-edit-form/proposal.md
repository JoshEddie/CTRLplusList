## Why

Sub-proposal **9.2** of the `test-coverage` initiative ([issue #112](https://github.com/JoshEddie/CTRLplusList/issues/112)) — a coverage-gap follow-up discovered at the §7.1 close-out audit (`openspec/changes/test-coverage/tasks.md` §9.2).

Sub-proposals **4.10** (`test-list-metadata`) and **4.11** (`test-list-visibility`) covered the list create/edit/metadata/visibility **actions + DAL** — `createList` / `updateList` / `deleteList` / `setListVisibility` and the `lists` reads — but the **form UI** that drives those actions was never carved out and sits at **0% unit coverage** today. Every file below is absent from `vitest.config.ts` `thresholds` (verified against the live map) and has no colocated test. These are the client surfaces a list owner actually touches to create, rename, re-date, re-occasion, re-subtitle, re-share, and delete a list, plus the server-component page shells that mount them and the hero shell that gates a list behind its visibility.

Two of these files carry **privacy-relevant** logic that is currently asserted only indirectly: `ListHeroSection.tsx` inlines the `list.visibility === OWNER && !isOwner → <ListPrivate>` interstitial guard **and** the `after()` visit-recording upsert — and 4.14 (`test-visit-history`) tested only a hand-written **mirror** of that upsert (`app/actions/__tests__/visitHistory.actions.test.ts:52` — *"Mirrors the visit-recording upsert inlined in ListHeroSection's `after()`"*), never the real inlined block. A regression in the real guard (recording an owner's own visit, recording an unauthenticated viewer, or rendering a Hidden list to a non-owner) would pass 4.14's mirror today.

Inherited constraints (surfaced by spec-grep — every active `spec.md` was grepped for the component names, file paths, DOM strings, action names, and behaviors these files touch):

- **`testing-foundation`** (active accumulator at `openspec/changes/test-coverage/specs/testing-foundation/spec.md`, plus the archived deltas of every prior sub-proposal) — governs verbatim: the vitest 4.x runner with the jsdom/node two-project split (`.test.tsx` → jsdom, `.test.ts` → node), four-gate pre-merge, `__tests__/` colocation, the universal per-file `COVERAGE_FLOOR` (`lines:98 / statements:98 / branches:95 / functions:100`) referenced from the single constant in `vitest.config.ts`, the no-backdoor disposition rule (`/* v8 ignore */` with a named rationale, never lower the floor), the assertion-substance bar, the `<State>_<Behavior>` `it()` shape and three-role `describe()` convention, the four-audit + invariant-elevation obligations, and the `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy. Two load-bearing mocking allowances apply: **(1) NextAuth is the network boundary** — `@/lib/auth`'s `auth()` is mocked, never the modules it wraps; **(2) `next/navigation`'s `redirect()` is mocked to throw a sentinel** for page-RSC guards (the `FollowingPage.test.tsx` / `ItemsContainer.test.tsx` precedent).
- **`list-metadata`** (active) — owns the list create/edit form's subtitle persistence + normalization contract. Its `ListForm`-naming requirements are binding and asserted as-is here: the **subtitle trim-to-`null`** client behavior ("the `ListForm` client … trims the raw input and maps the empty result to `null` before invoking the action") and the `maxLength={120}` mirror. The create-vs-edit **dispatch + post-success navigation** contract that `ListForm` enforces (create → `createList` → `/lists/{id}/choose-items?new=1`; edit-in-modal → `updateList` → `onSuccess` + `onClose` + `router.refresh`; edit-as-page → `updateList` → `/lists/{id}`) is **not** specced anywhere — a candidate for elevation (see `design.md`).
- **`list-visibility`** (active) — owns `VisibilityPicker` (the "three-item radio menu" requirement names the file and its `apply` early-return no-op verbatim) and the **private-list interstitial** (`ListPrivate`, "renders the private-list interstitial"). Both are asserted as-is. The picker's **optimistic-apply state machine** — set local state, fire `setListVisibility`, **roll back + `toast.error` on failure**, `toast.success` + `router.refresh` on success — is the data-integrity heart of the component and is **not** specced (the spec covers menu structure, label/icon/description, and the no-op early-return, but not rollback). Candidate for elevation.
- **`visit-history`** (active) — its requirement *"Authenticated visits to non-owned non-private lists SHALL be recorded"* already enumerates the exact who/when matrix the `ListHeroSection.after()` block implements (record iff authenticated **and** non-owner **and** `visibility !== private`; owner/unauthenticated/private excluded). This carve-out adds the **missing direct test of the real inlined block** against that existing SHALL — no new requirement; it replaces 4.14's mirror with a test of production code.
- **Primitive-family specs rendered *through*, not re-owned** — `form-shell-system` (`FormShell`/`FormShellFooter`), `form-field-system` (`TextField`/`DatalistField`/`DateField`/`FieldError`), `button-system` (`Button`/`LinkButton`), `menu-system` + `popover-trigger-system` (`Menu`/`MenuItemRadio`/`PopoverTrigger` inside `VisibilityPicker`), and `confirm-dialog-system` (`ConfirmDialog` inside `DeleteListButton`) own their primitives. All are already tested and floored; this carve-out renders through the **real** primitives (not mocked) to exercise the form/picker/delete integration, and asserts no primitive SHALL directly.

Cache-tag note: every file here drives the **mutation** side (`createList`/`updateList`/`deleteList` → `revalidateTag('lists')`; `setListVisibility` → `updateTag('lists')`) whose tag-invalidation obligation is owned and already tested by the action carve-outs (4.9 / 4.11). The server actions are **module-mocked** in these component tests (the established `vi.mock('@/app/actions/lists')` pattern), so the tests assert *which action the UI calls with which payload and how it navigates on the result*, not the tag invalidation. The server-component shells (`new/page.tsx`, `edit/EditListBody.tsx`, `ListHeroSection.tsx`) read through `auth()` + `lib/dal` reads and are tested via the async-RSC pattern (mock `auth()`, mock the DAL reads, sentinel `redirect()`); `ListHeroSection`'s deferred `after()` upsert is exercised by capturing the `next/server` `after` callback and invoking it against a seeded read-write boundary (see `design.md`).

## What Changes

- **NEW** colocated `.test.tsx` test files under `__tests__/` directories mirroring the source layout, one per executable source file. All are client/JSX or JSX-returning async RSC files, so all run under the **jsdom** project. The in-carve-out files (none currently enumerated in `vitest.config.ts`, all 0% today):

  - **Create/edit form (4 files)** under `app/(main)/lists/ui/components/`: `ListForm.tsx` (the 195-line form: client date-validation guard, subtitle trim-to-`null`, occasion datalist state, `useActionState` dispatch to `createList`/`updateList`, the create-vs-edit submit + navigation branch matrix, the conditional `DeleteListButton` footer slot when editing, `closeHref` derivation, pending-disabled fields); `ListFormContainer.tsx` (the modal-mode passthrough wrapper); `NewListButton.tsx` (the variant/label trigger that mounts `ListFormContainer` on open); `DeleteListButton.tsx` (the `ConfirmDialog`-gated delete affordance: `deleteList` dispatch, success → toast + `router.push('/lists')`, failure → `toast.error`).
  - **Visibility picker (1 file)**: `VisibilityPicker.tsx` (the popover three-row radio + the optimistic-apply / rollback-on-failure / toast / `router.refresh` state machine; `visibility-rows.tsx`, its row table, is already floored under 4.8 and rendered through, not re-owned).
  - **Interstitial + empty states (2 files)**: `ListPrivate.tsx` (the private-list interstitial, including the `!loggedIn` "please login" branch); `EmptyList.tsx` (the "No lists found" empty state + create CTA).
  - **`new` page shell (2 files)** under `app/(main)/lists/new/`: `page.tsx` (async RSC: `auth()` + `getUserIdByEmail` guards → `redirect('/')`, else `<ListForm />`); `loading.tsx` (the Suspense fallback shell).
  - **`edit` page shell (2 files)** under `app/(main)/lists/[id]/edit/`: `page.tsx` (the `Suspense` route shell); `EditListBody.tsx` (async RSC: `auth()` + `getUserIdByEmail` guards → `redirect('/')`, `getList(id)`, `<ListForm list isEditing />`).
  - **Hero shell (1–2 files)** under `app/(main)/lists/[id]/`: `ListHeroSection.tsx` (async RSC: `auth()`, `getUserIdByEmail`, `guardListViewable(getList(id), …)`, the `OWNER && !isOwner → <ListPrivate>` interstitial guard, the inlined `after()` visit-recording upsert with its authed-non-owner-non-private guard, `getUserById`, `<ListDetails …>` projection). - **NEW** ESLint per-file `sonarjs/cognitive-complexity = error` overrides in `eslint.config.mjs` for the same files, promoting them from the global `warn`, under the matching comment. `ListForm.tsx` and `ListHeroSection.tsx` are the complexity candidates; any function measuring ≥ 15 at HEAD is disposed in-carve-out by a single-file, behavior-preserving extraction (covered by the new tests, recorded in `tasks.md`) — never by raising the ceiling.

- **NEW** global ESLint hardening in `eslint.config.mjs`: `@typescript-eslint/no-unused-vars` promoted from the Next preset's `warn` to **`error`**, with **no** `^_` ignore pattern (an exemption becomes the lazy fix for a red build — rename `x` to `_x` — which is worse than the warning because it looks enforced). ## Capabilities

### New Capabilities

None. Every behavior under test belongs to an existing capability; this carve-out LOCKS the already-specced UI requirements and ADDS two currently-unspecced UI invariants.

### Modified Capabilities

- `list-metadata`: ONE **ADDED** requirement — the list-form create-vs-edit dispatch + post-success navigation contract. Existing subtitle requirements unchanged and LOCKED by the new tests.
- `list-visibility`: ONE **ADDED** requirement — the `VisibilityPicker` optimistic-apply / rollback-on-failure contract. Existing visibility-UI requirements unchanged and LOCKED.
- `testing-foundation`: Tier-2 carve-out bookkeeping (archive-only per design D13). No change to the active `testing-foundation` spec at apply-time (archive-time rollup per §7.11); no change to the parent `test-coverage` accumulator.

## Impact

- **New files:** ~13–14 `.test.tsx` files across `app/(main)/lists/ui/components/__tests__/`, `app/(main)/lists/new/__tests__/`, `app/(main)/lists/[id]/edit/__tests__/`, and `app/(main)/lists/[id]/__tests__/`. Shared test helpers (a `ListTable` fixture builder, an `auth()`-mock / viewer-seed factory, a `next/server` `after`-capture helper) are reused from the existing `app/actions/__tests__/test-helpers.ts` / `app/(main)/lists/ui/components/__tests__/test-helpers.ts` where applicable, or extracted into the colocated `__tests__/` only if the duplication audit finds reuse across 3+ files (the established threshold).
- **Deleted files:** `app/(main)/lists/ui/components/ListHeader.tsx` (dead code).
- **Modified config:** `vitest.config.ts` gains per-file `thresholds` entries (all → `COVERAGE_FLOOR`); `eslint.config.mjs` gains the same paths in the `sonarjs/cognitive-complexity = error` override array, plus a **global `@typescript-eslint/no-unused-vars = error`** promotion with no `^_` exemption (see §6.3).
- **Modified source:** none expected beyond the `ListHeader.tsx` deletion and any conditional audit-driven, behavior-preserving extraction in `ListForm.tsx` / `ListHeroSection.tsx` if a function exceeds the ceiling of 15 at HEAD.
- **Modified specs:** `openspec/specs/list-metadata/spec.md` and `openspec/specs/list-visibility/spec.md` each gain one ADDED requirement at archive (via this change's `specs/` deltas). The Tier-2 testing-foundation record lives only in this change's archive directory.
- **Parent governance:** `test-coverage/tasks.md` §9.2 checkbox flips on archive. No sibling sub-proposal is anticipated unless the audit surfaces cross-file scope.
- **CI:** the four-gate workflow runs unchanged; the `test` job grows by ~13–14 jsdom files. The async-RSC shells mock `auth()` + the DAL reads (and, for `ListHeroSection`, capture `after`); the client components mock `@/app/actions/lists` / `next/navigation` / `react-hot-toast` at the boundary.
- **Dependencies:** none added (`@testing-library/user-event`, jsdom harness already present).
- **Risk:** low–medium. Most files are small render-and-dispatch components with established precedents (the field/form tests, the `FollowContainer` optimistic-flip precedent for `VisibilityPicker`, the async-RSC precedents for the page shells). The one genuinely harder surface is `ListHeroSection`'s deferred `after()` visit-recording, which needs the `next/server` `after`-capture pattern to assert the upsert payload — documented in `design.md`; this is also the highest-value test (it closes the mirror-vs-real gap 4.14 left).
