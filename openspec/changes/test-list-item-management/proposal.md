## Why

Sub-proposal 4.9 of the `test-coverage` initiative — a HIGH-stakes capability-flow carve-out (claim under `quantity_limit` race, list reorder). The parent `test-coverage` `tasks.md` §4.9 names the carve-out as `app/actions/items.ts`, `app/actions/lists.ts`, and the item-management UI, and explicitly permits a mid-flight split. **This proposal takes the split: it covers the two server-action files only.** The item-management UI (the choose-items page, item form, purchase modal, delete-confirmation dialog, archive affordance, and `@dnd-kit/sortable` reorder surface) is deferred to a sibling sub-proposal (`test-list-item-management-ui`), tracked by a new GitHub issue opened against this carve-out.

The scope decision is forced by the coverage mechanism: vitest per-file thresholds are whole-file, so whichever sub-proposal enumerates `app/actions/items.ts` / `app/actions/lists.ts` in `vitest.config.ts` must bring the **entire** file to the floor in one change. Both files mix functions owned by several capabilities (see the cross-capability overlap below), so this sub-proposal owns the per-file coverage-floor attribution for both action files and exercises every exported and private function in them — while restricting its *invariant elevation* to the `list-item-management` capability it owns.

Both files are server actions reached only through the app's mutation paths; they have **zero** test coverage today. This is the first sub-proposal to test server actions against the real test database (pglite), consuming the patterns the `test-foundation-spike` PoC established (`test/helpers/db.ts`'s `bootPglite()`, `mockNextCache()`, the `sqlstateOf` `.cause.code` finding).

Inherited constraints surfaced by spec-grep (every active spec.md was grepped for the function names, table names, cache tags, and DOM/route strings these files touch):

- **`testing-foundation`** (active accumulator at `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus the archived foundation/primitive deltas) — runner (vitest 4.x, the `node` project for `.test.ts`), four-gate pre-merge, `__tests__/` colocation, the universal per-file `COVERAGE_FLOOR` (`lines:98 / statements:98 / branches:95 / functions:100`) referenced from the single constant in `vitest.config.ts`, the no-backdoor disposition rule, the assertion-substance bar, the `<State>_<Behavior>` `it()` shape, the three-role `describe()` convention, the four-audit + invariant-elevation obligations, the `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy, and — load-bearing here — the **"DAL functions are not mocked from action tests"** rule: internal modules (DAL, `lib/`, server actions) SHALL NOT be mocked; integration tests SHALL exercise them against the real test database. Only the **network boundary** (NextAuth Google OAuth) is mockable. Every requirement applies verbatim.
- **`list-item-management`** (active, substantive — 13 requirements) — owns the choose-items flow (`setListItems`), the `createPurchase` / `removePurchase` claim contract (including the no-client-`user_id` rule and the guest-must-supply-`purchase_id` rule), the Archive vs. Delete semantics, and the Delete-confirmation copy. The action-layer requirements are the authoritative contract these tests assert against. **One requirement MUST be MODIFIED** — "Purchase capacity SHALL be enforced atomically against concurrent callers" mandates a transactional `SELECT … FOR UPDATE` row lock that the `neon-http` driver cannot provide and the source does not implement (see "What Changes" and `design.md` Decision 4). **One requirement is ADDED** — the fractional-position reorder/rebalance contract for `updatePriority`, which the source enforces today but no requirement states.
- **`server-endpoint-authorization`** (active, HIGH-stakes, cross-cutting) — owns the global "no client-supplied `user_id`" rule and the per-caller-class authorization matrix (owner / authenticated non-owner / unauthenticated) for every action. The `list-item-management` spec's `createPurchase` requirement already names itself as "the list-item-management-specific application of the cross-cutting contract in `server-endpoint-authorization`." This carve-out exercises the owner-guard / auth-guard branches of all functions in both files for coverage, but the cross-cutting authorization-matrix **invariant elevation** is owned by sub-proposal 4.13 `test-server-endpoint-authorization`. This proposal does not add SHALLs to the `server-endpoint-authorization` spec.
- **`list-visibility`** (active, HIGH-stakes) — owns the three-state visibility contract. `lists.ts` contains `setListVisibility` (and `bookmarkList`'s viewability gate reads `fromDb(list.visibility)`). This carve-out covers `setListVisibility` for the per-file floor and asserts its observable response/persisted-state behavior, but the visibility-enforcement **invariant elevation** (the `shared_at` transition rules, the legacy-`shared` dual-write) is owned by sub-proposal 4.11 `test-list-visibility`. No SHALLs are added to `list-visibility` here.
- **`list-collections`** / **`visit-history`** (active) — own bookmarks (`bookmarkList`, `unbookmarkList`) and visit history (`clearVisitHistory`, `removeVisit`), all of which live in `lists.ts`. Covered here for the per-file floor; invariant elevation deferred to sub-proposals 4.6 `test-list-collections` and 4.14 `test-visit-history`.
- **`item-store-links`** (active) — owns the store-links surface. `items.ts` contains the private `updateItemStores` helper (reached via `createItem` / `updateItem`). Covered here for the per-file floor; store-link invariant elevation deferred to sub-proposal 4.4 `test-item-store-links`.
- **DB driver constraint (`db/index.ts`, `CLAUDE.md`, `DATABASE.md`)** — `drizzle-orm/neon-http`: no `db.transaction(...)`, no `SELECT … FOR UPDATE`. This is the constraint that makes the `list-item-management` capacity requirement's transactional-lock mandate un-implementable and drives the MODIFIED requirement below.

Cache-tag rule: these actions are the **mutation** side of the `'use cache'` reads in `lib/dal.ts`. They call `updateTag('items')`, `updateTag('lists')`, and `updateTag('list_visits')`. The tests assert these tag-invalidation calls fire on the success paths (via `mockNextCache()`, since `updateTag` is a compiler-level no-op under raw vitest). The `getItemEditData` action reads through DAL functions tagged `items` / `lists`; those reads run against pglite directly (the `'use cache'` directive degrades to a plain async call outside the Next runtime — the `test-foundation-spike` `dal-cache.test.ts` documents this).

## What Changes

- **NEW** two colocated test files under `app/actions/__tests__/` (the first `__tests__/` directory under `app/actions/`), both in the `node` vitest project (`.test.ts`, `environment: 'node'`):

  - `app/actions/__tests__/items.test.ts` — exercises every function in `app/actions/items.ts`: `getItemEditData`, `createPurchase`, `removePurchase`, `createItem`, `updateItem`, `archiveItem`, `deleteItem`, and (transitively, via `createItem` / `updateItem`) the private `updateItemStores`, `updateItemLists`, and `emptyStore` helpers. Tests run against a fresh `bootPglite()` database seeded per-case; `@/db` is module-mocked to the pglite drizzle instance; `@/lib/auth`'s `auth()` is module-mocked (the NextAuth network boundary) to return a fixture session, `null`, or a session whose email resolves to no user; `next/cache` is mocked via `mockNextCache()` so `updateTag` is a `vi.fn()` the tests assert on. DAL functions (`getItemById`, `getListsByUser`, `getUserIdByEmail`) and `isItemViewable` are **NOT** mocked — they run against pglite per the testing-foundation rule. Notable behaviors asserted: the `ItemSchema` Zod validation branches (name length, URL validity in `image_url` and per-store `link`, the all-or-nothing store-field refinement, `quantity_limit` integer/min/nullable); the `createPurchase` claim contract (authed uses session identity and discards payload `guest_name`; guest requires non-empty `guest_name`; non-viewable item → `Item not found`; duplicate claim → `Duplicate claim`; capacity reached → `Fully claimed`; the partial-unique-index `23505` catch path → `Duplicate claim`); the `removePurchase` guest-vs-authed ownership checks; ownership guards on `updateItem` / `archiveItem` / `deleteItem`; and the `updateTag('items')` invalidation on success paths.

  - `app/actions/__tests__/lists.test.ts` — exercises every function in `app/actions/lists.ts`: `createList`, `updateList`, `deleteList`, `setListVisibility`, `bookmarkList`, `unbookmarkList`, `clearVisitHistory`, `removeVisit`, `setListItems`, `updatePriority`, and the private `authedUserId`, `checkListBalance`, `rebalanceList` helpers (reached via the exported functions). Same pglite + `@/db` + `@/lib/auth` + `next/cache` harness. Notable behaviors asserted: list CRUD ownership guards and `ListSchema` validation (including the subtitle `'' → null` transform); `setListVisibility`'s `shared_at` transition matrix and legacy-`shared` dual-write; the bookmark / visit-history `onConflictDoUpdate` and bookmarked-row-survives-history-clear logic; `setListItems`'s add/remove diff, no-op short-circuit, fractional `MAX(position)+65536` placement, and the count-reporting message; and `updatePriority`'s midpoint-insertion algorithm plus the `checkListBalance` → `rebalanceList` path when the min-gap (`< 0.001`) is tripped.

- **NEW** per-file thresholds in `vitest.config.ts` for `app/actions/items.ts` and `app/actions/lists.ts`, each referencing the shared `COVERAGE_FLOOR` constant (no per-file numeric variation). These two enumerations are added by THIS sub-proposal; downstream sub-proposals (4.4 / 4.6 / 4.11 / 4.13 / 4.14) that touch these files inherit the already-enumerated thresholds and do NOT re-add them.

- **NEW** ESLint per-file `sonarjs/cognitive-complexity = error` overrides in `eslint.config.mjs` for `app/actions/items.ts` and `app/actions/lists.ts`, promoting them from the global `warn`. Measured complexity at HEAD is expected highest for `updatePriority` (the four-way midpoint-insertion branch ladder) and `createPurchase` (the claim guard ladder); both are expected under the ceiling of 15. If either is over 15 at HEAD, the disposition is an in-place single-file extraction inside the carve-out (recorded in `tasks.md` §5 complexity audit) — never a ceiling raise.

- **MODIFIED** `list-item-management` spec — the "Purchase capacity SHALL be enforced atomically against concurrent callers" requirement is rewritten to match the source and the `neon-http` no-transactions constraint. The current requirement mandates a transactional row lock (`SELECT … FOR UPDATE` inside a `db.transaction(...)`); this is un-implementable on the `neon-http` driver and the source does not do it. The MODIFIED requirement keeps the **partial unique index** layer (which closes the authenticated-duplicate race at the DB layer and is the only enforceable backstop), states the **best-effort** in-app capacity check (`existing.length >= quantity_limit` → `Fully claimed`), and documents the **residual capacity race** for guest claims and distinct-user claims on a limited item as an **accepted limitation** (matching the source comment at `app/actions/items.ts:231-247`). See `design.md` Decision 4. This is the spec-follows-source precedent established by `test-app-frame`'s R2 mobile-nav modification.

- **ADDED** `list-item-management` spec requirement — the `updatePriority` fractional-position reorder contract: positions are integers spaced by 65536, a moved item is inserted at the integer midpoint between its target and the target's neighbor (or `floor(target/2)` / `target+65536` at the ends), and when the two closest positions collide below a min-gap of `0.001` the whole list is rebalanced to `(index+1)*65536`. Non-obvious (fractional indexing + rebalance threshold), survives reimplementation (list ordering is a durable contract), protects a real failure mode (position collision → unstable/incorrect ordering of a shared gift list). See `design.md` Decision 5.

- **NEW** `testing-foundation` carve-out bookkeeping record (Tier 2 per parent design D13) — recorded ONLY in this sub-proposal's `specs/testing-foundation/spec.md` delta (and at archive under `openspec/changes/archive/<date>-test-list-item-management/`). Does NOT roll into the parent accumulator and does NOT create/modify `openspec/specs/testing-foundation/spec.md`.

- **NEW** four-audit + invariant-elevation findings recorded in `tasks.md` §5 (anticipated dispositions enumerated there).

- **NEW** sibling GitHub issue + `test-coverage/tasks.md` checkbox for `test-list-item-management-ui` (the deferred item-management UI carve-out), per the parent's "split mid-flight → open a sibling issue" instruction and the `testing-foundation` "deferred finding becomes a new checkbox" scenario.

- **NO** test for `lib/dal.ts`, `lib/listAccess.ts`, `lib/visibility.ts`, `lib/sqlstate.ts` — out of carve-out (DAL/visibility/listAccess are exercised *through* the actions but their own coverage floors are owned by 2.1 `test-pure-libs`, already archived, and the relevant capability sub-proposals).
- **NO** test for the item-management UI — deferred to `test-list-item-management-ui` (the split).

## Impact

- **New files:** two `.test.ts` files under `app/actions/__tests__/` (`items.test.ts`, `lists.test.ts`). A shared test-helper (`app/actions/__tests__/test-helpers.ts`) is extracted only if the §5 duplication audit finds the pglite/`@/db`/`auth` harness reused in a way that warrants it; otherwise the harness is inlined per file (excluded from coverage by the existing `**/__tests__/**` glob).
- **Modified config:** `vitest.config.ts` gains two per-file `thresholds` entries (both → `COVERAGE_FLOOR`). `eslint.config.mjs` gains two paths in the `sonarjs/cognitive-complexity = error` override array.
- **Modified source:** none expected. One conditional disposition: a complexity-driven in-place extraction in `updatePriority` or `createPurchase` if either exceeds 15 at HEAD (§5).
- **Modified specs:** `openspec/specs/list-item-management/spec.md` gains one MODIFIED requirement (capacity) and one ADDED requirement (reorder). The Tier-2 testing-foundation record lives only in this change's archive directory.
- **CI:** the four-gate workflow runs unchanged; the `test` job grows by two `node`-project files. Each `bootPglite()` call runs the full migration set into an in-memory Postgres — the spike measured this as the dominant per-test cost; tests batch setup per `describe` where the fixture is shared and reset between cases.
- **Dependencies:** none added (`@electric-sql/pglite` already present from foundation).
- **Downstream:** the parent `test-coverage` §4.9 checkbox does NOT flip on this change's archive — it flips only when BOTH this change and the sibling `test-list-item-management-ui` archive (the §4.9 line is split into two checkboxes). The per-file floors for both action files are in force after this change, so 4.4 / 4.6 / 4.11 / 4.13 / 4.14 inherit known-good, floor-locked action files and add only their capability-specific assertions + spec elevations.
- **Risk:** medium. The claim-race and reorder logic are the HIGH-stakes core; the pglite harness gives real-DB fidelity for the partial-unique-index `23505` path (the spike confirmed the SQLSTATE lands on `.cause.code` and `sqlstateOf` already handles it). The capacity-requirement MODIFICATION is a defensible spec-follows-source correction backed by the documented driver constraint, but a reviewer who reads only the spec delta must understand the direction — the proposal and `design.md` Decision 4 state it explicitly.
