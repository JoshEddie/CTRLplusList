## Why

Sub-proposal 4.5 of the `test-coverage` initiative — a capability-flow carve-out covering the **items browser chrome** under `app/(main)/items/`. Per the parent `test-coverage/tasks.md` §4.5, the carve-out is "items browser chrome under `app/(main)/items/`"; the parent leaves the file boundary to be fixed at proposal time, so this proposal pins it (see "Carve-out boundary" below) and defers the directory's non-chrome concerns to their owning capabilities.

The `items-browser-chrome` capability already has a substantive active spec at `openspec/specs/items-browser-chrome/spec.md` (created by archiving `compact-items-mobile-chrome`); this sub-proposal therefore does NOT create a new capability spec. It (a) covers the seven executable chrome source files at the universal `COVERAGE_FLOOR`, (b) ADDS five behavioral requirements to the `items-browser-chrome` spec for invariants the source enforces today that no requirement currently locks — the filter→sort→paginate pipeline, page clamping + the filtered-empty state, URL-as-source-of-truth with default-param omission, the `items_page_size` cookie contract, and the windowed pagination range — and (c) adds the carve-out bookkeeping under `testing-foundation` Tier 2 per the parent's D13 two-tier rollup. No existing requirement is MODIFIED — the spec's five layout requirements describe current source behavior (the tests lock the DOM-level half of those CSS/viewport requirements, see Decision 4 in `design.md`).

Inherited constraints surfaced by spec-grep:

- `testing-foundation` (active accumulator in `openspec/changes/test-coverage/specs/testing-foundation/spec.md` plus archived deltas through `test-app-frame`) — vitest 4.x runner with the jsdom/node two-project split, four-gate pre-merge, `__tests__/` colocation, universal per-file floor `lines:98 / statements:98 / branches:95 / functions:100` referenced from the single `COVERAGE_FLOOR` constant in `vitest.config.ts`, no-backdoor disposition rule (`/* v8 ignore */` with a named reason, never lower the floor), `<State>_<Behavior>` `it()` shape, three-role `describe()` convention, the four-audit obligation, the invariant-elevation audit's three-part test, the assertion-substance bar, the `sonarjs/cognitive-complexity` warn-globally / error-per-carve-out policy, and the NextAuth-as-network-boundary mocking allowance. Every requirement applies verbatim.
- `items-browser-chrome` (active spec) — 5 active requirements: the chrome scope (toolbar / `.item-grid`–`.item-list` view mode / `.items-pagination`, on `.container--items-library` and `.container--list-details`); mobile view mode is single-column list ≤599px; mobile view toggle hidden ≤599px (still mounted in DOM); mobile toolbar collapses to one `1fr auto` row with a compact filters icon; pagination floats over content at all viewports. The tests assert against these SHALLs as the authoritative contract; the CSS/viewport halves are locked at the DOM-presence level (jsdom does not compute layout — see `design.md` Decision 4). No scenario is MODIFIED.
- `items-price-filter` (active spec) — owns `PriceFilterPopover`'s debounce/inverted-pair/footer behavior. The spec explicitly delegates "the layout of the items toolbar row" to `items-browser-chrome` and composes the toolbar. `PriceFilterPopover.tsx` is OUT of this carve-out (see deferral below) and is module-mocked where `ItemsToolbar` renders it.
- `item-store-links` (active spec) and `list-item-management` (active spec) — own the item-card content (`Item.tsx`, `StoreLinks.tsx`, `Purchase.tsx`, `ItemPhoto.tsx`), the owner edit/archive/delete affordances, the item form, the purchase/claim modal, and the `@dnd-kit` reorder view (`SortItems.tsx` — parent §4.9 names "reorder via `@dnd-kit/sortable`" as HIGH-stakes 4.9 scope). `Item.tsx` is module-mocked where `Items` / `ItemsBrowser` render it.
- `popover-trigger-system`, `segmented-control-system`, `form-field-system`, `chip-system`, `button-system` (active specs) — own the primitives the chrome composes (`PopoverTrigger`, `SegmentedControl`, `SelectField` / `SearchField`, `Chip`, `Button`). These are already tested under sub-proposals 3.1–3.6 and are rendered (not mocked) so the chrome's integration through them is exercised; no primitive-spec SHALL is asserted directly from this carve-out.
- `app-frame` (active spec, sub-proposal 4.1 — archived) — `ItemsToolbar` consumes `useKeyboardOffset`, whose `--keyboard-offset` CSS-variable contract 4.1 elevated. The hook is in the app-frame carve-out (already tested); it is NOT mocked here. jsdom has no `visualViewport`, so the hook short-circuits to a no-op during these tests, which is the contract 4.1 locked.
- Cross-cutting design-system rule: the chrome composes governed primitives without page-scoped overrides of their classes; no primitive variant/size is added here, so there is no primitive-spec MODIFIED entry. Cache-tag rule does NOT apply — the carve-out files are client components / pure helpers with no `'use cache'` reads; the server reads that feed them (`getItemsByUser`, `getItemsByListId`) live in the page-shell files that are deferred below.

## Carve-out boundary

**IN the carve-out (12 executable files):**

The carve-out began as 7 files. During apply, `ItemsToolbar.tsx` measured cognitive-complexity 32 (the proposal estimated ~8–12); the complexity audit (9.3) disposed of it by decomposing it **in-place** into a co-located `itemsToolbar/` module (single-file refactor authority; behavior preserved by the new tests). The final file set:

- `app/(main)/items/ui/components/ItemsBrowser.tsx` — the filter → sort → paginate orchestrator (the heaviest logic in the carve-out).
- `app/(main)/items/ui/components/Items.tsx` — the view-mode wrapper that renders `.item-grid` vs `.item-list`.
- `app/(main)/items/ui/components/Pagination.tsx` — the pagination control and its `buildRange` windowing helper.
- `app/(main)/items/ui/components/PageSizeSelect.tsx` — the page-size `SelectField`.
- `app/(main)/items/ui/components/itemFilters.ts` — `compareItems`, `displayPrice`, `firstStoreName` (pure sort/price-derivation logic).
- `app/(main)/items/ui/components/paginationConstants.ts` — `DEFAULT_PAGE_SIZE`, `PAGE_SIZE_OPTIONS` (pure constants; covered transitively, enumerated for the floor).
- `app/(main)/items/ui/components/itemsToolbar/ItemsToolbar.tsx` — the toolbar orchestrator: URL-param wiring, the filters-sheet open/close state + Escape listener, search-commit wiring, the chip row, view toggle.
- `app/(main)/items/ui/components/itemsToolbar/FiltersSheet.tsx` — the filters sheet (sort / purchases / show selects, store + price popovers, close/done), presentational.
- `app/(main)/items/ui/components/itemsToolbar/PurchasesSelect.tsx` — the mode-aware purchases `SelectField`, presentational.
- `app/(main)/items/ui/components/itemsToolbar/SearchInputControl.tsx` — the debounced search field, presentational.
- `app/(main)/items/ui/components/itemsToolbar/utils.ts` — `buildQueryUrl`, `patchedParams`, `toggledStoreParams`, `sortOptionsFor`, `priceChipLabel`, `countActiveFilters`, `buildChips` (pure helpers).
- `app/(main)/items/ui/components/itemsToolbar/toolbarConstants.ts` — `SORT_LABELS`, `SORT_KEYS_BY_MODE`, the purchases/show label tables (pure data; named `toolbarConstants` rather than `constants` to avoid vite's Node-builtin `constants` resolver collision). The module's `types.ts` is type-only (coverage-excluded) and `index.ts` is a re-export barrel.

**DEFERRED (with rationale — not this carve-out):**

- `Item.tsx`, `ItemPhoto.tsx`, `Purchase.tsx`, `StoreLinks.tsx` — item-card content, owned by `item-store-links` (4.4) / `list-item-management` (4.9).
- `SortItems.tsx`, `SortItemsContainer.tsx` — owner `@dnd-kit` reorder view, explicitly named HIGH-stakes 4.9 scope ("reorder via `@dnd-kit/sortable`").
- `itemform/*`, `purchasemodal/*`, `DeleteItemButton.tsx`, `EditItemButton.tsx`, `[id]/page.tsx`, `[id]/ItemFormBody.tsx`, `returnTo.ts` — item create/edit form, claim/purchase modal, and the edit-page return-URL sanitizer (`sanitizeReturnTo` is consumed only by `[id]/ItemFormBody.tsx`). All owned by `list-item-management` (4.9).
- `PriceFilterPopover.tsx` — owned by the `items-price-filter` spec, which has NO enumerated sub-proposal in `test-coverage/tasks.md`. **Discovered sub-proposal:** added below.
- `StoreFilterPopover.tsx` — a filter popover with no dedicated capability spec; the `items-browser-chrome` spec explicitly scopes out "the filters bottom-sheet UX." **Discovered sub-proposal:** added below.
- `ItemsContainer.tsx`, `ItemsPage.tsx`, `app/(main)/items/page.tsx`, `app/(main)/items/loading.tsx` — page-level server/route shells. They mix concerns owned by other carve-outs: `auth()` + DAL reads (`getItemsByUser` / `getItemsByListId` / `getListsByUser`) + `next/headers` cookie parsing + `redirect()` (server-endpoint authorization, parent §4.13) and the new-item form mount (`ItemFormContainer`, 4.9). They are not "chrome *within* `ItemsBrowser`" (the spec's own scope phrasing). **Discovered sub-proposal:** added below for the items-library page shell so the directory does not fall through the cracks at governance close-out §7.1.

## Capabilities

### New Capabilities

None. The `items-browser-chrome` capability spec already exists with five substantive requirements.

### Modified Capabilities

- `items-browser-chrome`: FIVE new requirements are ADDED (filter→sort→paginate pipeline; out-of-range page clamp + filtered-empty clear-filters affordance; URL-derived view/sort/filter state with default-param omission and `page` reset; `items_page_size` cookie persistence + invalid-value normalization; windowed pagination range with disabled bounds). ## Impact

- **New files:** seven `*.test.ts(x)` files under `app/(main)/items/ui/components/__tests__/`. No shared `test-helpers` extracted on first authoring; the §duplication audit may surface a `next/navigation` mock factory or an `ItemDisplay` fixture builder if reused across 3+ files — extracted to `test/helpers/` or `test/fixtures/` only if so.
- **Modified config:** `eslint.config.mjs` gains seven paths in the per-file `sonarjs/cognitive-complexity = error` override array; `vitest.config.ts` gains seven entries in the per-file `thresholds` map, each pointing at `COVERAGE_FLOOR`.
- **Modified source:** none expected. Any in-place refactor (e.g. extracting a URL-param helper to drop `ItemsBrowser`/`ItemsToolbar` complexity below 15, or splitting a function the testability audit flags) is disposed of within the carve-out per the foundation's refactor authority, with the new tests proving behavior preservation. Recorded in `tasks.md` §audits.
- **Modified specs:** `openspec/specs/items-browser-chrome/spec.md` gains five ADDED requirements; no removals or modifications. The carve-out bookkeeping requirement lives ONLY in this sub-proposal's archive directory; it does NOT roll into the parent `test-coverage` accumulator and does NOT modify the active `openspec/specs/testing-foundation/spec.md`.
- **Modified governance:** `openspec/changes/test-coverage/tasks.md` §4 gains three discovered sub-proposal checkboxes (`test-items-price-filter`, `test-items-store-filter`, `test-items-library-shell`).
- **CI:** the four-gate workflow runs unchanged; the `test` job grows by seven files (five jsdom, two node). `ItemsBrowser.test.tsx` and `ItemsToolbar.test.tsx` are the heaviest (fake timers for the search debounce, `next/navigation` mocks, child stubs).
- **Dependencies:** none added.
- **Downstream unblock:** the parent `test-coverage` checkbox for 4.5 flips on archive. The three discovered sub-proposals make the rest of the `app/(main)/items/` directory's coverage ownership explicit.
- **Risk:** medium-low. Seven files, all testable as-shipped. Notable risks:
  - **`ItemsBrowser` transitively renders `Item`.** Mitigation: module-mock `./Item` to a stub in `ItemsBrowser.test.tsx` and `Items.test.tsx` (out-of-carve-out child), asserting on item ids / forwarded props, not card internals. Documented in `design.md` Decision 2.
  - **Search-debounce timing.** `SearchInputControl` commits 200ms after the last keystroke. Mitigation: `vi.useFakeTimers()` + `vi.advanceTimersByTime`, asserting a single `router.replace` per burst. Documented in `design.md` Decision 5.
  - **Filter `useMemo` dependency stability.** `filteredSorted` depends on `selectedStoresKey` (a `join('|')` projection) with an `exhaustive-deps` disable; tests assert behavior across store-set changes, not the deps array. Documented in `design.md` Decision 6.
  - **Branch coverage of the chip / filter-count matrix.** The chip builder and `filterCount` have a branch per filter type and per `mode`. Mitigation: a parameterized matrix over `mode ∈ {items, list, choose}` × active-filter combinations to hit `branches ≥ 95`. Documented in `design.md` Decision 7.
  - **The five elevated requirements are real behavioral claims worth defending.** Each survives a reasonable reimplementation and protects a real failure mode (wrong filter results, blank over-range pages, URL/cookie contract breaks with the server reader, broken pagination UI). The new tests fail loudly if any is regressed.
- **No runtime behavior change** other than any audit-surfaced in-place refactor (behavior-preserving, proven by the new tests).
